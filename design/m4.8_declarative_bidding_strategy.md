# **Feature design: Declarative Bidding Strategy (M4.8)**

**Purpose**: Design the transition from string-based `bidding_strategy` to a structured object allowing filtering by artefact type.
**Scope**: `holt.yml` schema, `pup` bidding logic, and `orchestrator` validation.
**Estimated tokens**: ~2,000 tokens
**Read when**: Implementing M4.8 or debugging agent bidding behavior.

Associated phase: Coordination
Status: Draft
***Template purpose:*** *This* document is a blueprint for a single, implementable milestone. Its purpose is to provide an unambiguous specification for a developer (human *or AI) to build a feature that is consistent with Holt's architecture and guiding principles.*

## **1\. The 'why': goal and success criteria**

### **1.1. Goal statement**
Enable agents to declaratively specify which artefact types they bid on via `holt.yml`, preventing unintended infinite loops (like bidding on their own output) without requiring custom `bid.sh` scripts for simple use cases.

### **1.2. User story**
As an agent developer, I want to configure my agent to only bid on `GoalDefined` artefacts using a simple configuration in `holt.yml`, so that I don't have to write and maintain a separate `bid.sh` script just to filter out other artefact types.

### **1.3. Success criteria**
*   A user can define `bidding_strategy` as an object in `holt.yml`: `bidding_strategy: { type: "exclusive", target_types: ["GoalDefined"] }`.
*   The `pup` binary respects this configuration and only bids on claims where the target artefact type matches one of the `target_types`.
*   **Breaking Change**: The legacy string format (e.g., `bidding_strategy: "exclusive"`) is **NO LONGER SUPPORTED**. Using it will result in a validation error.
*   The E2E test `TestE2E_Phase2_HappyPath` passes using this new configuration instead of the custom `bid.sh` workaround.

### **1.4. Non-goals**
*   Complex filtering logic (e.g., regex on payload, filtering by producer role) is out of scope for this milestone. This is strictly about *artefact type*.
*   Changing the underlying `bid` protocol (stdin/stdout of `bid.sh`). The `pup` will handle the filtering internally before deciding whether to run `bid.sh` or auto-bid.

## **2\. The 'what': component impact analysis**

### **2.1. Blackboard changes**
*   No changes.

### **2.2. Orchestrator changes**
*   **New/modified configurations (holt.yml)**:
    *   Update `Agent` struct in `internal/config/config.go`.
    *   Change `BiddingStrategy` field type from `string` to `BiddingStrategyConfig`.
    *   Update validation logic to reject string values (or rely on YAML unmarshalling error, but a custom error is better).

### **2.3. Agent pup changes**
*   **New/modified logic**:
    *   `pup` needs to parse the new `bidding_strategy` structure.
    *   In the `WatchClaims` loop, `pup` must check if the claim's target artefact type matches the `target_types` list.
    *   If `target_types` is empty/nil, it implies "all types" (but we might want to enforce explicit types? For now, let's say empty list = all types, but the *format* must be an object).

### **2.4. CLI changes**
*   No user-facing CLI changes, but `holt check` or `holt up` will fail for old configs.

## **3\. The 'how': implementation & testing plan**

### **3.1. Key design decisions & risks**
*   **Breaking Change**: We are intentionally breaking backward compatibility to force explicit configuration and prevent "bidding on everything" bugs.
    *   *Decision*: Remove support for string-based `bidding_strategy`.
    *   *Risk*: User frustration during upgrade.
    *   *Mitigation*: Provide a clear error message when a string is detected: "bidding_strategy must be an object with 'type' and 'target_types' fields. Legacy string format is no longer supported."
*   **Default Behavior**:
    *   *Decision*: `target_types` is optional. If omitted or empty, it means "match all types". This allows power users to still "bid on everything" if they really want to, but they have to use the object syntax `{ type: "exclusive" }`.

### **3.2. Implementation steps**
*   [ ] **[Config]** Define `BiddingStrategyConfig` struct:
    ```go
    type BiddingStrategyConfig struct {
        Type        string   `yaml:"type"`
        TargetTypes []string `yaml:"target_types,omitempty"`
    }
    ```
*   [ ] **[Config]** Implement `UnmarshalYAML` for `BiddingStrategyConfig` to **detect and reject** string inputs with a helpful error message.
*   [ ] **[Config]** Update `Agent` struct to use `BiddingStrategyConfig`.
*   [ ] **[Pup]** Update `pup` to use the new config struct.
*   [ ] **[Pup]** Update `claimWatcher` to filter claims based on `TargetTypes`.
*   [ ] **[E2E]** Update `TestE2E_Phase2_HappyPath` to use the new object format.
*   [ ] **[Docs]** Update documentation to reflect the breaking change and new syntax.

### **3.3. Migration & Fix Strategy**
Since this is a breaking change, we must systematically update the codebase:

**Phase 1: Core Configuration & Unit Tests**
*   **Update `internal/config/config.go`**: Implement the new struct and validation logic.
*   **Fix Unit Tests**: Update all inline YAML in `internal/config/*_test.go` (approx. 20+ occurrences) to use the new object format.
    *   Example: `bidding_strategy: "exclusive"` -> `bidding_strategy: { type: "exclusive" }`

**Phase 2: Propagation (Orchestrator & Pup)**
*   **Update Orchestrator (`internal/orchestrator/workers.go`)**: Serialize the `BiddingStrategyConfig` object (likely to JSON) when passing it to the agent container via environment variables.
*   **Update Pup (`internal/pup/config.go`, `engine.go`)**: Update `pup` to parse the serialized strategy from environment variables and implement the filtering logic.

**Phase 3: E2E Tests & Demos**
*   **Update `internal/testutil/e2e.go`**: Update helper functions (`GitAgentHoltYML`, etc.) to generate valid object-format YAML.
*   **Update Demos**: Update `demos/dev-lifecycle-demo/holt.yml` and other demo configs to use the new syntax.

### **3.3. Performance & resource considerations**
*   Negligible impact.

### **3.4. Testing strategy**
*   **Unit tests**:
    *   `TestConfig_BiddingStrategy_String`: Verify `bidding_strategy: "exclusive"` **FAILS** with correct error message.
    *   `TestConfig_BiddingStrategy_Object`: Verify `bidding_strategy: { type: "exclusive", target_types: ["GoalDefined"] }` parses correctly.
    *   `TestPup_Filter`: Verify `pup` ignores claims for non-matching types.
*   **E2E tests**:
    *   `TestE2E_Phase2_HappyPath`: Verify the agent only bids on `GoalDefined` and ignores `CodeCommit`.
