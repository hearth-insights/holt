# **Feature design: M4.11 - Production Gatekeeper Demo & Direct-to-Human Routing**

**Purpose**: Design for the "Production Gatekeeper" demo and enabling "Direct to Human" question routing.  
**Scope**: Orchestrator (Question Handler), Agent (Demo), Documentation.  
**Estimated tokens**: ~2,500 tokens  
**Read when**: Understanding the "Human-in-the-Loop" capabilities for M4.11.

Associated phase: Human-in-the-Loop  
Status: Draft

***Template purpose:*** *This document serves as the blueprint for the M4.11 milestone, defining the requirements for the "Production Gatekeeper" demo and the necessary system enhancements to support it.*

## **1\. The 'why': goal and success criteria**

### **1.1. Goal statement**

Enable agents to explicitly request human intervention ("Direct to Human") regardless of the artifact producer, and demonstrate this capability with a "Production Gatekeeper" demo where deployment is paused for human approval.

### **1.2. User story**

"As an operator, I want to deploy a 'gatekeeper' agent that runs automated checks but strictly pauses for my manual approval before deploying to production. Even though the 'checks' artifact was produced by the agent, the question 'Deploy to prod?' should come directly to me, not back to the agent."

### **1.3. Success criteria**

1.  **Direct-to-Human Routing**: An agent can output a Question with `routing: "human"`, causing the Orchestrator to emit a `human_input_required` event instead of creating a feedback claim for the agent itself.
2.  **Gatekeeper Demo**: A new demo agent (`demos/production-gatekeeper`) runs a simple workflow:
    *   Simulates "Running Tests" -> Produces "TestResults".
    *   Asks "Tests passed. Authorize deployment? (yes/no)" targeting the "TestResults".
    *   Waits for human answer via CLI (`holt answer`).
    *   If "yes", produces "DeploymentManifest". If "no", produces "DeploymentAborted".
3.  **End-to-End Verification**: A user can run the demo and successfully approve the deployment using the CLI.

### **1.4. Non-goals**

*   Complex RBAC for who can answer questions (any user with CLI access can answer).
*   UI integration (CLI only for this milestone).

## **2\. The 'what': component impact analysis**

### **2.1. Blackboard changes**

*   **New/modified data structures**:
    *   `QuestionPayload` (inside `internal/orchestrator/question_handler.go`): Add `Routing` field.
        ```go
        type QuestionPayload struct {
            QuestionText      string `json:"question_text"`
            TargetArtefactID  string `json:"target_artefact_id"`
            Routing           string `json:"routing,omitempty"` // "auto" (default) or "human"
        }
        ```
*   **New Pub/Sub channels**: None.

### **2.2. Orchestrator changes**

*   **New/modified logic**:
    *   Update `handleQuestionArtefact` in `internal/orchestrator/question_handler.go`.
    *   Logic change:
        *   Parse `Routing` field from payload.
        *   If `Routing == "human"`, **bypass** producer lookup and directly call `publishHumanInputRequiredEvent`.
        *   Else (default), preserve existing "Percolate Up" behavior.
*   **New/modified configurations**: None.

### **2.3. Agent pup changes**

*   **New/modified logic**: None (agents just populate the JSON payload).
*   **Changes to tool execution contract**:
    *   Agents can now include `"routing": "human"` in their `artefact_payload` JSON when creating a Question.

### **2.4. CLI changes**

*   **New/modified commands**: None (`holt answer` already exists).
*   **Changes to user output**: None.

## **3\. The 'how': implementation & testing plan**

### **3.1. Key design decisions & risks**

*   **Explicit Routing Field**: We chose to add an explicit `routing` field rather than inferring intent. This gives agent developers precise control.
    *   *Risk*: Agents might misuse it to spam humans. *Mitigation*: Future rate-limiting (out of scope for M4.11).
*   **String Field vs Enum**: Using string `"human"` allows for future extensibility (e.g., `"audit_team"`, `"manager"`).

### **3.2. Implementation steps**

1.  **[Orchestrator]** Update `QuestionPayload` struct in `internal/orchestrator/question_handler.go`.
2.  **[Orchestrator]** Update `handleQuestionArtefact` logic to respect `routing: "human"`.
3.  **[Demo]** Create `demos/production-gatekeeper` directory.
4.  **[Demo]** Implement `run.sh` for the gatekeeper agent:
    *   Step 1: Sleep 2s, output "TestResults".
    *   Step 2: Output Question with `routing: "human"`.
    *   Step 3: Read Answer. If "yes", output "DeploymentManifest".
5.  **[Demo]** Create `holt.yml` for the demo.
6.  **[Docs]** Update `docs/tutorials/human-in-the-loop.md` (or similar) to document the `routing` field.

### **3.3. Performance & resource considerations**

*   **Resource usage**: Negligible.
*   **Scalability limits**: Human speed is the bottleneck. System handles waiting questions efficiently (they are just artifacts).

### **3.4. Testing strategy**

*   **Unit tests**:
    *   `internal/orchestrator/question_handler_test.go`: Add test case for `Routing: "human"`. Verify `human_input_required` event is emitted and NO feedback claim is created.
*   **Integration tests**:
    *   Manual verification using the new demo.
*   **E2E tests**:
    *   We can script the CLI interaction in a test (start demo, wait for event, send answer, verify completion).

## **4\. Principle compliance check**

### **4.1. YAGNI**

*   We need this feature because "Production Gatekeeper" is a core use case for auditable AI. The current "Percolate Up" only works for peer review, not final authorization.

### **4.2. Auditability**

*   The Question and the human's Answer are both immutable artifacts, providing a perfect audit trail of *who* authorized the deployment and *when*.

### **4.3. Small, single-purpose components**

*   Changes are localized to `question_handler.go`.

### **4.4. Security considerations**

*   **Attack surface**: Ensure `routing` field doesn't allow bypassing other checks (it only routes to human, which is safe/blocking).
*   **Input validation**: Validate `Routing` values (allow list or just treat unknown as auto).

### **4.5. Backward compatibility**

*   **Additive**: Old agents without `routing` field will default to `"auto"` (current behavior). No breaking changes.

### **4.6. Dependency impact**

*   None.
