# **Feature design: M4.1 - Advanced Question/Answer System**

**Purpose**: Enable agents to ask clarifying questions when encountering ambiguous artefacts, treating questions as a form of "late review feedback" that triggers the M3.3 automated feedback loop.

**Scope**: Question/Answer workflow for Agent-to-Agent and Agent-to-Human clarification, reusing existing feedback loop machinery.

Associated phase: Human-in-the-Loop
Status: Draft

---

## **1. The 'why': goal and success criteria**

### **1.1. Goal statement**

To enable an agent to signal that a source artefact is ambiguous by producing a Question artefact, triggering a feedback loop that results in a clearer version of the source artefact from its original author.

### **1.2. User story**

**As an agent developer**, when my agent encounters an ambiguous instruction in a source artefact, I want it to ask a question directly to the author of that artefact. The author will then produce a new, clearer version that incorporates the answer, allowing my agent to proceed with unambiguous requirements rather than guessing or failing.

**As a human operator**, when an agent questions an artefact I produced (like a GoalDefined), I want to be notified via the CLI so I can provide a clarified version of my goal, keeping the workflow moving and preventing wasted work on incorrect assumptions.

**Core Insight**: An artefact ambiguous enough to require a question is a defective artefact. Rather than building pause/resume machinery, we treat Questions as a form of "late review feedback." The original claim is terminated, and the author is assigned to produce a clearer version. This maintains agent statelessness and prevents downstream complexity where future agents would have to parse long Q&A chains.

### **1.3. Success criteria**

1. **Agent-to-Agent Q&A**: An agent working on `spec-v1` can produce a Question artefact referencing `spec-v1`. The orchestrator terminates the agent's claim, creates a `pending_assignment` claim for the author of `spec-v1`, who then produces `spec-v2` with the Question ID in `source_artefacts`. A new standard claim is created for `spec-v2` and the workflow continues naturally.

2. **Agent-to-Human Q&A**: An agent can question a `GoalDefined` artefact produced by the "user" role. A human operator runs `holt questions`, sees the question, runs `holt answer <id> "clarified text"`, and the CLI creates a new `GoalDefined-v2` artefact. The orchestrator creates a new claim and agents can bid on the clarified goal.

3. **Iteration Limits Enforced**: If an artefact is questioned repeatedly beyond the `orchestrator.max_review_iterations` limit, the orchestrator creates a Failure artefact and terminates the workflow, preventing infinite Q&A loops.

4. **Human Observability**: The `holt watch` command displays a distinct `HUMAN_INPUT_REQUIRED` event when a Question targets the "user" role, making it immediately obvious that human intervention is needed.

### **1.4. Non-goals**

- This milestone will **not** include a separate Answer structural type. The "answer" is always embodied in a new version of the questioned artefact.
- This milestone will **not** implement a pause/resume mechanism for claims. Questions terminate the original claim (exactly like review feedback in M3.3).
- This milestone will **not** support questions directed to multiple roles simultaneously. A Question references a single target artefact.
- This milestone will **not** include automatic timeouts for unanswered questions. Questions targeting the "user" role will wait indefinitely until `holt answer` is run.

## **2. The 'what': component impact analysis**

### **2.1. Blackboard changes**

**New/modified data structures:**

- **Question Artefact (`structural_type: Question`)**: This existing structural type will be formally used. Its `payload` will be a JSON-encoded string with the following schema:
  ```json
  {
    "question_text": "Is null handling in scope for this API?",
    "target_artefact_id": "<uuid-of-the-artefact-being-questioned>"
  }
  ```
  The `target_role` field is NOT included. The orchestrator will extract the `produced_by_role` from the referenced artefact.

- **Claim Schema**: No new fields required. The existing `pending_assignment` status and `additional_context_ids` array (from M3.3) are reused.

- **Answer Structural Type**: This will **not be used**. The Answer type remains in the enum for backward compatibility but is not part of the M4.1 flow. The "answer" is a new version of the questioned artefact.

**New Pub/Sub channels:**
- No new channels required. Existing `artefact_events`, `claim_events`, and `workflow_events` channels are sufficient.

**New Redis keys:**
- No new key patterns required. The flow reuses existing claim and artefact storage patterns.

### **2.2. Orchestrator changes**

**New/modified logic:**

1. **Question Detection**: When an agent with a granted claim produces an artefact with `structural_type: Question`, the orchestrator detects this (similar to how it detects `Review` artefacts in M3.3).

2. **Payload Parsing**: The orchestrator parses the Question's payload as JSON to extract `target_artefact_id` and `question_text`. If parsing fails or `target_artefact_id` is missing, the orchestrator logs an error and skips the Question (it does not crash).

3. **Target Artefact Lookup**: The orchestrator loads the target artefact from Redis using the extracted `target_artefact_id` to determine its `produced_by_role`.

4. **Claim Termination**: The orchestrator terminates the original claim (the one that produced the Question) by setting its status to `terminated` and populating `termination_reason` with a message like: `"Agent asked a clarifying question (Question artefact: <question-id>)"`.

5. **Feedback Claim Creation**: The orchestrator creates a new claim with:
   - `artefact_id`: The `target_artefact_id` from the Question payload (NOT the Question artefact itself)
   - `status`: `pending_assignment`
   - `granted_exclusive_agent`: The `produced_by_role` from the target artefact
   - `additional_context_ids`: An array containing the Question artefact's ID
   - This new claim bypasses the bidding process and is directly assigned to the original author.

6. **Iteration Limit Check**: Before creating the feedback claim, the orchestrator checks the `version` of the target artefact against `orchestrator.max_review_iterations` from `holt.yml`. If the limit is exceeded, it creates a Failure artefact with a message like: `"Max review iterations exceeded for artefact <logical_id>. Latest question: <question_text>"`

7. **Human Role Handling**: If the target artefact's `produced_by_role` is "user", the claim is created as `pending_assignment` but waits indefinitely. No pup is listening for the "user" role. The claim remains in Redis until a human runs `holt answer`.

8. **New Version Detection**: When a new artefact is created with the same `logical_id` as the questioned artefact and an incremented `version`, and the Question artefact's ID is in its `source_artefacts` array, this satisfies the feedback claim. The orchestrator marks the feedback claim as `complete` and creates a new standard claim for the new version.

9. **Workflow Events**: The orchestrator publishes a new event type to the `workflow_events` channel:
   ```json
   {
     "event_type": "human_input_required",
     "question_id": "<question-artefact-id>",
     "question_text": "<question-text>",
     "target_artefact_id": "<target-artefact-id>",
     "timestamp_ms": 1234567890
   }
   ```
   This event is emitted when a Question targets the "user" role.

**New/modified configurations (holt.yml):**
- No changes required. The existing `orchestrator.max_review_iterations` configuration applies to the Q&A loop.

### **2.3. Agent pup changes**

**New/modified logic:**
- The pup already supports receiving artefacts via `additional_context_ids` (M3.3 feedback loop). No new logic is required. When the pup receives a `pending_assignment` grant with a Question in the context, it passes that Question to the tool script as part of the `additional_context` array in stdin.

**Changes to the tool execution contract (stdin/stdout):**

- **Input (stdin)**: No changes required. The existing contract already includes `additional_context` (M3.3):
  ```json
  {
    "claim_type": "exclusive",
    "target_artefact": { /* the artefact being reworked */ },
    "context_chain": [ /* historical context */ ],
    "additional_context": [
      {
        "id": "<question-artefact-id>",
        "structural_type": "Question",
        "type": "ClarificationNeeded",
        "payload": "{\"question_text\": \"...\", \"target_artefact_id\": \"...\"}",
        "produced_by_role": "coder-agent",
        ...
      }
    ]
  }
  ```

- **Output (stdout)**: The existing contract already supports `structural_type` (see `internal/pup/contract.go:76`). Agents signal a Question by returning:
  ```json
  {
    "structural_type": "Question",
    "artefact_type": "ClarificationNeeded",
    "artefact_payload": "{\"question_text\": \"Is null handling in scope?\", \"target_artefact_id\": \"abc-123\"}",
    "summary": "Asking for clarification on API requirements"
  }
  ```
  If `structural_type` is omitted, it defaults to "Standard" (backward compatible).

### **2.4. CLI changes**

**New commands:**

1. **`holt questions [flags]`**

   **Purpose**: Display unanswered Question artefacts, with special focus on those requiring human input.

   **Default behavior** (no flags):
   - Query Redis for the OLDEST unanswered Question artefact
   - An unanswered Question Q about target artefact T is one where NO artefact exists with:
     - `logical_id == T.logical_id`
     - `version > T.version`
     - `Q.id` in `source_artefacts`
   - If an unanswered Question is found: Display it and exit
   - If no unanswered Questions exist: Subscribe to `artefact_events`, wait for a Question artefact to appear, display it, then exit

   **Flags**:
   - `--watch`: Continuously display Question artefacts as they appear (stream mode). Never exit until Ctrl+C.
   - `--exit-on-complete`: Used with `--watch`. Stream questions until a Terminal artefact is created, then exit gracefully.
   - `--since <duration>`: Display ALL unanswered Questions from the specified time range (e.g., `1h`, `30m`, `2d`). Display as a formatted list and exit immediately.
   - `--name <instance>`: Target a specific Holt instance (default: infer from workspace).
   - `--output jsonl`: Output questions as line-delimited JSON for scripting.

   **Output format** (default mode):
   ```
   Question abc-123-def-456 (about artefact xyz-789 "DesignSpec" v1)
     Asked by: coder-agent
     Question: "Is null handling in scope for this API?"

     Answer with: holt answer abc-123 "your clarified requirements"
   ```

   **Output format** (`--since` mode, multiple questions):
   ```
   Unanswered Questions (last 1 hour):

   1. Question abc-123 (asked by coder-agent, 15 minutes ago)
      About: DesignSpec v1 (xyz-789)
      Question: "Is null handling in scope for this API?"

   2. Question def-456 (asked by test-agent, 5 minutes ago)
      About: GoalDefined v1 (uvw-012)
      Question: "Should we test error cases or just happy path?"

   Answer with: holt answer <id> "your clarified text"
   ```

   **Exit codes**:
   - 0: Success (question displayed or watch completed)
   - 1: Error (instance not found, Redis connection failed, etc.)

2. **`holt answer <question-id> "<answer-text>" [flags]`**

   **Purpose**: Respond to a Question by creating a new version of the questioned artefact with clarified content.

   **Arguments**:
   - `<question-id>`: ID of the Question artefact (supports prefix matching, minimum 6 characters if multiple questions exist)
   - `<answer-text>`: The clarified text for the new artefact version. Multi-line text is supported and preserved.

   **Flags**:
   - `--then-questions`: After creating the answer artefact, immediately execute `holt questions` (default behavior) to watch for the next question.
   - `--name <instance>`: Target a specific Holt instance (default: infer from workspace).

   **Behavior**:
   1. **ID Validation**:
      - Query Redis for all unanswered Questions
      - If ≥2 questions exist AND provided ID length < 6 characters: ERROR with message: "Multiple questions exist. Provide at least 6 characters of the question ID."
      - Otherwise: Match ID via prefix match

   2. **Question Loading**:
      - Load the Question artefact by ID
      - Parse the `payload` JSON to extract `target_artefact_id`

   3. **Target Artefact Replication**:
      - Load the target artefact T from Redis
      - Create a new artefact with:
        - `id`: New UUID
        - `logical_id`: T.logical_id (same logical thread)
        - `version`: T.version + 1 (incremented)
        - `structural_type`: T.structural_type (typically "Standard")
        - `type`: T.type (e.g., "GoalDefined", "DesignSpec")
        - `payload`: The provided `<answer-text>` (replaces old payload)
        - `source_artefacts`: [T.id, Question.id] (links to both original and question)
        - `produced_by_role`: "user"

   4. **Artefact Creation**:
      - Write the new artefact to Redis
      - Publish to `artefact_events` channel
      - Orchestrator will detect the new version and create a standard claim

   5. **Chaining** (if `--then-questions`):
      - After successful artefact creation, execute `holt questions` (default behavior)
      - Equivalent to: `holt answer ... && holt questions`

   **Example**:
   ```bash
   # Basic usage
   holt answer abc-123 "Build REST API with JWT auth (not OAuth)"

   # Multi-line answer
   holt answer def-456 "Requirements:
   1. Support null values
   2. Return 400 for invalid input
   3. Document edge cases"

   # Answer and watch for next question
   holt answer abc-123 "Clarified text here" --then-questions
   ```

**Changes to user output:**

- **`holt watch` enhancements**: Add display of the new `human_input_required` workflow event:
  ```
  [2025-01-15 14:32:10] HUMAN_INPUT_REQUIRED
    Question abc-123-def-456 by coder-agent
    "Is null handling in scope for this API?"
    Target: DesignSpec v1 (xyz-789)

    → Run: holt questions
  ```

- **Color coding**: Display `HUMAN_INPUT_REQUIRED` events in a distinct color (e.g., yellow/amber) to make them visually prominent.

## **3. The 'how': implementation & testing plan**

### **3.1. Key design decisions & risks**

**Key Decision 1: Reuse M3.3 Feedback Loop Machinery**

Questions are treated as "late review feedback" that terminates claims and triggers the existing automated feedback loop from M3.3. This decision:
- Eliminates the need for new pause/resume state management
- Reuses battle-tested termination and pending_assignment logic
- Maintains agent statelessness (no paused claims in memory)
- Prevents Q&A chain pollution (always supersede with clearer versions)
- Leverages existing iteration limits to prevent infinite loops

**Alternative Considered**: Implementing a new `pending_question` status with pause/resume logic. Rejected because it adds complexity, requires new state management, and violates the statelessness principle.

**Key Decision 2: Orchestrator Extracts Target Role**

The Question payload only contains `target_artefact_id`, not `target_role`. The orchestrator loads the target artefact and extracts its `produced_by_role`. This decision:
- Reduces agent burden (agents don't need to track roles)
- Makes Questions more robust (role cannot be misspecified)
- Ensures questions always reach the correct author
- Simplifies the agent tool contract

**Alternative Considered**: Requiring agents to specify `target_role` in the Question payload. Rejected because it's error-prone and couples agents to role knowledge.

**Key Decision 3: CLI Auto-Replication of Artefact Structure**

The `holt answer` command automatically replicates the structure of the questioned artefact, requiring only the new payload text from the human. This decision:
- Provides excellent UX (minimal input required)
- Prevents errors (CLI handles version increment, linking, etc.)
- Maintains consistency (logical_id, type, structural_type preserved)

**Alternative Considered**: Requiring humans to specify all artefact fields explicitly. Rejected as too verbose and error-prone.

**Risks:**

1. **JSON Parsing Failures**: Question payloads are JSON strings that must be parsed. Malformed JSON could crash the orchestrator.
   - **Mitigation**: Wrap parsing in try/catch, log errors, skip malformed Questions (don't crash).

2. **Infinite Q&A Loops**: Agents could repeatedly question the same artefact indefinitely.
   - **Mitigation**: Enforce `max_review_iterations` limit. Fail workflow explicitly when exceeded.

3. **Concurrent Questions**: Multiple agents could question the same artefact simultaneously.
   - **Mitigation**: Each Question is independent. Multiple versions may be created, but the artefact graph remains consistent. No locking required.

4. **Orchestrator Restart During Q&A**: If orchestrator crashes while a `pending_assignment` claim is waiting for human input, will it recover?
   - **Mitigation**: M3.5 recovery logic handles `pending_assignment` claims correctly. No new recovery code needed.

### **3.3. Performance & resource considerations**

**Resource usage:**
- Each Question adds 1 artefact to Redis (~500 bytes)
- Each feedback claim adds 1 claim to Redis (~300 bytes)
- New versions add standard artefact storage
- **Total impact**: ~1KB per Q&A exchange (negligible)

**Scalability limits:**
- No impact on agent scaling (controller-worker pattern unaffected)
- No new subscriptions or channels (reuses existing)
- "Unanswered" query scales O(n) with total artefacts but is infrequent (human-triggered)

**Performance requirements:**
- Question detection: <10ms (in-memory JSON parsing)
- Orchestrator processing: <50ms (same as review feedback in M3.3)
- CLI `holt questions`: <200ms for query, instant for streaming
- CLI `holt answer`: <100ms for artefact creation

**Latency impact:**
- None for standard workflows (Questions are opt-in)
- Human response time is the bottleneck (minutes to hours), not system latency

## **4. Principle compliance check**

### **4.1. YAGNI (You Ain't Gonna Need It)**

**Third-party dependencies**: None. This feature uses only existing dependencies:
- Redis (already used for blackboard)
- Docker SDK (already used for orchestrator)
- Cobra (already used for CLI)

**New complexity justification**: The Q&A flow is essential for Phase 4's human-in-the-loop goals. However, we avoid over-engineering by:
- Reusing M3.3 feedback loop (no new state machinery)
- Not implementing timeouts (YAGNI for V1)
- Not supporting multi-role questions (YAGNI for V1)
- Not implementing question prioritization (YAGNI for V1)

### **4.2. Auditability**

This feature **significantly improves** auditability:

**Complete Q&A History**: Every question and its resolution is captured as an immutable artefact chain:
- Question artefact: Records what was asked, when, and by whom
- New version artefact: Contains the answer (clarified content) and links back to the Question via `source_artefacts`
- Terminated claim: Records that the original work was halted due to ambiguity

**Traceable Decision Path**: Future auditors can traverse the artefact graph to understand:
- Why an agent paused work (terminated claim with Question reference)
- What was unclear (Question payload)
- How it was resolved (new version with clarified content)
- Who provided the answer (`produced_by_role` in new version)

**Compliance Value**: For regulated industries (healthcare, finance), this provides:
- Full transparency into AI decision-making
- Human oversight points explicitly captured
- Clarification history for regulatory review
- Proof that ambiguities were resolved before proceeding

### **4.3. Small, single-purpose components**

**Component responsibilities remain clear**:

- **Orchestrator**: Detects Questions, terminates claims, creates feedback assignments (pure coordination logic, no domain knowledge)
- **Pup**: Passes Question context to tools, executes tool scripts (unchanged responsibility)
- **CLI**: Provides human interface for viewing and answering questions (pure UI/UX layer)
- **Agents**: Produce Questions when encountering ambiguity (domain logic, application-specific)

**No tight coupling introduced**:
- Orchestrator does not interpret Question content (opaque JSON payload)
- CLI does not implement orchestration logic (creates artefacts, orchestrator handles rest)
- Pups do not parse Questions (pass to tool script as-is)

**Reuse over duplication**:
- M3.3 feedback loop machinery is reused entirely
- No duplication of termination, pending_assignment, or iteration limit logic

### **4.4. Security considerations**

**No new attack surfaces**:
- Questions use existing artefact creation flow (same auth/validation as all artefacts)
- CLI commands use existing instance discovery and Redis connection (same security model as `holt forage`, `holt hoard`)

**Data exposure**:
- Question payloads are stored in Redis (same as all artefact payloads)
- No new sensitive data types introduced
- Questions may contain clarifications about requirements (not credentials or secrets)

**Input validation**:
- Question payload JSON is parsed with error handling (malformed JSON logged, not executed)
- `holt answer` validates ID length and existence (prevents accidental wrong-question answers)
- Multi-line text in `holt answer` is sanitized by Go's JSON marshaling (no injection risk)

**Container isolation**:
- No changes to agent container privileges or isolation
- Questions are data, not executable code
- No new container-to-container communication paths

**Privilege boundaries**:
- "user" role is special (only humans can produce artefacts as "user" via CLI)
- Agents cannot impersonate "user" (pup always sets `produced_by_role` to agent's configured role)
- No elevation of privileges required

### **4.5. Backward compatibility**

**Additive changes only**:
- Question structural type already exists in enum (no schema change)
- `structural_type` field in tool output contract already exists (M3.2+)
- `additional_context_ids` field already exists in Claim schema (M3.3)
- New CLI commands don't break existing commands

**Existing workflows preserved**:
- Agents that never produce Questions are unaffected
- Workflows without Questions behave identically to Phase 3

## **6. Error scenarios & edge cases**

### **6.1. Failure modes**

**Orchestrator failures:**

1. **Malformed Question Payload**:
   - Scenario: Agent produces Question with invalid JSON in payload
   - Behavior: Orchestrator logs error, skips Question processing, does NOT crash
   - Recovery: Original claim continues (not terminated), agent may retry or fail naturally
   - Test: Unit test with invalid JSON payloads

2. **Missing `target_artefact_id`**:
   - Scenario: Question payload is valid JSON but missing required field
   - Behavior: Orchestrator logs validation error, skips Question
   - Recovery: Same as malformed payload
   - Test: Unit test with `{"question_text": "..."}` (no target)

3. **Target Artefact Not Found**:
   - Scenario: Question references artefact ID that doesn't exist in Redis
   - Behavior: Orchestrator logs error, creates Failure artefact for original claim
   - Recovery: Original claim terminates with Failure
   - Test: Integration test with non-existent UUID

4. **Iteration Limit Exceeded**:
   - Scenario: Artefact version >= `max_review_iterations`, new Question arrives
   - Behavior: Orchestrator creates Failure artefact, terminates workflow
   - Recovery: None (workflow stops, human must intervene with new goal)
   - Test: E2E test with limit=2, produce 3 questions

5. **Redis Connection Lost During Question Processing**:
   - Scenario: Redis unavailable between Question detection and claim creation
   - Behavior: Orchestrator retries with exponential backoff (existing M3.5 logic)
   - Recovery: Question processing resumes when Redis reconnects
   - Test: Integration test with testcontainers (stop/start Redis)

**CLI failures:**

6. **Question Not Found in `holt answer`**:
   - Scenario: User provides question ID that doesn't exist
   - Behavior: CLI displays error: "Question abc-123 not found"
   - Recovery: User checks ID with `holt questions` and retries
   - Test: Unit test with non-existent ID

7. **Ambiguous ID in `holt answer`**:
   - Scenario: User provides 2-char ID when 5 questions exist
   - Behavior: CLI displays error: "Multiple questions exist (5 found). Provide at least 6 characters."
   - Recovery: User provides longer ID
   - Test: Unit test with multiple questions, short ID

8. **Target Artefact Not Found in `holt answer`**:
   - Scenario: Question references artefact deleted from Redis (edge case)
   - Behavior: CLI displays error: "Target artefact xyz-789 not found"
   - Recovery: Question cannot be answered (orphaned), human must create new goal
   - Test: Integration test (create Question, delete target, try to answer)

9. **Redis Connection Lost in `holt questions --wait`**:
   - Scenario: Redis unavailable while waiting for new Question
   - Behavior: CLI displays error, exits with code 1
   - Recovery: User retries when Redis is back
   - Test: Integration test (start watch, stop Redis container)

**Agent failures:**

10. **Agent Crash After Producing Question**:
    - Scenario: Agent produces Question, then container crashes before claim is terminated
    - Behavior: Orchestrator still processes Question (event already published), terminates claim normally
    - Recovery: Workflow continues, new claim created for target agent
    - Test: E2E test (agent produces Question, kill container immediately)

### **6.2. Concurrency considerations**

**Multiple Agents Question Same Artefact**:
- Scenario: Agent A and Agent B both question `spec-v1` simultaneously
- Behavior:
  - Both agents' claims are terminated independently
  - Two separate `pending_assignment` claims created for the spec author
  - Author receives first assignment, produces `spec-v2`
  - `spec-v2` satisfies first feedback claim
  - Author receives second assignment (still references `spec-v1`)
  - Author produces `spec-v3` answering second Question
- Safety: No race conditions, all claims are independent
- Test: E2E test with 2 agents simultaneously questioning same artefact

**Question During Feedback Loop**:
- Scenario: Agent receives feedback, starts rework, then questions the review feedback
- Behavior:
  - Agent's rework claim is terminated
  - Review agent receives `pending_assignment` for the Review artefact
  - Review agent produces clarified Review (new version)
  - Original agent receives updated review, continues rework
- Safety: Feedback loops can nest, iteration limits apply cumulatively
- Test: E2E test with review → question → clarified review

**Concurrent Questions and Answers**:
- Scenario: Agent A questions `spec-v1` while human is answering previous question about `goal-v1`
- Behavior: Independent workflows, no interference
- Safety: Each Question creates separate claim, no shared state
- Test: Integration test with simultaneous `holt answer` and agent Question

**Controller-Worker with Questions**:
- Scenario: Worker agent produces Question
- Behavior:
  - Worker's claim is terminated (normal flow)
  - Worker container exits
  - Orchestrator cleans up worker
  - Target agent receives `pending_assignment`
- Safety: Worker cleanup happens after Question is published, no data loss
- Test: E2E test with controller-worker agent producing Question

**Orchestrator Restart During Pending Question**:
- Scenario: Question creates `pending_assignment` claim, orchestrator crashes, restarts
- Behavior:
  - M3.5 recovery scans for `pending_assignment` claims
  - Claim is still active, waiting for answer artefact
  - When answer arrives, claim completion proceeds normally
- Safety: All state in Redis, no in-memory state lost
- Test: Integration test (create pending question claim, restart orchestrator, answer)

### **6.3. Edge case handling**

**Empty Question Text**:
- Scenario: Agent produces Question with `"question_text": ""`
- Behavior: Orchestrator accepts (payload is valid JSON), CLI displays empty string
- Justification: Agent may use Questions for general clarification requests
- Handling: No special logic, treat as valid Question

**Extremely Long Question Text**:
- Scenario: Agent produces Question with 10KB of text
- Behavior: Stored in Redis payload field (strings are unbounded), displayed by CLI
- Performance: Redis handles large strings efficiently, no impact
- Handling: No truncation or validation, accept as-is

**Question About Question**:
- Scenario: Agent receives Question artefact, produces another Question referencing it
- Behavior: Valid workflow, creates feedback loop on the Question itself
- Termination: Still subject to iteration limits (prevents infinite meta-questions)
- Handling: Treat as normal artefact, no special logic

**Answering Already-Answered Question**:
- Scenario: Human runs `holt answer abc-123 "text"` twice
- Behavior:
  - First execution creates version 2
  - Second execution creates version 3 (another increment)
  - Both versions are valid, each creates new claim
- Justification: Idempotency not critical for human operations
- Handling: No deduplication, allow multiple answers

**Question Targeting Non-Existent Role**:
- Scenario: Agent questions artefact with `produced_by_role: "deleted-agent"`
- Behavior:
  - Orchestrator creates `pending_assignment` for "deleted-agent"
  - No pup listening for that role
  - Claim waits indefinitely (same as "user" role)
- Resolution: Human must answer via CLI (treating role as "user")
- Handling: No special logic, Questions to non-existent roles wait forever

**Zero Questions Exist, `--since` Flag**:
- Scenario: User runs `holt questions --since 1h` when no Questions in range
- Behavior: Display message: "No unanswered questions found in the last 1 hour."
- Exit code: 0 (success, query completed)
- Handling: Empty result set is valid

**Terminal Artefact Before Question Answered**:
- Scenario: Workflow creates Terminal while `pending_assignment` claim waits
- Behavior:
  - Terminal triggers workflow completion
  - `pending_assignment` claim remains in Redis (orphaned)
  - "Unanswered" algorithm still shows it
- Justification: Questions are independent of Terminal (may answer for future workflows)
- Handling: No automatic cleanup, human can answer or ignore

**Multi-Byte UTF-8 in Question/Answer**:
- Scenario: Question text contains emoji, CJK characters, etc.
- Behavior: Go's JSON marshaling handles UTF-8 natively, stored correctly in Redis
- Display: CLI outputs UTF-8, terminal rendering depends on font support
- Handling: No special logic, full UTF-8 support

## **8. AI agent implementation guidance**

### **8.1. Development approach**

**Start with the simplest path**:
1. Implement orchestrator Question detection first (1 function, ~50 lines)
2. Reuse existing M3.3 termination/pending_assignment logic (no new code)
3. Implement `holt questions` without flags first (basic query + display)
4. Implement `holt answer` without flags first (basic replication + creation)
5. Add flags and advanced features incrementally

**Error handling from the start**:
- Wrap all JSON parsing in `json.Unmarshal` with error checks
- Check Redis key existence before loading artefacts
- Validate Question payload has required fields before processing
- Use Go's error wrapping (`fmt.Errorf("context: %w", err)`) for stack traces

**Test-Driven Development**:
- Write unit tests for Question payload parsing before implementing
- Write unit test for "unanswered" algorithm before implementing CLI query
- Write integration test for termination flow before connecting components
- Write E2E test for full Agent-to-Human flow before declaring feature complete

**Defensive programming**:
- Assume Question payloads may be malformed (don't crash on bad JSON)
- Assume target artefacts may be deleted (check existence)
- Assume Redis may be slow (use context timeouts)
- Assume users will provide wrong IDs (validate and give clear errors)

**Incremental validation**:
- After each implementation step, run full test suite to catch regressions
- Test in clean environment (new Holt instance, fresh Redis)
- Verify existing Phase 3 workflows still work (no breaking changes)

### **8.2. Common pitfalls to avoid**

**Pitfall 1: Forgetting M3.3 Logic Is Already There**
- Don't reimplement termination or pending_assignment logic
- Study `internal/orchestrator/feedback_loop.go` first
- Reuse `createFeedbackClaim()` function with Question in context

**Pitfall 2: Parsing JSON Incorrectly**
- Question payload is a JSON *string* (not a nested object)
- Must call `json.Unmarshal([]byte(artefact.Payload), &questionPayload)`
- Don't forget to handle `json.Unmarshal` errors

**Pitfall 3: Misunderstanding "Unanswered" Logic**
- "Unanswered" means no *newer version* with Question in sources
- Not "no Answer artefact exists" (Answer type is unused)
- Must query artefacts by logical_id, compare versions, check source_artefacts

**Pitfall 4: Breaking Orchestrator Recovery (M3.5)**
- All claim state must be in Redis (no in-memory tracking of Questions)
- Test orchestrator restart during pending question flow
- Verify M3.5 recovery tests still pass

**Pitfall 5: CLI ID Validation Off-by-One**
- Validation logic: `if questionCount >= 2 AND idLength < 6`
- Edge case: 1 question, 2-char ID → should succeed (no ambiguity)
- Edge case: 2 questions, 6-char ID → should succeed (meets minimum)

**Pitfall 6: Forgetting Multi-Line Text**
- Use `cmd.Flags().StringVar()` not `StringP()` for answer text (allows newlines)
- Don't trim or sanitize answer text (preserve exact input)
- Test with answer containing `\n` characters

**Pitfall 7: Watch Event Filtering**
- `human_input_required` is a workflow event, not artefact event
- Must subscribe to `workflow_events` channel, not `artefact_events`
- Test that watch displays Questions targeting "user" but not other roles

### **9.2. Rollback and disaster recovery**

**Feature can be disabled**:
- No configuration flag needed (Questions are opt-in by agents)
- If agents never produce Questions, feature is dormant
- No orchestrator flag to "turn off" Questions (they're just artefacts)

**Rollback procedure**:
1. Deploy previous orchestrator version (pre-M4.1)
2. Previous version ignores Question structural_type (no special handling)
3. Questions remain as artefacts but don't trigger feedback loops
4. Agents that produce Questions will see claims continue (not terminated)
5. Human-answered artefacts remain valid (just new versions)

**Data migration for rollback**:
- None required (Questions are standard artefacts, no special storage)
- Pending `pending_assignment` claims will wait indefinitely (agents won't produce answers)
- Recommendation: Terminate pending question claims manually before rollback:
  ```bash
  # Find pending question claims in Redis
  redis-cli --scan --pattern "holt:*:claim:*" | \
    xargs -I{} redis-cli HGET {} status | \
    grep pending_assignment

  # Manually terminate via Redis CLI or create Terminal artefacts
  ```

**Disaster recovery scenarios**:

**Scenario 1: Orchestrator Stuck on Question**
- Symptom: Workflow not progressing, `pending_assignment` claim visible
- Diagnosis: Check if target role agent is running (`holt list`, `docker ps`)
- Recovery:
  - If role="user": Human must run `holt answer`
  - If role=agent: Restart agent container, resend grant via orchestrator restart
  - If agent is deleted: Human can answer on agent's behalf

**Scenario 2: Question Payload Corrupted**
- Symptom: Orchestrator logs "failed to parse Question payload"
- Diagnosis: Check artefact payload in Redis (`holt hoard <id>`)
- Recovery:
  - Create new artefact with corrected payload
  - Manually create pending_assignment claim via Redis
  - Or create Terminal artefact to end workflow

**Scenario 3: Infinite Question Loop**
- Symptom: Iteration limit exceeded, Failure artefacts created
- Diagnosis: Check Question history (`holt hoard --since 1h | grep Question`)
- Recovery:
  - Analyze question pattern (same question repeatedly = agent bug)
  - Fix agent logic or provide clearer initial artefact
  - Restart workflow with new goal (`holt forage --goal "clearer text"`)

**Recovery time objectives**:
- Restart orchestrator to recover stuck workflow: <30 seconds (M3.5 recovery)
- Human answer to unblock workflow: <5 minutes (depends on human availability)
- Fix corrupted Question payload: <10 minutes (manual Redis intervention)
