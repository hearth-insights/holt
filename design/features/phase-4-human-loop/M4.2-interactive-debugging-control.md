# **Feature design: M4.2 - Interactive Debugging & Control**

**Purpose**: Enable human operators to interactively debug and control running Holt workflows through breakpoint-based pausing, inspection, and manual intervention.

**Scope**: Traditional debugger-style control for Holt workflows, allowing operators to pause, inspect, and manually intervene in agent workflows at runtime.

Associated phase: Human-in-the-Loop
Status: Draft

---

## **1. The 'why': goal and success criteria**

### **1.1. Goal statement**

To provide a breakpoint-based interactive debugging interface that allows human operators to pause Holt workflows at specific conditions, inspect state, and manually intervene (e.g., approve/reject reviews) without modifying agent code or restarting the orchestrator.

### **1.2. User story**

**As a workflow operator**, when debugging a complex multi-agent workflow, I want to set breakpoints on specific artefact types or claim states (e.g., "pause when a CodeCommit is created" or "pause on pending_review claims") so that I can inspect the current state of the workflow and manually approve or reject work before the agents continue. This debugging session must be ephemeral—when I disconnect, all breakpoints are cleared and the workflow resumes automatically to prevent accidental workflow stalling.

**As a compliance officer**, when monitoring a regulated workflow, I want to manually review and approve critical artefacts (like deployment specifications) before they proceed to the next phase, using the same debugging interface to pause the workflow, inspect the artefact, and make a decision.

**As a developer troubleshooting agent behavior**, I want to step through a workflow event-by-event to understand exactly when and why an agent makes specific decisions, without having to tail logs or write custom inspection scripts.

### **1.3. Success criteria**

1. **Interactive Debugging Session**: A human operator can run `holt debug -b artefact.type=CodeCommit` and the Orchestrator pauses when the next CodeCommit artefact is created. The operator sees a `(holt-debug)` prompt, can inspect the artefact with `print`, and resume with `continue`.

2. **Manual Review Intervention**: An operator can set a breakpoint with `break claim.status=pending_review`, and when a claim enters that state, can use `reviews` to list pending claims and `review <claim-id> --approve` or `review <claim-id> --reject "needs more tests"` to manually complete the review phase without involving any review agents.

3. **Safe Disconnect Behavior**: When the debugger client disconnects (either via `exit` command or network failure), the Orchestrator automatically clears all breakpoints and resumes normal operation. No workflows are left in a paused state after debugger disconnect.

4. **Event Stepping**: An operator can use `next` to single-step through orchestrator events one at a time, allowing precise observation of each claim phase transition, artefact creation, and bid collection.

5. **Orchestrator State Inspection**: The `print <artefact-id>` command displays full artefact details including payload, source chain, and metadata. The `breakpoints` command lists all active breakpoints with their IDs and conditions.

### **1.4. Non-goals**

- This milestone will **not** implement time-travel debugging or the ability to rewind workflow state.
- This milestone will **not** support modifying artefact payloads or claim state directly from the debugger (read-only inspection only, except for manual review commands).
- This milestone will **not** persist breakpoints across orchestrator restarts. All breakpoints are ephemeral per debugging session.
- This milestone will **not** implement a web-based UI. The debugging interface is CLI-only.
- This milestone will **not** support multiple simultaneous debugger connections to a single instance (only one active debug session allowed).
- This milestone will **not** implement conditional breakpoints beyond glob pattern matching (no expression evaluation like "pause if payload contains X").

## **2. The 'what': component impact analysis**

### **2.1. Blackboard changes**

**New/modified data structures:**
- No changes to core artefact or claim schemas.

**New Pub/Sub channels:**

1. **`holt:<instance>:debug:command`** - Command channel for CLI → Orchestrator communication
   - CLI publishes debug commands to this channel
   - Commands are JSON-encoded with schema:
   ```json
   {
     "command_type": "set_breakpoints|clear_breakpoint|continue|step_next|clear_all|inspect_artefact",
     "session_id": "unique-session-uuid",
     "payload": { /* command-specific data */ }
   }
   ```

2. **`holt:<instance>:debug:event`** - Event channel for Orchestrator → CLI communication
   - Orchestrator publishes debug events to this channel
   - Events are JSON-encoded with schema:
   ```json
   {
     "event_type": "paused_on_breakpoint|step_complete|breakpoint_set|session_active",
     "session_id": "unique-session-uuid",
     "payload": { /* event-specific data */ }
   }
   ```

**New Redis keys:**

1. **`holt:<instance>:debug:session`** - Active debug session metadata (Redis Hash)
   - Fields:
     - `session_id`: UUID of active session
     - `connected_at_ms`: Timestamp when session connected
     - `last_heartbeat_ms`: Timestamp of last heartbeat
     - `is_paused`: Boolean (true if orchestrator is currently paused)
   - TTL: 30 seconds (refreshed by CLI heartbeat every 5 seconds)

2. **`holt:<instance>:debug:breakpoints`** - Active breakpoints (Redis List)
   - Each entry is a JSON-encoded breakpoint:
   ```json
   {
     "id": "bp-1",
     "condition_type": "artefact.type|artefact.structural_type|claim.status|agent.role|event.type",
     "pattern": "glob-pattern"
   }
   ```
   - Cleared automatically when session key expires

3. **`holt:<instance>:debug:pause_context`** - Context when paused (Redis Hash)
   - Fields:
     - `paused_artefact_id`: ID of artefact that triggered breakpoint (if applicable)
     - `paused_claim_id`: ID of claim that triggered breakpoint (if applicable)
     - `breakpoint_id`: ID of breakpoint that was hit
     - `paused_at_ms`: Timestamp when paused
   - Cleared on continue/step

### **2.2. Orchestrator changes**

**New/modified logic:**

1. **Debug Session Detection**: On startup and periodically, check for active debug session key in Redis. Load session metadata and breakpoints into memory if present.

2. **Heartbeat Monitoring**: Monitor the debug session key's TTL. If it expires (debugger disconnected), automatically:
   - Clear all breakpoints from Redis
   - Resume normal operation (set `is_paused` to false)
   - Publish `session_disconnected` event

3. **Main Event Loop Modification**: Before processing each event in the orchestrator's main loop:
   ```go
   // Pseudocode
   for event := range events {
       if debugSessionActive() {
           // Check if this event matches any breakpoint
           if matchedBreakpoint := evaluateBreakpoints(event); matchedBreakpoint != nil {
               pauseOrchestrator(event, matchedBreakpoint)
               waitForDebugCommand() // Blocks here until continue/step
           }
       }
       processEvent(event) // Normal processing
   }
   ```

4. **Breakpoint Evaluation**: New function `evaluateBreakpoints()` that checks the current event context against all active breakpoints:
   - For artefact creation events: Match against `artefact.type`, `artefact.structural_type`
   - For claim state transitions: Match against `claim.status`
   - For agent grants: Match against `agent.role`
   - Use glob pattern matching (e.g., `*Spec`, `coder-*`)

5. **Pause State Management**: When paused:
   - Set `is_paused` flag in session key
   - Populate `pause_context` with current event details
   - Publish `paused_on_breakpoint` event to debug event channel
   - Block main event loop in a select statement waiting for:
     - `continue` command → Resume normal processing
     - `step_next` command → Process exactly one event, then pause again
     - Session key expiration → Auto-resume

6. **Command Processing**: Subscribe to debug command channel and handle:
   - `set_breakpoints`: Validate and store breakpoints in Redis list
   - `clear_breakpoint`: Remove breakpoint by ID from Redis list
   - `clear_all`: Clear all breakpoints
   - `continue`: Unblock main event loop
   - `step_next`: Unblock main event loop with single-step flag
   - `inspect_artefact`: Load artefact from blackboard and return via event channel
   - `terminate_claim`: Terminate currently paused claim with audit trail

7. **Manual Review Command**: New orchestrator logic to handle `manual_review` command:
   - Validate that orchestrator is currently paused on a `claim.status=pending_review` breakpoint
   - Validate that claim is in `pending_review` status
   - If `--approve`: Mark all granted review agents as having submitted approval reviews (empty payload)
   - If `--reject`: Create a Review artefact with rejection payload attributed to "user" role
   - Transition claim to next phase or trigger feedback loop (reuse M3.3 logic)
   - Publish `review_complete` event

8. **Terminate Claim Command**: New orchestrator logic to handle `terminate_claim` command:
   - Validate that orchestrator is currently paused on a claim (pause context contains ClaimID)
   - Validate that claim exists and is not already terminated or completed
   - Set claim status to `terminated`
   - Set termination reason with session ID, timestamp, and "MANUAL TERMINATION" prefix
   - Clean up orchestrator internal state:
     - Remove from `phaseStates` map
     - Remove from `pendingAssignmentClaims` map
   - Create structured log event with "error" level for high visibility
   - Auto-resume orchestrator (send resume signal)
   - Publish `claim_terminated` event for audit trail

9. **Session Concurrency Control**: Only allow one active debug session at a time. If a session key exists with a recent heartbeat, reject new `holt debug` attempts with error: "Debug session already active (started at <timestamp>)".

**New/modified configurations (holt.yml):**
- No changes to `holt.yml` required. Debugging is a runtime operation, not a configuration.

### **2.3. Agent pup changes**

**New/modified logic:**
- No changes required. Agents are unaware of debugging sessions. When the orchestrator is paused, agents simply wait for new claim grants (normal behavior).

**Changes to the tool execution contract (stdin/stdout):**
- No changes required.

### **2.4. CLI changes**

**New commands:**

1. **`holt debug [flags]`**

   **Purpose**: Attach an interactive debugger to a running Holt instance.

   **Flags**:
   - `--name <instance>` - Target instance name (default: infer from workspace)
   - `--break <condition>` (alias `-b`) - Set breakpoint on startup (repeatable)

   **Behavior**:
   1. **Session Initialization**:
      - Generate unique `session_id` UUID
      - Check for existing active session in Redis
      - If session exists: Error with message "Debug session already active"
      - Create new session key in Redis with 30-second TTL
      - Write session metadata (session_id, connected_at_ms)

   2. **Heartbeat Loop**:
      - Launch background goroutine to refresh session key TTL every 5 seconds
      - Stop heartbeat on exit or disconnect

   3. **Breakpoint Initialization**:
      - For each `--break` flag value, parse condition and send `set_breakpoints` command
      - Wait for confirmation events from orchestrator

   4. **Interactive Prompt**:
      - Display `(holt-debug)` prompt using `github.com/c-bata/go-prompt` library
      - Auto-completion for commands and artefact IDs
      - Command history support

   5. **Event Listener**:
      - Subscribe to debug event channel in background goroutine
      - Display events to user (breakpoint hits, step completions)
      - Update internal state (current pause context, breakpoint list)

   6. **Graceful Shutdown**:
      - On `exit` command or Ctrl+C: Send `clear_all` command, delete session key, exit
      - Session key expiration acts as dead-man's switch for network failures

   **Example Usage**:
   ```bash
   # Basic session
   holt debug

   # Pre-set breakpoints
   holt debug -b artefact.type=CodeCommit -b claim.status=pending_review

   # Target specific instance
   holt debug --name my-workflow
   ```

**Interactive Session Commands:**

The `(holt-debug)` prompt supports the following commands:

1. **`continue` (alias: `c`)**
   - Sends `continue` command to orchestrator
   - Resumes event loop until next breakpoint hit
   - Displays: "Continuing..."

2. **`next` (alias: `n`)**
   - Sends `step_next` command to orchestrator
   - Processes exactly one event, then pauses again
   - Displays: "Stepped. Current event: <event-type>"

3. **`break <condition>` (alias: `b`)**
   - Parses condition string (e.g., `artefact.type=*Spec`)
   - Sends `set_breakpoints` command with new breakpoint
   - Displays: "Breakpoint bp-3 set on artefact.type=*Spec"

   **Supported condition formats**:
   - `artefact.type=<glob>` - Match artefact type field
   - `artefact.structural_type=<type>` - Match structural type (Question, Review, etc.)
   - `claim.status=<status>` - Match claim status (pending_review, pending_exclusive, etc.)
   - `agent.role=<glob>` - Match agent role on grant (e.g., `coder-*`)
   - `event.type=<event_name>` - Match specific orchestrator event types

4. **`breakpoints` (alias: `bp`)**
   - Queries Redis for current breakpoint list
   - Displays formatted list:
   ```
   Active Breakpoints:
     1. bp-1: artefact.type=CodeCommit
     2. bp-2: claim.status=pending_review
   ```

5. **`clear <id>`**
   - Sends `clear_breakpoint` command with breakpoint ID
   - Displays: "Breakpoint bp-2 cleared"

6. **`print [artefact-id]` (alias: `p`)**
   - Without ID: Displays artefact that triggered current breakpoint (from pause_context)
   - With ID: Sends `inspect_artefact` command, displays returned artefact
   - Format:
   ```
   Artefact abc-123-def-456
     Type: CodeCommit
     Structural Type: Standard
     Produced By: coder-agent
     Version: 1
     Payload: git:a1b2c3d4e5f6...
     Source Artefacts: [xyz-789]
     Created: 2025-01-15 14:32:10 UTC
   ```

7. **`reviews`**
   - Queries Redis for all claims in `pending_review` status
   - Displays list:
   ```
   Pending Reviews:
     1. claim-abc123 (artefact: DesignSpec v1)
        Granted to: [architect-reviewer, security-reviewer]
        Waiting for: 2 reviews
     2. claim-def456 (artefact: CodeCommit v1)
        Granted to: [code-reviewer]
        Waiting for: 1 review
   ```

8. **`review <claim-id> [--approve | --reject "<text>"]`**
   - Validates that orchestrator is paused on `claim.status=pending_review`
   - Validates that provided claim-id matches current pause context
   - Sends `manual_review` command with approve/reject decision
   - Orchestrator completes review phase and transitions claim
   - Displays: "Review approved. Claim transitioning to pending_parallel." or "Review rejected. Feedback claim created for <role>."

   **Context-aware validation**:
   - Command fails if not paused on a review breakpoint
   - Command fails if claim-id doesn't match paused claim
   - Must provide exactly one of `--approve` or `--reject`

9. **`help`**
   - Displays command reference with aliases and examples

10. **`exit`**
    - Sends `clear_all` command to orchestrator
    - Deletes session key from Redis
    - Gracefully exits debug session
    - Displays: "Debug session ended. Breakpoints cleared."

11. **`forage --goal "<text>"`**
    - Creates a new `GoalDefined` artefact on the blackboard, starting a new workflow.
    - Uses the debugger's existing blackboard client.
    - `produced_by_role` is set to "user".
    - Displays: "✓ GoalDefined artefact created: <artefact-id>"
    - This removes the need for a separate terminal to start a workflow while debugging.

12. **`terminate` (alias: `kill`)**
    - Forcibly terminates the currently paused claim
    - Only available when paused on a claim
    - Creates permanent termination record with session ID and timestamp
    - Sets claim status to `terminated` with detailed reason
    - Cleans up orchestrator internal state (phaseStates, pendingAssignmentClaims)
    - Auto-resumes workflow after termination
    - Displays warning before execution with clear impact explanation
    - **Use case**: Emergency intervention for runaway or problematic claims
    - **Audit trail**: Creates high-visibility log event with "error" level for compliance tracking

    **Context-aware validation**:
    - Command fails if not paused on a claim (requires ClaimID in pause context)
    - Command fails if claim already terminated or completed
    - Shows detailed warning before executing (irreversible action)

**Changes to user output:**
- No changes to existing commands. The `holt debug` command is entirely new.

## **3. The 'how': implementation & testing plan**

### **3.1. Key design decisions & risks**

**Key Decision 1: Ephemeral Breakpoints with Dead-Man's Switch**

Breakpoints are stored in Redis and tied to a TTL-based session key. When the session key expires (either due to explicit disconnect or network failure), breakpoints are automatically cleared by Redis (via EXPIRE). This decision:
- Prevents workflows from hanging indefinitely if debugger crashes
- Eliminates need for explicit cleanup in all failure scenarios
- Leverages Redis TTL as a reliable dead-man's switch
- Requires debugger to send heartbeats every 5 seconds to maintain session

**Alternative Considered**: Persistent breakpoints that survive debugger disconnect. Rejected because it creates too much risk of accidental workflow stalling and requires complex state management for cleanup.

**Key Decision 2: Orchestrator Pause in Main Event Loop**

The orchestrator's main event loop is modified to check for breakpoints before processing each event and block when paused. This decision:
- Provides precise control over workflow execution
- Allows single-stepping through events
- Maintains orchestrator state consistency (no mid-event pausing)
- Blocks the entire event loop (all claims pause, not just one)

**Alternative Considered**: Per-claim pausing with selective blocking. Rejected because it adds significant complexity and makes reasoning about system state much harder for operators.

**Key Decision 3: Pub/Sub Communication Protocol**

The debugger CLI and orchestrator communicate via Redis Pub/Sub channels rather than HTTP endpoints. This decision:
- Leverages existing Redis infrastructure (no new server required)
- Provides asynchronous, event-driven communication
- Allows orchestrator to push events to CLI without polling
- Consistent with existing Holt architecture (artefact/claim events)

**Alternative Considered**: HTTP-based REST API on orchestrator. Rejected because it requires orchestrator to run an HTTP server, adds dependencies, and doesn't fit Holt's event-driven model.

**Key Decision 4: Context-Aware Manual Review Command**

The `review` command only works when paused on a `claim.status=pending_review` breakpoint. This decision:
- Prevents accidental review submissions for wrong claims
- Makes manual intervention explicit and traceable
- Ensures operator understands context before making decision
- Reduces risk of workflow corruption from mistakes

**Alternative Considered**: Allow review command at any time with claim-id lookup. Rejected because it's too easy for operators to make mistakes when not actively paused on the review.

**Key Decision 5: Single Active Session Only**

Only one debugger can attach to an instance at a time. This decision:
- Prevents conflicting commands from multiple operators
- Simplifies session state management
- Makes breakpoint ownership clear
- Reduces risk of workflow corruption

**Alternative Considered**: Multi-user debugging with session isolation. Rejected as YAGNI for V1—adds significant complexity and unclear use case.

**Risks:**

1. **Orchestrator Performance Degradation**: Checking breakpoints on every event could slow down the orchestrator.
   - **Mitigation**: Breakpoint evaluation is O(n) where n = number of breakpoints. Limit to 100 breakpoints per session. Benchmark with 100 breakpoints to ensure <10ms overhead per event.

2. **Deadlock from Debugger Crash**: If debugger crashes without clearing session key, workflow hangs until TTL expires (30 seconds).
   - **Mitigation**: 30-second TTL is acceptable for debugging use case. Document this behavior. Consider adding `holt debug --force-disconnect` command to manually clear stuck sessions.

3. **Race Condition on Session Creation**: Two `holt debug` commands starting simultaneously could both create sessions.
   - **Mitigation**: Use Redis `SET NX` (set if not exists) for atomic session creation. Second command will fail gracefully.

4. **Manual Review Bypassing Audit Trail**: Manual reviews don't create Review artefacts from configured review agents.
   - **Mitigation**: Create Review artefact attributed to "user" role with payload containing manual decision. Maintains full audit trail.

5. **Breakpoint Pattern Complexity**: Complex glob patterns could cause unexpected pauses.
   - **Mitigation**: Use Go's `filepath.Match` for glob evaluation (well-tested, limited syntax). Document supported patterns clearly. Validate patterns on breakpoint creation and reject invalid syntax.

### **3.3. Performance & resource considerations**

**Resource usage:**
- Debug session key: ~200 bytes in Redis
- Breakpoint storage: ~100 bytes per breakpoint × max 100 = 10KB max
- Pause context: ~500 bytes when paused
- Pub/Sub subscription: 2 additional channels (negligible overhead)
- **Total impact**: ~11KB per active debug session

**Scalability limits:**
- Only one debug session per instance (by design)
- Maximum 100 breakpoints per session (enforced)
- Breakpoint evaluation: O(n) per orchestrator event where n ≤ 100
- Expected overhead: <10ms per event with 100 breakpoints (glob matching is fast)

**Performance requirements:**
- Breakpoint evaluation: <10ms per event (measured with 100 active breakpoints)
- Command latency: <100ms from CLI command to orchestrator response
- Event delivery: <50ms from orchestrator pause to CLI notification
- Heartbeat overhead: Negligible (<1ms every 5 seconds)

**Latency impact:**
- When no debug session active: Zero overhead (no breakpoint checks)
- When debug session active but not paused: <10ms per orchestrator event
- When paused: Workflow completely blocked (intentional for debugging)

**Concurrency impact:**
- Orchestrator main event loop blocks when paused (all claims affected)
- Agents continue running but won't receive new grants
- Worker containers for scaled agents may time out if paused too long (acceptable for debugging)

## **4. Principle compliance check**

### **4.1. YAGNI (You Ain't Gonna Need It)**

**Third-party dependencies:**
- **New**: `github.com/c-bata/go-prompt` - Interactive prompt library for CLI
  - **Justification**: Building a full interactive prompt with auto-completion, history, and command parsing from scratch would be a significant undertaking. This library is mature, well-maintained (7k+ stars), and provides exactly the features needed for the debugger interface. It's the de facto standard for building CLI prompts in Go.

**Complexity justification:**
- Debugging is essential for production workflows. Operators need to inspect and control complex multi-agent workflows.
- However, we avoid over-engineering by:
  - Not implementing time-travel debugging (YAGNI for V1)
  - Not implementing expression evaluation in breakpoints (glob matching sufficient)
  - Not implementing multi-user debugging (single session adequate)
  - Not implementing web UI (CLI sufficient for operator use case)
  - Not persisting breakpoints across restarts (ephemeral is safer)

### **4.2. Auditability**

This feature **maintains and enhances** auditability:

**Complete Audit Trail Preserved:**
- Manual review decisions create Review artefacts with "user" role (same as agent reviews)
- Manual approvals create empty-payload Review artefacts (same as approval from agents)
- Manual rejections create Review artefacts with rejection payload
- All manual interventions are timestamped and immutable
- Debugger actions are logged as structured JSON events

**Traceable Manual Intervention:**
- Every manual review creates an artefact on the blackboard
- Future auditors can see exactly when and why a human intervened
- Debugger session start/stop logged to orchestrator logs
- Breakpoint hits logged with context (what triggered pause, when)

**Compliance Value:**
- For regulated industries, manual review provides required human oversight
- Audit trail shows WHO made decision (user role) and WHEN
- Review payloads capture REASON for approval/rejection
- Full transparency into manual vs. automated decisions

**No Audit Trail Gaps:**
- Pausing workflow doesn't create gaps (all events still logged)
- Inspection commands are read-only (don't modify state)
- Manual reviews use same artefact creation flow as agent reviews
- Session expiration logged (explains why workflow resumed)

### **4.3. Small, single-purpose components**

**Component responsibilities remain clear:**

- **Orchestrator**: Adds debug session awareness and breakpoint evaluation, but core orchestration logic unchanged. Debug logic is isolated in `internal/orchestrator/debug/` package.
- **CLI**: New `holt debug` command is self-contained, doesn't modify existing commands. Interactive prompt logic isolated in `cmd/holt/commands/debug/` package.
- **Blackboard**: No changes to core data structures. Debug uses new Pub/Sub channels and ephemeral Redis keys.
- **Agents**: Completely unaware of debugging. No changes to pup or tool contract.

**No tight coupling introduced:**
- Debug logic is additive, not invasive (can be disabled by not starting session)
- Orchestrator's core event processing unchanged (debug is pre-processing hook)
- CLI debug command doesn't depend on other commands
- Pub/Sub protocol is standalone (doesn't affect artefact/claim events)

**Isolation of debug logic:**
- New packages: `internal/orchestrator/debug/`, `cmd/holt/commands/debug/`
- Clear separation from core orchestration logic
- Debug features can be tested independently
- Debug can be extended without touching core components

### **4.4. Security considerations**

**Access Control:**
- No authentication on debug session (same Redis access model as all Holt operations)
- Whoever can access Redis can debug workflows
- **Risk**: Malicious actor could pause production workflows
- **Mitigation**: Document that Redis should be network-isolated (same security model as existing Holt architecture)
- **Future Enhancement**: Consider adding session tokens for debug access control (out of scope for M4.2)

**Data Exposure:**
- Debug commands can inspect any artefact payload (including sensitive data)
- Manual review sees full artefact content
- **Risk**: Debugger logs could leak sensitive information
- **Mitigation**: Don't log artefact payloads in orchestrator debug logs. Payloads only sent to CLI via Pub/Sub (same exposure as existing `holt hoard` command)

**Denial of Service:**
- Debugger can pause workflow indefinitely (if session maintained)
- Breakpoints on high-frequency events could degrade performance
- **Risk**: Accidental or malicious workflow stalling
- **Mitigation**:
  - Session TTL acts as timeout (max 30 seconds stalled if heartbeat stops)
  - Limit to 100 breakpoints per session
  - Document best practices (pause only when needed, use specific breakpoints)

**Command Injection:**
- Breakpoint patterns are glob strings (could contain special characters)
- **Risk**: Pattern parsing could be exploited
- **Mitigation**: Use Go's standard `filepath.Match` (safe, no shell evaluation). Validate patterns on input and reject invalid syntax.

**Privilege Boundaries:**
- Manual review as "user" role is a special privilege (bypasses agent logic)
- **Risk**: Operator could approve work that should fail review
- **Mitigation**: This is intentional—debugging requires trust. Document that manual review should be used carefully. Audit trail shows all manual interventions.

**Container Isolation:**
- No changes to agent containers (debug is orchestrator-only)
- No new container-to-container communication
- No privilege escalation required

### **4.5. Backward compatibility**

**Additive changes only:**
- No modifications to existing artefact or claim schemas
- New Pub/Sub channels don't interfere with existing channels

**No breaking changes:**

**Deprecation path:**

**Migration requirements:**

## **6. Error scenarios & edge cases**

### **6.1. Failure modes**

**Orchestrator failures:**

1. **Malformed Debug Command**:
   - Scenario: CLI sends command with invalid JSON or missing required fields
   - Behavior: Orchestrator logs error, ignores command, doesn't crash
   - Recovery: CLI shows no response, can retry with corrected command
   - Test: Integration test with malformed JSON payloads

2. **Invalid Breakpoint Pattern**:
   - Scenario: CLI sends breakpoint with invalid glob pattern (e.g., unclosed bracket)
   - Behavior: Orchestrator validates pattern with `filepath.Match`, returns error event
   - Recovery: CLI displays "Invalid breakpoint pattern: <error>", user corrects
   - Test: Unit test for pattern validation with various invalid patterns

3. **Session Key Expired During Command**:
   - Scenario: Heartbeat fails, session expires while orchestrator processes command
   - Behavior: Orchestrator detects expired session, ignores command, publishes `session_expired` event
   - Recovery: CLI receives expiration event, exits gracefully
   - Test: Integration test (send command immediately after stopping heartbeat)

4. **Breakpoint Hit on Invalid Artefact**:
   - Scenario: Breakpoint matches artefact ID that doesn't exist in Redis (deleted or corrupted)
   - Behavior: Orchestrator logs error, skips pause, continues processing
   - Recovery: Workflow continues normally, operator sees log message
   - Test: Integration test (create breakpoint, delete artefact, trigger)

5. **Manual Review on Non-Review Claim**:
   - Scenario: Operator sends `manual_review` command when paused on artefact creation (not review)
   - Behavior: Orchestrator validates context, returns error: "Not paused on review claim"
   - Recovery: CLI displays error, operator can only use review command on review breakpoints
   - Test: E2E test (pause on artefact, attempt review command)

6. **Redis Connection Lost During Pause**:
   - Scenario: Redis unavailable while orchestrator is paused waiting for continue command
   - Behavior: Orchestrator's existing M3.5 recovery logic applies. On reconnect, checks for session key. If expired, resumes. If still valid, continues waiting.
   - Recovery: Automatic on Redis reconnection
   - Test: Integration test with testcontainers (pause, stop Redis, restart Redis)

**CLI failures:**

7. **Session Already Active**:
   - Scenario: User runs `holt debug` while another session is active
   - Behavior: CLI attempts SET NX, fails, displays error: "Debug session already active (started at <timestamp>)"
   - Recovery: User waits for existing session to end or manually deletes session key
   - Test: Integration test (start two debuggers simultaneously)

8. **Heartbeat Goroutine Crash**:
   - Scenario: Heartbeat goroutine panics due to Redis connection issue
   - Behavior: Recover from panic, log error, stop heartbeat (session will expire)
   - Recovery: Session expires after 30 seconds, orchestrator auto-resumes
   - Test: Unit test with mock Redis that returns errors

9. **Invalid Artefact ID in Print Command**:
   - Scenario: User runs `print abc-123` with non-existent artefact ID
   - Behavior: CLI sends inspect command, orchestrator returns error event
   - Recovery: CLI displays "Artefact abc-123 not found"
   - Test: E2E test (run print with random UUID)

10. **Network Partition During Active Session**:
    - Scenario: Network partition between CLI and Redis
    - Behavior: Heartbeat fails, session key expires after 30 seconds
    - Recovery: Orchestrator auto-resumes, CLI eventually reconnects and sees session expired
    - Test: Integration test (simulate network partition with Redis proxy)

**Agent failures:**

11. **Workflow Paused During Worker Execution**:
    - Scenario: Controller-worker agent has workers running when debugger pauses orchestrator
    - Behavior: Workers continue executing (unaffected by orchestrator pause), complete normally
    - Recovery: When orchestrator resumes, it processes worker completion as normal
    - Test: E2E test (launch worker, pause orchestrator, verify worker completes)

### **6.2. Concurrency considerations**

**Multiple Breakpoints Hit Simultaneously**:
- Scenario: Single event matches multiple breakpoints (e.g., artefact.type=CodeCommit AND agent.role=coder)
- Behavior: Orchestrator evaluates breakpoints in order added, pauses on first match
- Safety: No race conditions, deterministic ordering
- Test: Integration test (set two breakpoints, verify first wins)

**Debugger Disconnect During Event Processing**:
- Scenario: Session expires while orchestrator is in middle of processing an event (after pause check, before completion)
- Behavior: Event completes processing (not interrupted mid-event). Next event checks for session, sees it expired, continues normally.
- Safety: Events are atomic, never interrupted mid-processing
- Test: Integration test (trigger long-running event, expire session mid-event)

**Manual Review During Agent Review**:
- Scenario: Review agents are producing reviews while operator uses manual review
- Behavior: Manual review creates Review artefact alongside agent reviews. All reviews collected normally. M3.2 logic applies (any rejection triggers feedback loop).
- Safety: Manual and agent reviews are independent artefacts, both valid
- Test: E2E test (configure review agent, pause on pending_review, submit manual review, verify agent review also processed)

**Concurrent Continue and Step Commands**:
- Scenario: Operator sends `continue` and `next` commands in quick succession
- Behavior: Commands processed in order received (Pub/Sub maintains order). First command executes, second command takes effect on next pause.
- Safety: Pub/Sub provides ordering guarantees, no race
- Test: Integration test (send rapid commands, verify execution order)

**Orchestrator Restart During Paused State**:
- Scenario: Orchestrator crashes while paused on breakpoint
- Behavior: M3.5 recovery loads active claims from Redis, but debug session key expires during restart (30 seconds). Orchestrator resumes normal operation.
- Safety: Ephemeral session ensures no permanent pause state. Workflow always resumes.
- Test: Integration test (pause orchestrator, kill process, restart, verify auto-resume)

**Heartbeat Timing Edge Cases**:
- Scenario: Heartbeat sent at exactly same time as session expires (race condition)
- Behavior: Redis EXPIRE is atomic. Either heartbeat succeeds (session extended) or expires (session gone). No intermediate state.
- Safety: Redis atomicity guarantees prevent race
- Test: Integration test with controlled timing (stop heartbeat, send single refresh at T+29s, verify outcome)

### **6.3. Edge case handling**

**Zero Breakpoints Set**:
- Scenario: User starts `holt debug` without `--break` flags and never sets breakpoints
- Behavior: Valid debugging session, orchestrator runs normally (no pauses)
- Handling: CLI can still use `print`, `reviews` commands for inspection
- Test: E2E test (start debug session, use print command, no pauses)

**Breakpoint on Every Event**:
- Scenario: User sets `break event.type=*` (matches all events)
- Behavior: Orchestrator pauses on every single event
- Justification: Valid use case for deep debugging
- Handling: Operator must manually continue/step through every event (tedious but functional)
- Test: Integration test (set wildcard breakpoint, verify pauses on all events)

**Extremely Long Pause (Hours)**:
- Scenario: Operator pauses workflow, goes to lunch, leaves for hours
- Behavior: Heartbeat continues refreshing session key. Workflow remains paused indefinitely.
- Justification: Intentional—debugging may require extended inspection
- Handling: Orchestrator logs warning after 10 minutes of pause. Session remains valid.
- Test: Integration test (pause, wait 15 minutes, verify still paused)

**Manual Review Claim Mismatch**:
- Scenario: Operator pauses on claim-A, tries to review claim-B
- Behavior: Context validation detects mismatch, returns error: "Claim <claim-B> is not the currently paused claim"
- Handling: Operator must review the claim shown in pause context
- Test: Integration test (pause on one claim, send review for different claim ID)

**Print Command on Knowledge Artefact (M4.3)**:
- Scenario: User runs `print <knowledge-id>` on a Knowledge artefact (structural_type: Knowledge)
- Behavior: Print displays artefact normally (Knowledge is valid artefact type)
- Handling: No special logic needed, treat as any other artefact
- Test: E2E test with M4.3 Knowledge artefacts

**Breakpoint Pattern with Special Characters**:
- Scenario: User sets breakpoint with pattern containing `[`, `]`, `*`, `?`
- Behavior: Go's `filepath.Match` handles these correctly (standard glob syntax)
- Handling: Validate pattern on input, reject if `filepath.Match` returns error
- Test: Unit test with various special character patterns

**Session Key Deleted Manually**:
- Scenario: Operator manually deletes session key via Redis CLI while debugging
- Behavior: Next heartbeat fails to refresh (key doesn't exist), CLI detects and exits
- Recovery: CLI displays "Session lost (key deleted)", exits gracefully
- Test: Integration test (start session, manually delete key, verify CLI exits)

**Breakpoint on Terminal Artefact**:
- Scenario: User sets `break artefact.structural_type=Terminal`
- Behavior: Orchestrator pauses when Terminal artefact created (before marking workflow complete)
- Justification: Useful for inspecting final workflow state
- Handling: After continue, workflow completes normally
- Test: E2E test (set Terminal breakpoint, verify pause, continue, verify workflow completion)

**Multiple Claims in Pending Review Simultaneously**:
- Scenario: Multiple artefacts trigger review phase at same time, all in pending_review
- Behavior: `reviews` command lists all pending review claims. Manual review targets one claim. Other claims wait for agent reviews.
- Handling: Manual review is per-claim, doesn't affect other claims
- Test: E2E test (create two review-requiring artefacts, manually review one, verify other continues independently)

**UTF-8 in Review Rejection Text**:
- Scenario: Operator provides review rejection with emoji or CJK characters
- Behavior: Go's JSON marshaling handles UTF-8 natively, stored correctly in Review artefact payload
- Handling: No special logic needed, full UTF-8 support
- Test: E2E test (manual review with emoji in rejection text)

## **8. AI agent implementation guidance**

### **8.1. Development approach**

**Start with the simplest path**:
1. Implement orchestrator session detection first (load from Redis on startup)
2. Implement basic breakpoint evaluation (simple glob matching)
3. Implement pause logic (block on channel until continue)
4. Implement CLI session creation and heartbeat (establish connection)
5. Implement basic interactive prompt (go-prompt integration)
6. Implement commands incrementally (start with continue, then next, then break)
7. Add manual review as final feature (most complex)

**Error handling from the start**:
- Wrap all Redis operations in error checks (connection may fail)
- Wrap all JSON parsing in error checks (commands may be malformed)
- Wrap all go-prompt operations in panic recovery (terminal issues)
- Validate all breakpoint patterns before storing (filepath.Match may return error)
- Validate session_id on all commands (prevent command spoofing)

**Test-Driven Development**:
- Write unit tests for breakpoint evaluation before implementing
- Write integration test for session creation before implementing CLI
- Write E2E test for full debug flow before declaring feature complete
- Test failure modes explicitly (don't just test happy path)

**Defensive programming**:
- Assume Redis may be slow or unavailable (use context timeouts)
- Assume debugger may crash (TTL acts as cleanup)
- Assume users will provide invalid input (validate everything)
- Assume network may partition (heartbeat must be resilient)
- Assume orchestrator may restart during pause (recovery must work)

**Incremental validation**:
- After each implementation step, run full test suite
- Test in clean environment (new Holt instance, fresh Redis)
- Verify existing Phase 3 workflows still work (no breaking changes)
- Benchmark breakpoint evaluation overhead (must be <10ms)

### **8.2. Common pitfalls to avoid**

**Pitfall 1: Forgetting Session Expiration Cleanup**
- Don't just delete session key—also clear breakpoints and pause context
- Session expiration must be atomic (use Redis MULTI/EXEC if needed)
- Test: Integration test that verifies all keys cleaned up on expiration

**Pitfall 2: Blocking Orchestrator Indefinitely**
- Pause logic must select on both continue command AND session expiration
- If session expires while paused, must auto-resume (don't wait forever)
- Test: Integration test (pause, stop heartbeat, verify auto-resume after 30s)

**Pitfall 3: Race on Session Creation**
- Must use Redis SET NX to atomically check-and-create session
- Can't check existence then create (race condition window)
- Test: Integration test with concurrent session creation attempts

**Pitfall 4: Heartbeat Goroutine Leaks**
- Must stop heartbeat goroutine on CLI exit (use context cancellation)
- Must clean up if CLI crashes (defer cleanup in main)
- Test: Unit test that verifies goroutine stops on context cancel

**Pitfall 5: Manual Review Without Artefact Creation**
- Manual review must create Review artefact (maintain audit trail)
- Must use "user" role for produced_by_role
- Must include rejection text in payload if rejecting
- Test: E2E test that verifies Review artefact exists after manual review

**Pitfall 6: Breakpoint Evaluation Performance**
- Don't load artefacts from Redis for every breakpoint check (use event metadata)
- Cache breakpoint list in memory (don't query Redis every event)
- Benchmark: Ensure <10ms overhead with 100 breakpoints
- Test: Performance test (create 100 breakpoints, process 1000 events, measure latency)

**Pitfall 7: go-prompt Terminal Compatibility**
- go-prompt requires TTY (doesn't work in non-interactive environments)
- Must detect TTY and fail gracefully if not available
- Document: "holt debug requires interactive terminal"
- Test: Test in non-TTY environment (e.g., CI pipeline), verify graceful error

**Pitfall 8: Pub/Sub Message Ordering**
- Don't assume message order across different channels
- Commands and events may arrive out of order
- Use session_id to correlate commands with responses
- Test: Integration test with rapid command sequences

### **9.2. Rollback and disaster recovery**

**Feature can be disabled**:
- No configuration flag needed (debugging is opt-in)
- If no one runs `holt debug`, feature is dormant (zero overhead)
- Can't disable debug while session active (must wait for expiration or exit)

**Rollback procedure**:
1. Deploy previous orchestrator version (pre-M4.2)
2. Previous version ignores debug commands (no debug logic)
3. Active debug sessions become ineffective (commands ignored)
4. Breakpoints in Redis are harmless (orphaned keys, eventually expire)
5. CLI debugger receives no events (orchestrator doesn't publish)
6. Workflows run normally (no pausing)

**Data migration for rollback**:
- None required (all debug state is ephemeral)
- Orphaned debug keys will expire via TTL (30 seconds)
- No persistent state changes from debugging
- Recommendation: Manually exit active debug sessions before rollback

**Disaster recovery scenarios**:

**Scenario 1: Orchestrator Stuck in Paused State**
- Symptom: Workflow not progressing, orchestrator logs show "paused_on_breakpoint"
- Diagnosis: Check for active debug session (`holt debug` running)
- Recovery:
  - Primary: Debugger operator sends `continue` or `exit` command
  - Fallback: Manually delete session key via Redis CLI: `DEL holt:<instance>:debug:session`
  - Orchestrator auto-resumes within 1 second of key deletion

**Scenario 2: Debugger Crashed, Workflow Hung**
- Symptom: No debugger running, but workflow paused
- Diagnosis: Check session key TTL: `TTL holt:<instance>:debug:session`
- Recovery:
  - Wait for TTL expiration (max 30 seconds)
  - Or manually delete session key: `DEL holt:<instance>:debug:session`
  - Orchestrator auto-resumes immediately

**Scenario 3: Manual Review Corrupted Workflow**
- Symptom: Workflow terminated unexpectedly after manual review
- Diagnosis: Check orchestrator logs for manual_review events, inspect Review artefact
- Recovery:
  - Analyze Review artefact (produced_by_role="user")
  - If invalid: No automated recovery (manual intervention in history)
  - Prevention: Document manual review best practices, train operators

**Scenario 4: Breakpoint Pattern Causing Infinite Pauses**
- Symptom: Orchestrator pauses on every event (e.g., `event.type=*`)
- Diagnosis: Check breakpoint list: `LRANGE holt:<instance>:debug:breakpoints 0 -1`
- Recovery:
  - Debugger operator clears problematic breakpoint: `clear bp-1`
  - Or deletes all breakpoints: `exit` command
  - Or manually: `DEL holt:<instance>:debug:breakpoints`

**Recovery time objectives**:
- Auto-resume from crashed debugger: <30 seconds (TTL expiration)
- Manual session key deletion: <5 seconds (immediate resume)
- Operator-initiated continue: <1 second (command latency)
- Rollback to pre-M4.2 orchestrator: <5 minutes (standard deployment)
