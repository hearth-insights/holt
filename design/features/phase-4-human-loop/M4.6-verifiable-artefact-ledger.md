# **Feature design: Verifiable artefact ledger**

**Purpose**: Cryptographically tamper-evident audit trail using content-addressable Merkle DAG
**Scope**: Phase 4 deliverable - Production-ready security for regulated industries
**Estimated tokens**: ~8,000 tokens

Associated phase: Human-in-the-Loop
Status: Implementation Ready - Clarifications Confirmed (2025-11-26)

**Implementation Decisions** (confirmed with human architect):
- **Security Alert Persistence**: Alerts persist in Redis List (`holt:{instance}:security:alerts:log`) + Pub/Sub channel
- **Workflow Pause Mechanism**: Global lockdown via `holt:{instance}:security:lockdown` key, halts entire orchestrator
- **Lockdown Recovery**: Audited CLI command `holt security --unlock --reason "<text>"` creates override event
- **ContextForRoles**: Included in ArtefactHeader hash computation, uses `omitempty` (excluded if empty)
- **Orphan Block Action**: Triggers global lockdown (exception: root artefacts with `ParentHashes: []`)
- **LogicalThreadID Inheritance**: V1=new UUID, V2+=inherit from parent
- **Naming Convention**: `ParentHashes` (not `SourceArtefacts`) - aligns with Merkle DAG semantics
- **RFC 8785 Library**: `github.com/gibson042/canonicaljson-go`
- **Alert Schema**: `orchestrator_action: "global_lockdown"`, includes `drift_ms` for timestamp alerts

**Security Addendum: Grant Linkage & Topology Validation** (2025-11-27):
- **ClaimID Field**: Added to ArtefactHeader (with `omitempty`), included in hash computation
- **System Artefacts**: Link to triggering claim if exists, empty for global events (ProducedByRole="orchestrator")
- **Root Artefacts**: Must have ParentHashes=[] AND ClaimID="" (ProducedByRole="user"/"cli")
- **Versioning**: New claim per version (V2 references new ClaimID, not terminated V1 claim)
- **Parent Linkage**: At least one parent must match claim's ArtefactID (allows merge operations)
- **Topology Violations**: New alert type `unauthorized_topology` with 4 subtypes, triggers global lockdown
- **Environment Contract**: Orchestrator sets `HOLT_CLAIM_ID` env var, Pup injects into header

***Template purpose:*** This document specifies the migration from UUID-based artefacts to a content-addressable Merkle DAG architecture, enabling cryptographic verification of the entire audit trail. This is not an incremental enhancement—it is a fundamental re-architecture of the blackboard's data integrity model to satisfy banking, defence, and healthcare regulatory requirements.

---

## **1. The 'why': goal and success criteria**

### **1.1. Goal statement**

Transform Holt's artefact ledger from an append-only event log into a cryptographically verifiable Merkle DAG where every artefact's identity is its SHA-256 content hash, enabling independent verification that no artefact has been tampered with after creation.

### **1.2. User story**

As a compliance officer conducting a regulatory audit of an AI-assisted workflow, I need absolute cryptographic proof that the decision chain has not been altered retroactively. When I inspect an artefact with ID `a3f2b9...`, I must be able to independently recompute that hash from the artefact's content and parent references, knowing with mathematical certainty that if the hash matches, the content is authentic. If an agent (or a compromised Pup) attempts to submit falsified data, the Orchestrator must detect the hash mismatch immediately, reject the submission, and emit a high-priority security alert that halts the workflow for forensic investigation.

### **1.3. Success criteria**

1. **Prover/Verifier contract enforced**: An agent Pup computes the SHA-256 hash of a canonicalised artefact structure and submits it as the artefact ID. The Orchestrator independently re-canonicalises the received data, recomputes the hash, and rejects the submission if hashes do not match bit-for-bit.

2. **DAG integrity with orphan block prevention**: Before accepting any artefact, the Orchestrator verifies that all parent hashes in `ParentHashes[]` exist in Redis. Exception: Root artefacts with `ParentHashes: []` are valid. If any referenced parent is missing, the artefact is rejected, an `orphan_block` security alert is published, and **global lockdown** is triggered. This ensures the Merkle DAG remains a connected graph with complete lineage.

3. **Tamper detection with global lockdown**: When a hash mismatch occurs, the Orchestrator triggers a **three-step alert mechanism**: (1) LPUSH to permanent audit log `holt:{instance}:security:alerts:log`, (2) SET lockdown key `holt:{instance}:security:lockdown` with alert payload, (3) PUBLISH to Pub/Sub channel `holt:{instance}:security:alerts`. The orchestrator halts ALL processing and enters global lockdown. Agent containers remain running for forensic investigation. Recovery requires audited manual unlock via `holt security --unlock --reason "<text>"`.

4. **End-to-end test case**: A deliberately malicious Pup submits an artefact with hash ID `abc123...` but payload content that hashes to `def456...`. The Orchestrator detects the mismatch within 100ms, rejects the write to Redis, publishes a tamper alert, and enters a paused state. The test verifies that the malicious artefact never appears in `holt hoard` output.

### **1.4. Non-goals**

- **Digital signatures**: This milestone implements content addressing (hash-based integrity). Cryptographic signatures for non-repudiation (proving *who* created an artefact) are explicitly out of scope for v2.
- **Blockchain integration**: The Merkle DAG is stored in Redis and Git, not a distributed ledger.
- **Migration tooling**: V2 instances do not read or migrate v1 UUID-based artefacts. This is a clean break.
- **Hash algorithm flexibility**: SHA-256 is hardcoded. Support for BLAKE3 or other algorithms is deferred to future work.

**Core type replacement** (`pkg/blackboard/types.go`):

```go
// VerifiableArtefact replaces the v1 Artefact struct.
// The ID field is the SHA-256 hash of Canonical(Header || Payload).
type VerifiableArtefact struct {
	// ID is the SHA-256 hash (hex-encoded, 64 characters)
	// This is the artefact's immutable, content-derived address.
	ID string `json:"id"`

	Header  ArtefactHeader  `json:"header"`
	Payload ArtefactPayload `json:"payload"`
}

// ArtefactHeader contains metadata and provenance links.
// All fields in this struct are included in the hash computation.
type ArtefactHeader struct {
	// ParentHashes replaces SourceArtefacts - now stores SHA-256 hashes, not UUIDs
	ParentHashes []string `json:"parent_hashes"` // Array of parent artefact SHA-256 hashes

	// LogicalThreadID groups versions of the same conceptual artefact
	// Retained for O(1) "latest version" lookups via Redis ZSET
	LogicalThreadID string `json:"logical_thread_id"` // UUID format

	// Version counter within the logical thread (starts at 1)
	Version int `json:"version"`

	// Timestamp of creation (part of hashed content - CRITICAL)
	CreatedAtMs int64 `json:"created_at_ms"` // Unix milliseconds

	// Agent that produced this artefact
	ProducedByRole string `json:"produced_by_role"`

	// Orchestration role
	StructuralType StructuralType `json:"structural_type"`

	// User-defined domain type
	Type string `json:"type"`

	// M4.3: Context caching - INCLUDED in hash for security/visibility scope
	// Uses omitempty: empty/nil slice excluded from canonical JSON
	ContextForRoles []string `json:"context_for_roles,omitempty"`

	// M4.6 Security Addendum: Grant Linkage & Topology Validation
	// ClaimID cryptographically binds this artefact to the authorization that permitted its creation
	// MUST be present for agent-produced artefacts (unless root artefact with ParentHashes=[])
	// Empty for root artefacts (CLI-generated) and some orchestrator-generated artefacts
	ClaimID string `json:"claim_id,omitempty"`
}

// ArtefactPayload is the actual content.
// HARD LIMIT: 1MB. Larger content must be written to disk/git.
type ArtefactPayload struct {
	Content string `json:"content"` // Max 1MB (1,048,576 bytes)
}
```

**Canonicalisation function** (`pkg/blackboard/canonical.go`):

```go
import (
	"crypto/sha256"
	"encoding/hex"
	"fmt"

	canonicaljson "github.com/gibson042/canonicaljson-go"
)

// ComputeArtefactHash computes the SHA-256 hash of a VerifiableArtefact.
// Uses RFC 8785 JSON Canonicalization Scheme to ensure deterministic serialization.
func ComputeArtefactHash(a *VerifiableArtefact) (string, error) {
	// Step 1: Create a canonical representation (Header + Payload)
	canonical := struct {
		Header  ArtefactHeader  `json:"header"`
		Payload ArtefactPayload `json:"payload"`
	}{
		Header:  a.Header,
		Payload: a.Payload,
	}

	// Step 2: Canonicalise using RFC 8785
	canonicalBytes, err := canonicaljson.Marshal(canonical)
	if err != nil {
		return "", fmt.Errorf("canonicalization failed: %w", err)
	}

	// Step 3: Hash with SHA-256
	hash := sha256.Sum256(canonicalBytes)

	// Step 4: Return hex-encoded hash
	return hex.EncodeToString(hash[:]), nil
}

// ValidateArtefactHash verifies that the provided ID matches the computed hash.
// This is the Orchestrator's verification step.
func ValidateArtefactHash(a *VerifiableArtefact) error {
	computed, err := ComputeArtefactHash(a)
	if err != nil {
		return fmt.Errorf("hash computation failed: %w", err)
	}

	if computed != a.ID {
		return &HashMismatchError{
			Expected: computed,
			Actual:   a.ID,
		}
	}

	return nil
}

// HashMismatchError is returned when verification fails.
type HashMismatchError struct {
	Expected string // Hash computed by verifier
	Actual   string // Hash claimed in artefact ID
}

func (e *HashMismatchError) Error() string {
	return fmt.Sprintf("hash mismatch: expected %s, got %s", e.Expected, e.Actual)
}

// ArtefactExists checks if an artefact with the given hash ID exists in Redis.
// Used by the Orchestrator to prevent orphan blocks in the DAG.
func (c *Client) ArtefactExists(hashID string) (bool, error) {
	key := fmt.Sprintf("holt:%s:artefact:%s", c.instanceName, hashID)
	exists, err := c.redis.Exists(c.ctx, key).Result()
	if err != nil {
		return false, fmt.Errorf("redis EXISTS check failed: %w", err)
	}
	return exists == 1, nil
}
```

### **2.2. Orchestrator changes**

#### **New logic: Comprehensive artefact verification on submission**

When the Orchestrator receives a new artefact notification via `artefact_events`, it performs a **four-stage validation** before accepting the artefact into the DAG:

1. **Fetch artefact** from Redis at `holt:{instance}:artefact:{hash_id}`

2. **Parent existence check** (prevents orphan blocks):
   ```go
   // Exception: Root artefacts with empty ParentHashes are valid (e.g., GoalDefined)
   if len(a.Header.ParentHashes) > 0 {
       // CRITICAL: Verify all parent hashes exist in the blackboard
       for _, parentHash := range a.Header.ParentHashes {
           exists, err := o.blackboard.ArtefactExists(parentHash)
           if err != nil {
               return fmt.Errorf("failed to check parent existence: %w", err)
           }
           if !exists {
               // SECURITY EVENT: Orphan block detected - trigger global lockdown
               alert := createOrphanBlockAlert(a.ID, parentHash, a.Header.ProducedByRole)
               triggerGlobalLockdown(alert) // Three-step: LPUSH log + SET lockdown + PUBLISH
               return fmt.Errorf("parent hash does not exist: %s (orphan block)", parentHash)
           }
       }
   }
   ```

3. **Timestamp validation** (prevents time-based attacks):
   ```go
   now := time.Now().UnixMilli()
   drift := 5 * 60 * 1000 // 5 minutes in milliseconds

   if a.Header.CreatedAtMs > now+drift {
       return fmt.Errorf("timestamp too far in future: %d ms drift", a.Header.CreatedAtMs-now)
   }
   if a.Header.CreatedAtMs < now-drift {
       return fmt.Errorf("timestamp too far in past: %d ms drift", now-a.Header.CreatedAtMs)
   }
   ```

4. **Topology validation** (M4.6 Security Addendum - prevents unauthorized work):
   ```go
   // Validation rules based on artefact origin:

   // Rule 1: Root artefacts (CLI/user-generated) validation
   if a.Header.ProducedByRole == "user" || a.Header.ProducedByRole == "cli" {
       // Root artefacts must have empty parents AND empty claim
       if len(a.Header.ParentHashes) == 0 && a.Header.ClaimID == "" {
           // Valid root artefact - skip further topology checks
           goto hashVerification
       }
       // SECURITY EVENT: Root artefact has invalid topology
       alert := createTopologyAlert(a, "root_artefact_with_claim")
       triggerGlobalLockdown(alert)
       return fmt.Errorf("root artefact must have empty ParentHashes and ClaimID")
   }

   // Rule 2: Orchestrator artefacts (Failure/Terminal) validation
   if a.Header.ProducedByRole == "orchestrator" {
       // Orchestrator artefacts MAY have ClaimID (if triggered by specific claim)
       // or empty ClaimID (if global event). No parent linkage enforcement.
       goto hashVerification
   }

   // Rule 3: Agent-produced artefacts MUST have topology binding
   if a.Header.ClaimID == "" {
       // SECURITY EVENT: Missing required ClaimID
       alert := createTopologyAlert(a, "missing_claim_id")
       triggerGlobalLockdown(alert)
       return fmt.Errorf("agent artefact missing required ClaimID")
   }

   // Rule 4: Validate ClaimID references an active/granted claim for this agent
   claim, err := o.blackboard.FetchClaim(a.Header.ClaimID)
   if err != nil || claim == nil {
       alert := createTopologyAlert(a, "invalid_claim_reference")
       triggerGlobalLockdown(alert)
       return fmt.Errorf("ClaimID does not exist: %s", a.Header.ClaimID)
   }

   // Verify claim status (must be Active or Granted, not Terminated/Complete)
   if claim.Status != "Active" && claim.Status != "Granted" {
       alert := createTopologyAlertWithClaim(a, claim, "invalid_claim_reference")
       triggerGlobalLockdown(alert)
       return fmt.Errorf("ClaimID references non-active claim: %s (status: %s)",
           a.Header.ClaimID, claim.Status)
   }

   // Verify claim belongs to this agent
   if claim.AgentRole != a.Header.ProducedByRole {
       alert := createTopologyAlertWithClaim(a, claim, "invalid_claim_reference")
       triggerGlobalLockdown(alert)
       return fmt.Errorf("ClaimID belongs to different agent: expected %s, got %s",
           claim.AgentRole, a.Header.ProducedByRole)
   }

   // Rule 5: Parent linkage check - at least one parent must match claim's target
   // This ensures the agent is building upon the work it was assigned
   if len(a.Header.ParentHashes) > 0 {
       hasValidParent := false
       for _, parentHash := range a.Header.ParentHashes {
           if parentHash == claim.ArtefactID {
               hasValidParent = true
               break
           }
       }

       if !hasValidParent {
           // SECURITY EVENT: Parent linkage violation
           alert := createParentLinkageAlert(a, claim)
           triggerGlobalLockdown(alert)
           return fmt.Errorf("parent linkage violation: no parent matches claim target %s",
               claim.ArtefactID)
       }
   }

**Global Lockdown Implementation (`triggerGlobalLockdown`)**:
```go
func (o *Orchestrator) triggerGlobalLockdown(alert SecurityAlert) error {
    alertJSON, _ := json.Marshal(alert)

    // Step 1: LPUSH to permanent audit log (newest first)
    if err := o.redis.LPush(ctx, "holt:"+o.instance+":security:alerts:log", alertJSON).Err(); err != nil {
        return fmt.Errorf("failed to persist alert to log: %w", err)
    }

    // Step 2: SET lockdown key (circuit breaker state)
    if err := o.redis.Set(ctx, "holt:"+o.instance+":security:lockdown", alertJSON, 0).Err(); err != nil {
        return fmt.Errorf("failed to set lockdown key: %w", err)
    }

    // Step 3: PUBLISH to real-time notification channel
    if err := o.redis.Publish(ctx, "holt:"+o.instance+":security:alerts", alertJSON).Err(); err != nil {
        return fmt.Errorf("failed to publish alert: %w", err)
    }

    log.Error("SECURITY LOCKDOWN TRIGGERED", "type", alert.Type, "instance", o.instance)
    return nil
}
```

6. **If all validations pass**: Proceed with normal claim creation

**Validation order rationale**:
- **Parent check first**: Fast Redis existence check prevents orphans early
- **Timestamp second**: Lightweight arithmetic check before Redis fetches
- **Topology third**: Redis claim fetch + status validation (O(1) lookups, moderate cost)
- **Hash last**: Most CPU-intensive operation, only performed if structure and authorization are valid

func (o *Orchestrator) createFailureArtefact(reason string, parentHashes []string) error {
	artefact := &blackboard.VerifiableArtefact{
		Header: blackboard.ArtefactHeader{
			ParentHashes:    parentHashes,
			LogicalThreadID: uuid.New().String(),
			Version:         1,
			CreatedAtMs:     time.Now().UnixMilli(),
			ProducedByRole:  "orchestrator", // Trusted authority
			StructuralType:  blackboard.StructuralTypeFailure,
			Type:            "SystemFailure",
		},
		Payload: blackboard.ArtefactPayload{
			Content: reason,
		},
	}

	// Orchestrator computes hash for its own artefacts
	hash, err := blackboard.ComputeArtefactHash(artefact)
	if err != nil {
		return err

	artefact.ID = hash

	// No verification needed - we trust ourselves
	return o.blackboard.WriteArtefact(artefact)
}

### **2.3. Agent pup changes**

#### **New logic: Hash computation before submission**

When the Pup's Work Executor completes a tool execution:

1. **Receive tool output** (JSON from stdout)
2. **Assemble Header**:
   ```go
   header := blackboard.ArtefactHeader{
       ParentHashes:    extractParentHashes(targetArtefact),
       LogicalThreadID: targetArtefact.Header.LogicalThreadID, // Inherit or new
       Version:         targetArtefact.Header.Version + 1,
       CreatedAtMs:     time.Now().UnixMilli(),
       ProducedByRole:  os.Getenv("HOLT_AGENT_ROLE"),
       StructuralType:  toolOutput.StructuralType,

   payload := blackboard.ArtefactPayload{

## **3. The 'how': implementation & testing plan**

### **3.1. Key design decisions & risks**

#### **Decision 1: RFC 8785 for canonicalisation**

**Rationale**: RFC 8785 (JSON Canonicalization Scheme) is an IETF standard specifically designed for cryptographic applications. It guarantees deterministic JSON serialization by:
- Sorting object keys lexicographically
- Removing insignificant whitespace
- Normalizing number representation
- Escaping Unicode consistently

**Library choice**: `github.com/gibson042/canonicaljson-go` is a stable, well-tested implementation used in production systems.

**Risk mitigation**: Extensive unit tests comparing Pup-computed hashes with Orchestrator-computed hashes on identical data structures.

#### **Decision 2: Halt workflow on hash mismatch**

**Rationale**: A hash mismatch is a **critical security event**, not a routine error. Potential causes:
- Malicious Pup attempting to forge artefacts
- Memory corruption in Pup container
- Network MITM attack (unlikely with Redis over localhost, but possible)
- Software bug in canonicalisation logic

**Trade-off**: Halting the workflow is disruptive, but necessary. In banking/defence contexts, false audit trails are catastrophic. Pausing preserves evidence for investigation.

**Alternative considered and rejected**: Terminating just the failing claim would allow a compromised agent to continue operating.

#### **Decision 3: 1MB payload hard limit**

**Rationale**: Redis is designed for sub-megabyte values. Large payloads degrade performance and increase memory pressure.

**Enforcement**: The blackboard's `Validate()` method rejects oversized payloads *before* hash computation, providing immediate feedback to agent developers.

**Agent pattern**: For large files (e.g., generated PDFs, datasets), agents must:
1. Write content to the workspace filesystem
2. Compute file hash (SHA-256)
3. Store hash in payload: `{"file": "report.pdf", "hash": "e3b0c442..."}`
4. Commit to Git if persistence is required

#### **Risk: Clock skew in distributed deployments**

**Scenario**: If the Orchestrator and Pup containers have system clocks >5 minutes apart, all artefacts will be rejected.

**Mitigation**:
- **Documentation**: Require NTP synchronization in production environments
- **Error message**: Clearly state "timestamp drift detected" with actual drift value
- **Future enhancement**: Configurable drift tolerance per environment (tighter for local dev, looser for multi-region)

### **3.3. Performance & resource considerations**

#### **Resource usage**

- **CPU**: SHA-256 hashing is fast (~500 MB/s on modern CPUs). For a 1MB payload, hashing takes ~2ms.
- **Memory**: RFC 8785 canonicalisation allocates temporary buffers. Estimate 2x payload size during canonicalisation.
- **Redis**: Hash IDs are 64 bytes (hex-encoded SHA-256) vs. 36 bytes (UUID). Marginal increase in key size.

#### **Scalability limits**

- **Throughput**: Hash computation is CPU-bound. On an 8-core machine, expect >1000 artefacts/second.
- **DAG depth**: No practical limit. Parent hash lookups are O(1) Redis operations.
- **Concurrent verification**: Orchestrator can verify artefacts in parallel (goroutine per verification).

#### **Performance requirements**

- **Hash computation (Pup)**: <5ms for 1MB payload
- **Hash verification (Orchestrator)**: <10ms including Redis fetch
- **Security alert latency**: Alert published within 100ms of mismatch detection

## **4. Principle compliance check**

### **4.1. YAGNI (You ain't gonna need it)**

**New dependencies**:
1. **`github.com/gibson042/canonicaljson-go`** (or equivalent RFC 8785 library)
   - **Justification**: Deterministic JSON serialization is impossible with stdlib `encoding/json` due to map key ordering. This is a fundamental requirement, not a nice-to-have.

**No other dependencies required**. SHA-256 is in Go's stdlib (`crypto/sha256`).

### **4.2. Auditability**

**New artefacts created**:
- **Security alerts**: Published to `holt:{instance}:security:alerts` when hash mismatch detected. These are **not stored as artefacts** to avoid recursive verification issues. Instead, they are ephemeral Pub/Sub events.

**Enhanced auditability**:
- Every artefact's ID is now **mathematically bound** to its content and lineage
- Retroactive tampering is cryptographically detectable
- Third-party auditors can independently verify the entire chain using the `holt verify` CLI

This is auditability as a **first-class security property**, not just a logging feature.

### **4.3. Small, single-purpose components**

**Component separation maintained**:
- **Blackboard**: Owns hash computation and validation logic (`pkg/blackboard/canonical.go`)
- **Orchestrator**: Enforces verification policy and publishes alerts (`internal/orchestrator/verifier.go`)
- **Pup**: Generates hashes before submission (`internal/pup/hash.go`)
- **CLI**: Provides human verification tools (`cmd/holt/commands/verify.go`)

No component crosses boundaries. The `blackboard` package remains the single source of truth for cryptographic operations.

### **4.4. Security considerations**

#### **Attack surface analysis**

**Threat 1: Malicious Pup forges artefact**
- **Mitigation**: Hash mismatch detected by Orchestrator, workflow halted, alert published
- **Residual risk**: None if Orchestrator verification is implemented correctly

**Threat 2: Compromised Orchestrator accepts invalid hash**
- **Mitigation**: External auditors can re-verify using `holt verify` CLI, which uses the **same blackboard library** as Orchestrator
- **Residual risk**: If the blackboard library itself is compromised, the entire system is compromised (standard software supply chain risk)

**Threat 3: MITM attack on Redis connection**
- **Mitigation**: Out of scope for this milestone. Future work: TLS for Redis connections
- **Residual risk**: High in untrusted network environments

**Threat 4: Timestamp manipulation to evade drift detection**
- **Mitigation**: Timestamps are **included in the hash**. Modifying timestamp invalidates the hash.
- **Residual risk**: None

**Threat 5: Unauthorized work - valid hash, wrong task (M4.6 Security Addendum)**
- **Scenario**: Malicious agent produces correctly-hashed artefact but references unrelated parents or ignores assigned work
- **Mitigation**: Topology validation enforces Grant Linkage. ClaimID is cryptographically bound in hash, parent linkage checked.
- **Residual risk**: None if orchestrator correctly validates claim status and parent intersection

**Threat 6: Claim ID reuse attack (M4.6 Security Addendum)**
- **Scenario**: Agent attempts to reuse a terminated/completed claim ID from a previous task
- **Mitigation**: Orchestrator validates claim status must be Active/Granted, rejects Terminated/Complete claims
- **Residual risk**: None with proper claim lifecycle management

**Threat 7: Cross-agent claim injection (M4.6 Security Addendum)**
- **Scenario**: Agent A attempts to submit work using Agent B's claim ID
- **Mitigation**: Orchestrator validates claim.AgentRole matches artefact.ProducedByRole
- **Residual risk**: None if agent roles are properly isolated

#### **Credential and secret handling**

**No secrets introduced**. SHA-256 is a public hash function. No signing keys required.

#### **Container isolation**

**No changes to container isolation**. Pups remain unprivileged, non-root containers.

#### **Network communications**

**No new network protocols**. All communication remains via Redis Pub/Sub.

### **4.5. Backward compatibility**

**Breaking changes**:
1. **Artefact schema**: `VerifiableArtefact` replaces `Artefact`
2. **ID format**: SHA-256 hashes (64 hex chars) replace UUIDs (36 chars with dashes)
3. **Redis keys**: `artefact:{hash}` replaces `artefact:{uuid}`

**Migration path**: None. This is a **clean break**. V2 instances do not read V1 data.

**Justification**: The security model is fundamentally incompatible with UUID-based addressing. Attempting hybrid support would compromise cryptographic integrity.

**Deprecation**: V1 is immediately deprecated upon V2 release. Production users must:
1. Complete in-flight workflows on V1
2. Archive V1 audit trails (export to JSON)
3. Upgrade to V2
4. Start new workflows

### **4.6. Dependency impact**

#### **Go version**

**No change**. SHA-256 and JSON are in stdlib since Go 1.0.

#### **Redis version**

**No change**. All Redis operations (HSET, ZSET, Pub/Sub) are in Redis 2.0+.

#### **Docker version**

**No change**. No new Docker API features required.

#### **Build dependencies**

**One new dependency**: RFC 8785 canonicalisation library.

**CI/CD impact**: `go mod download` will fetch the new library. No changes to Dockerfile or Makefile targets.

## **6. Error scenarios & edge cases**

### **6.1. Failure modes**

#### **Scenario 1: Redis unavailable during hash verification**

**Trigger**: Orchestrator receives artefact event, but Redis connection fails during fetch

**Handling**:
```go
artefact, err := o.blackboard.FetchArtefact(hash)
if err != nil {
    // Retry with exponential backoff (existing resilience)
    return fmt.Errorf("failed to fetch artefact for verification: %w", err)
}
```

**Outcome**: Orchestrator retries per existing policy. If Redis is down >1 minute, orchestrator exits (existing health check behavior).

#### **Scenario 2: Canonicalisation library panics**

**Trigger**: Malformed JSON in artefact structure causes RFC 8785 library to panic

**Handling**:
```go
func ComputeArtefactHash(a *VerifiableArtefact) (hash string, err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("canonicalization panic: %v", r)
        }
    }()

    canonical, err := canonicaljson.Marshal(...)
    // ...
}
```

**Outcome**: Panic converted to error, Orchestrator rejects artefact, publishes security alert.

#### **Scenario 3: Pup clock skew >5 minutes**

**Trigger**: Pup container has incorrect system time (e.g., NTP failure)

**Handling**:
```go
// Orchestrator timestamp validation
if a.Header.CreatedAtMs > now+drift {
    return &TimestampDriftError{
        ArtefactTime: a.Header.CreatedAtMs,
        OrchestratorTime: now,
        Drift: a.Header.CreatedAtMs - now,
    }
}
```

**Outcome**: Artefact rejected with clear error message. **Recommendation**: Log drift value to help diagnose clock sync issues.

#### **Scenario 4: Payload exactly 1MB (boundary condition)**

**Trigger**: Agent generates payload with 1,048,576 bytes

**Handling**:
```go
const MaxPayloadSize = 1 * 1024 * 1024

if len(p.Content) > MaxPayloadSize {
    return fmt.Errorf("payload exceeds 1MB limit: %d bytes", len(p.Content))
}
```

**Outcome**: 1MB exactly is **accepted**. 1MB + 1 byte is rejected. Clear boundary.

#### **Scenario 5: Orphan block (missing parent hash)**

**Trigger**: Agent submits artefact referencing a parent hash that doesn't exist in Redis

**Root causes**:
- **Bug in agent logic**: Agent fabricates a parent hash instead of reading from the blackboard
- **Race condition**: Parent artefact was deleted/expired before child submission (unlikely with Redis persistence)
- **Malicious agent**: Deliberately attempting to bypass DAG integrity

**Handling**:
```go
// Orchestrator parent existence check
for _, parentHash := range a.Header.ParentHashes {
    exists, err := o.blackboard.ArtefactExists(parentHash)
    if err != nil {
        return fmt.Errorf("failed to check parent existence: %w", err)
    }
    if !exists {
        // SECURITY EVENT
        publishSecurityAlert(SecurityAlert{
            Type:          "orphan_block",
            ArtefactID:    a.ID,
            MissingParent: parentHash,
            AgentRole:     a.Header.ProducedByRole,
        })
        pauseWorkflow()
        return fmt.Errorf("parent hash does not exist: %s (orphan block)", parentHash)
    }
}
```

**Outcome**: Artefact rejected, security alert published, workflow paused. The orphan artefact is **not written to Redis**, preventing DAG corruption.

**Forensic investigation**:
1. Check agent logs for parent hash generation logic
2. Verify agent's context assembly is correctly fetching from blackboard
3. Inspect `ParentHashes` field in rejected artefact (logged in alert)

### **6.2. Concurrency considerations**

#### **Race condition: Multiple Pups submit identical artefact**

**Scenario**: Two Pups independently produce the exact same work (rare but possible).

**Analysis**: They will compute **identical hashes** (deterministic). First write to `artefact:{hash}` wins. Second write is idempotent (same data).

**Outcome**: Safe. Redis HSET is idempotent. Both Pups can reference the same hash ID.

#### **Race condition: Orchestrator verifying while Pup updating**

**Scenario**: Impossible. Artefacts are **immutable**. Once written to `artefact:{hash}`, they never change.

#### **Concurrency: Parallel hash verification**

**Implementation**:
```go
func (o *Orchestrator) verifyArtefacts(artefacts []*VerifiableArtefact) error {
    errChan := make(chan error, len(artefacts))

    for _, a := range artefacts {
        go func(artefact *VerifiableArtefact) {
            errChan <- blackboard.ValidateArtefactHash(artefact)
        }(a)
    }

    // Collect results
    for range artefacts {
        if err := <-errChan; err != nil {
            return err // Fail fast on first mismatch
        }
    }
    return nil
}
```

**Safety**: Hash computation is pure (no shared state). Goroutines are safe.

### **6.3. Edge case handling**

#### **Edge case 1: Empty payload**

**Scenario**: Agent produces artefact with zero-length payload

**Handling**: Valid. Empty string hashes to a deterministic value.

```go
payload := ArtefactPayload{Content: ""}
err := payload.Validate() // No error (size = 0 < 1MB)
```

#### **Edge case 2: Payload with null bytes**

**Scenario**: Binary data in JSON-encoded payload

**Handling**: JSON marshalling escapes null bytes (`\u0000`). Canonicalisation preserves escaping.

**Test case**:
```go
func TestHashWithNullBytes(t *testing.T) {
    a := createArtefact(payload: "hello\x00world")
    hash, _ := ComputeArtefactHash(a)
    assert.NotEmpty(t, hash) // Valid hash produced
}
```

#### **Edge case 3: Maximum DAG depth**

**Scenario**: Artefact with 10,000 ancestors in chain

**Handling**: ParentHashes field only stores **direct parents** (typically 1-2). DAG depth does not affect hash computation time (O(1) operation).

**Limit**: No architectural limit. Redis can store arbitrarily long hash strings.

#### **Edge case 4: Hash collision (theoretical)**

**Scenario**: Two different artefacts produce the same SHA-256 hash

**Probability**: 2^-256 (effectively impossible with current computing power)

**Handling**: Not handled. SHA-256 collision resistance is a foundational assumption. If broken, the entire security model fails (same as Bitcoin, TLS, etc.).

## **6.4. M4.6 Security Addendum: Grant Linkage & Topology Validation**

### **The Problem: Content Integrity vs. Contextual Authorization**

The base M4.6 implementation provides **cryptographic content integrity** via Merkle DAG:
- ✓ Hash-based artefact IDs prevent tampering
- ✓ Parent hash verification prevents orphan blocks
- ✓ Timestamp validation prevents time-based attacks

However, it does not enforce **contextual authorization**. A malicious agent could:
1. Produce a validly-hashed artefact (passes content verification)
2. Reference unrelated parent artefacts (not the work it was assigned)
3. Submit work while ignoring its granted claim
4. "Phone in" semantically invalid work that appears structurally correct

**Example Attack Scenario**:
```
1. Orchestrator grants Claim-123 to Agent A: "Review code in file X"
2. Malicious Agent A ignores the claim
3. Agent A produces valid artefact with hash abc123... for unrelated file Y
4. Artefact passes hash verification (content is valid)
5. BUT: Agent A never reviewed file X (authorization violated)
```

### **The Solution: Cryptographic Grant Binding**

**Core Principle**: Every agent-produced artefact must be cryptographically bound to the authorization (Claim) that permitted its creation.

**Implementation**: Add `ClaimID` field to `ArtefactHeader`:
- Field is included in hash computation (immutable binding)
- Orchestrator validates ClaimID references an active/granted claim
- Orchestrator validates parent linkage (at least one parent matches claim's target artefact)

### **Topology Validation Rules**

#### **Rule 1: Root Artefacts (CLI/User-Generated)**
```go
if ProducedByRole == "user" || ProducedByRole == "cli" {
    REQUIRE: ParentHashes == [] AND ClaimID == ""
    REASON: Root artefacts (GoalDefined) have no parents and no authorization
}
```

#### **Rule 2: Orchestrator Artefacts (Failure/Terminal)**
```go
if ProducedByRole == "orchestrator" {
    ALLOW: ClaimID may be set (if triggered by specific claim) OR empty (if global event)
    SKIP: Parent linkage enforcement (orchestrator is trusted authority)
}
```

#### **Rule 3: Agent Artefacts (Primary Security Check)**
```go
if ProducedByRole is an agent role {
    REQUIRE: ClaimID != ""
    REQUIRE: Claim exists in Redis
    REQUIRE: Claim.Status == "Active" OR "Granted" (NOT "Terminated" or "Complete")
    REQUIRE: Claim.AgentRole == ProducedByRole (prevent cross-agent injection)
    REQUIRE: Intersection(ParentHashes, Claim.ArtefactID) > 0 (parent linkage)
}
```

### **Security Properties Achieved**

1. **Authorization Binding**: Every artefact is cryptographically bound to a specific grant
2. **Work Provenance**: Agents cannot submit work unrelated to their assigned task
3. **Claim Lifecycle Enforcement**: Terminated claims cannot be reused
4. **Agent Isolation**: Agent A cannot use Agent B's claims
5. **Merge Operation Support**: Agents can reference multiple parents (e.g., code + review context) as long as one parent is the assigned work

### **Attack Prevention Matrix**

| Attack Type | Mitigation | Detection Point |
|-------------|-----------|----------------|
| **Unbound Artefact** | ClaimID required for agent artefacts | Topology validation (Rule 3) |
| **Claim Reuse** | Claim status must be Active/Granted | Claim status check |
| **Cross-Agent Injection** | Claim.AgentRole must match producer | Agent role validation |
| **Parent Spoofing** | At least one parent must match claim target | Parent linkage check |
| **Root Escalation** | Root artefacts must have empty claim+parents | Root validation (Rule 1) |

### **Environment Variable Contract**

**Orchestrator Responsibility**: When granting work via `docker run`, set:
```bash
HOLT_CLAIM_ID=<claim-uuid>
```

**Pup Responsibility**: Inject env var into artefact header:
```go
header.ClaimID = os.Getenv("HOLT_CLAIM_ID")
```

**Failure Mode**: If `HOLT_CLAIM_ID` is not set, Pup creates artefact with empty ClaimID → Orchestrator rejects as `missing_claim_id` topology violation → Global lockdown.

### **Versioning and Rework Cycles**

**Question**: When an agent creates V2 of an artefact (rework), which ClaimID should it use?

**Answer**: **New claim per version**. The orchestrator's rework cycle:
1. Terminates old claim (status = "Terminated")
2. Creates new `pending_assignment` claim for V2
3. Grants new claim to agent
4. Agent's V2 artefact must reference the **new** ClaimID

**Validation**: Orchestrator rejects artefacts referencing terminated claims, ensuring V2 cannot use V1's authorization.

### **Edge Case: Multi-Parent Artefacts**

**Scenario**: Agent is granted work on artefact A, but also needs context from artefact B (e.g., code review requires both implementation and requirements).

**Solution**: Parent linkage requires **at least one** parent to match the claim's target:
```go
ParentHashes = ["hash-of-A", "hash-of-B"]
Claim.ArtefactID = "hash-of-A"
// Validation: "hash-of-A" ∈ ParentHashes → PASS
```

This allows agents to assemble rich context while ensuring the assigned work is included.

### **Forensic Value**

When topology violation occurs:
1. Alert includes `claim_id_provided`, `claim_status`, `expected_parent_artefact`, `actual_parent_hashes`
2. Investigator can reconstruct: "Agent was granted work on X, but submitted work for Y"
3. Timeline analysis: Compare claim grant timestamp vs. artefact creation timestamp
4. Agent logs: Search for ClaimID reference to identify where agent obtained the ID

## **7. Open questions & decisions**

### **Question 1: Forensic preservation duration**

**Status**: Resolved in design

When a hash mismatch occurs, the workflow pauses and containers remain running. **How long should they remain running before automatic cleanup?**

**Decision**: Manual intervention required. Operator must inspect logs, extract container state, and explicitly run `holt down` when ready. No automatic timeout.

### **Question 2: Orchestrator-generated artefact audit trail**

**Status**: Open

When the Orchestrator creates Failure/Terminal artefacts, it acts as a Trusted Authority. Should these artefacts include a special marker (e.g., `trusted_origin: true`) to distinguish them from agent-generated artefacts in audit reports?

**Recommendation**: Yes. Add optional field:
```go
type ArtefactHeader struct {
    // ...
    TrustedOrigin bool `json:"trusted_origin,omitempty"` // True if created by Orchestrator
}
```

This enables auditors to identify which artefacts bypassed Prover/Verifier checks.

### **Question 3: CLI verification vs. Orchestrator verification**

**Status**: Resolved

Should `holt verify` use the **same code path** as Orchestrator verification?

**Decision**: Yes. Both must call `pkg/blackboard.ValidateArtefactHash()`. This ensures identical verification logic and prevents subtle bugs.

### **Question 4: Hash algorithm versioning**

**Status**: Deferred to future work

If SHA-256 is ever deprecated (quantum computing), how do we migrate?

**Future design**: Add `hash_algorithm` field to ArtefactHeader:
```go
type ArtefactHeader struct {
    HashAlgorithm string `json:"hash_algorithm"` // "sha256", "sha3-256", "blake3", etc.
    // ...
}
```

For v2, this field is hardcoded to `"sha256"`. Future versions can support multiple algorithms.

### **8.2. Common pitfalls to avoid**

#### **Pitfall 1: Forgetting to include timestamp in hash**

**Incorrect**:
```go
canonical := struct {
    Type    string
    Payload string
}{
    Type:    a.Header.Type,
    Payload: a.Payload.Content,
}
```

**Correct**:
```go
canonical := struct {
    Header  ArtefactHeader
    Payload ArtefactPayload
}{
    Header:  a.Header, // Includes CreatedAtMs
    Payload: a.Payload,
}
```

#### **Pitfall 2: Using stdlib JSON marshal**

**Incorrect**:
```go
bytes, _ := json.Marshal(canonical) // Non-deterministic!
```

**Correct**:
```go
bytes, _ := canonicaljson.Marshal(canonical) // RFC 8785
```

#### **Pitfall 3: Not validating payload size before hashing**

**Incorrect**:
```go
hash := ComputeArtefactHash(a) // Compute 10MB hash, then fail
```

**Correct**:
```go
if err := a.Payload.Validate(); err != nil {
    return err // Fail fast before expensive hash
}
hash := ComputeArtefactHash(a)
```

#### **Pitfall 4: Terminating containers on tamper alert**

**Incorrect**:
```go
if hashMismatch {
    docker.Kill(container) // Destroys evidence!
}
```

**Correct**:
```go
if hashMismatch {
    publishSecurityAlert(...)
    pauseWorkflow() // Leave containers running
}
```

### **9.2. Rollback and disaster recovery**

#### **Rollback procedure**

**Scenario**: Critical bug discovered in hash verification logic after v2 deployment

**Steps**:
1. **Stop all v2 instances**: `holt down` on all hosts
2. **Archive v2 Redis database**: `redis-cli SAVE` → backup file
3. **Redeploy v1 binaries**: `git checkout v1.x && make build`
4. **Restore v1 Redis database** from pre-upgrade backup
5. **Restart v1 instances**: `holt up`

**Data loss**: All v2 workflows are lost (in-progress only; v2 cannot resume on v1).

#### **Disaster recovery: Redis data corruption**

**Scenario**: Redis database file corrupted, all artefacts lost

**Recovery**:
1. **Restore from Git**: All CodeCommit artefacts are backed by Git. Workflow can be reconstructed from `git log`.
2. **Restore from Redis backup**: Daily Redis backups via RDB snapshots (standard practice).
3. **Partial recovery**: If only some artefacts are corrupted, use `holt verify` to identify valid artefacts, discard invalid ones.

**Prevention**: Enable Redis AOF (Append-Only File) persistence in production.

#### **Feature disable mechanism**

**Not applicable**. Hash verification is a foundational security property, not a toggleable feature. Disabling it would compromise the entire audit trail.

## **Appendix B: Migration from v1 to v2**

### **Breaking changes summary**

| Component | V1 Behavior | V2 Behavior | Breaking? |
|-----------|-------------|-------------|-----------|
| Artefact ID | UUID (36 chars) | SHA-256 hash (64 hex chars) | ✅ Yes |
| Redis key | `artefact:{uuid}` | `artefact:{hash}` | ✅ Yes |
| SourceArtefacts field | Array of UUIDs | Renamed to ParentHashes, array of hashes | ✅ Yes |
| Timestamp validation | None | 5-minute drift window enforced | ✅ Yes |
| Payload size | Unlimited (Redis 512MB limit) | 1MB hard limit | ✅ Yes |

### **Migration strategy**

**For development/testing environments**:
1. Stop v1 instance
2. Clear Redis database (`redis-cli FLUSHDB`)
3. Deploy v2 binaries
4. Start v2 instance

**For production environments**:
1. **Complete all v1 workflows** before upgrade
2. **Export v1 audit trail**: `holt hoard --output jsonl > v1-audit-trail.jsonl`
3. **Archive v1 Redis snapshot**: `redis-cli SAVE && cp dump.rdb v1-backup-$(date +%s).rdb`
4. **Deploy v2 to new Redis instance** (avoid data contamination)
5. **Run parallel for 1 week**: New workflows on v2, legacy queries on v1
6. **Decommission v1** after validation period

### **No automated migration**

There is **no tool to convert v1 UUIDs to v2 hashes** because:
- Hash must be computed from original content
- V1 did not store canonical representation
