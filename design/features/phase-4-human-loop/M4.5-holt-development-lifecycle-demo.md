## **1. The 'why': goal and success criteria**

### **1.1. Goal statement**

To design and implement a multi-agent workflow that demonstrates Holt's capability to manage complex, multi-stage development processes from high-level goal to tested, reviewed implementation. This workflow serves as both a demonstration of Holt's coordination capabilities and a practical tool for Holt's own development.

### **1.2. User story**

As a Holt developer, I want to run a single `holt forage --goal "Implement feature X"` command and have the system:

1. **Design Phase**: Use AI to create a high-level design, ask me clarifying questions, and produce a draft specification
2. **Human Review**: Pause for my review and approval of the design
3. **Detailed Specification**: Expand the approved draft into a complete, implementation-ready design document
4. **Final Design Review**: Get my approval on the final specification
5. **Implementation**: Write code and tests based on the approved design, with the ability to ask architectural questions
6. **Automated Quality Gates**: Run tests and security scans automatically
7. **Iterative Refinement**: If tests fail or security issues are found, automatically rework until they pass
8. **Final Human Review**: Get engineering review and approval before completion

This creates a complete audit trail showing the evolution from goal to implementation, with human oversight at critical decision points.

### **1.3. Success criteria**

**Design Workflow (Workflow 1):**
1. DesignerGeminiAgent can use M4.1 Q&A to ask clarifying questions before drafting
2. DesignSpecDraft artefact is created with high-level design proposal
3. HumanReviewerAgent prompts console for human input and creates Review artefact
4. Review rejection triggers M3.3 automated rework loop back to DesignerGeminiAgent
5. ClaudeDesignAgent expands approved draft into complete DesignSpec artefact
6. Second human review approves final spec and creates GoalDefined for implementation
7. Complete design phase audit trail visible in `holt hoard`

**Implementation Workflow (Workflow 2):**
8. CoderClaudeAgent can ask questions of ArchitectGeminiAgent using M4.1 during development
9. ArchitectGeminiAgent uses M4.3 Knowledge artefact with project documentation
10. CoderClaudeAgent produces ChangeSet artefact (JSON with commit_hash, test_summary, change_summary)
11. TestRunnerAgent and SecurityScannerAgent review in parallel (M3.2 review phase)
12. Failed automated reviews trigger M3.3 rework loop back to CoderClaudeAgent
13. M3.2 Review Gate logic prevents LeadEngineerGeminiAgent from reviewing until automated reviews pass
14. LeadEngineerGeminiAgent performs final code review with exclusive claim
15. Final approval creates Terminal artefact, completing the workflow

**E2E Testing:**
16. Complete workflow from GoalDefined to Terminal succeeds in single test
17. Negative path test: automated reviewer failure triggers rework, then succeeds
18. Context caching test: ArchitectGeminiAgent's Knowledge artefact is reused across queries
19. Human review simulation test: automated approval/rejection in test environment

**Configuration & Documentation:**
20. Example holt.yml demonstrates all 8 agents with proper configuration
21. System prompts configured for each AI agent type
22. All agents properly configured for their bidding strategies (review/claim/exclusive)

### **1.4. Non-goals**

- **Production LLM integration**: Demo uses mock agents or requires external LLM API keys (not bundled)
- **Git commit automation**: ChangeSet contains commit info but doesn't auto-push
- **Dynamic agent selection**: Fixed set of 8 agents, no runtime agent creation
- **Multi-repo support**: Single workspace, single repository
- **Continuous deployment**: No automatic deployment after approval
- **Cost optimization**: No LLM cost tracking or optimization
- **Parallel implementation paths**: Linear workflow only (no A/B design variations)

### **2.4. New agents to be implemented**

M4.5 introduces 8 new agents. Each agent is implemented as a standalone tool script that follows the standard agent contract.

#### **2.4.1. DesignerGeminiAgent**

**Purpose:** Create high-level design drafts and ask clarifying questions

**Implementation approach:** Wrapper for Gemini CLI or API

**Bidding strategy:** `exclusive` (first agent in design workflow)

**Tool script logic:**
1. Receive GoalDefined artefact in context
2. **First run**: Check if context has any Answer artefacts
   - If no Answers: Use M4.1 to create Question artefact asking for clarification
   - Output: `{"artefact_type": "Question", "artefact_payload": "...", "summary": "..."}`
3. **Subsequent runs** (after Answers received):
   - Use Gemini to generate DesignSpecDraft based on GoalDefined + all Answers
   - Output: `{"artefact_type": "DesignSpecDraft", "artefact_payload": "<markdown>", "summary": "..."}`

**System prompt (in holt.yml):**
```yaml
prompts:
  execution: |
    You are a senior product designer. Your role is to:
    1. Understand high-level goals and ask clarifying questions
    2. Create concise, high-level design proposals (1-2 pages)
    3. Focus on the "what" and "why", not the "how"

    Output format:
    - If unclear: Create a Question artefact
    - If clear: Create a DesignSpecDraft artefact with markdown content
```

#### **2.4.2. HumanReviewerAgent**

**Purpose:** Pause workflow for human review via console input

**Implementation approach:** Custom Go binary that prompts stdin/stdout

**Bidding strategy:** `review` (reviews DesignSpecDraft and DesignSpec)

**Tool script logic:**
1. Receive target artefact (DesignSpecDraft or DesignSpec) in context
2. Display artefact payload to console with formatting
3. Prompt: `Review this <artefact_type>. Approve? (y/n/comment): `
4. Read user input:
   - `y` or `yes`: Output `{"artefact_type": "Review", "artefact_payload": "{}", "summary": "Approved by human"}`
   - `n` or `no`: Prompt for feedback, output `{"artefact_type": "Review", "artefact_payload": "{\"feedback\": \"...\"}", "summary": "Rejected by human"}`
   - `comment <text>`: Output `{"artefact_type": "Review", "artefact_payload": "{\"feedback\": \"<text>\"}", "summary": "Rejected by human"}`

**Implementation file:** `agents/human-reviewer/main.go`

**Key features:**
- No LLM required (pure console I/O)
- Timeout configurable (default: 5 minutes)
- Supports non-interactive mode for testing (env var `AUTO_APPROVE=true`)

**Implementation approach:** Wrapper for Claude API (e.g., via `anthropic` CLI or API client)

**Bidding strategy:** `exclusive` (processes approved DesignSpecDraft)

**Tool script logic:**
1. Receive approved DesignSpecDraft in context
2. Use Claude to expand draft into full DesignSpec following template
3. Output: `{"artefact_type": "DesignSpec", "artefact_payload": "<full markdown doc>", "summary": "..."}`


#### **2.4.4. CoderClaudeAgent**

**Purpose:** Implement code and tests based on approved design

**Implementation approach:** Wrapper for Claude API with git integration

**Bidding strategy:** `exclusive` (implements DesignSpec)

**Tool script logic:**
1. Receive DesignSpec in context
2. **If needed**: Create Question artefacts to ask ArchitectGeminiAgent for clarification
3. Use Claude to:
   - Write code based on design

4. Run tests locally to verify
5. Output: `{"artefact_type": "ChangeSet", "artefact_payload": "<JSON with commit info>", "summary": "..."}`

#### **2.4.6. TestRunnerAgent**

**Purpose:** Run automated tests on ChangeSet

**Implementation approach:** Shell script wrapper around `make test-failed`

**Bidding strategy:** `review` (reviews ChangeSet artefacts)

**Tool script logic:**
1. Receive ChangeSet artefact in context
2. Parse JSON to get commit_hash
3. Checkout commit: `git checkout <commit_hash>`
4. Run: `make test-failed`
5. If tests pass: Output `{"artefact_type": "Review", "artefact_payload": "{}", "summary": "All tests passed"}`
6. If tests fail: Output `{"artefact_type": "Review", "artefact_payload": "{\"test_failures\": \"<log excerpt>\"}", "summary": "Tests failed"}`

**Implementation file:** `agents/test-runner/run.sh`

**No LLM required** (pure shell script)

#### **2.4.7. SecurityScannerAgent**

**Purpose:** Run security scans on ChangeSet

**Implementation approach:** Shell script wrapper around `gosec` or similar tool

**Bidding strategy:** `review` (reviews ChangeSet artefacts)

**Tool script logic:**
1. Receive ChangeSet artefact in context
2. Parse JSON to get commit_hash and files_changed
3. Checkout commit: `git checkout <commit_hash>`
4. Run: `gosec ./...` or scan only files_changed
5. If no issues: Output `{"artefact_type": "Review", "artefact_payload": "{}", "summary": "No security issues"}`
6. If issues found: Output `{"artefact_type": "Review", "artefact_payload": "{\"security_issues\": \"<findings>\"}", "summary": "Security issues found"}`

**Implementation file:** `agents/security-scanner/run.sh`

**No LLM required** (pure shell script)

#### **2.4.8. LeadEngineerGeminiAgent**

**Purpose:** Final human-style code review before approval

**Implementation approach:** Gemini API with code review prompt

**Bidding strategy:** `exclusive` (final review after automated reviews pass)

**Tool script logic:**
1. Receive ChangeSet in context (only granted after automated reviews approve via M3.2)
2. Parse ChangeSet JSON to get commit_hash
3. Use git to get diff: `git show <commit_hash>`
4. Use Gemini to perform code review (check quality, maintainability, design adherence)
5. If approved: Output `{"artefact_type": "Terminal", "artefact_payload": "Implementation approved", "summary": "..."}`
6. If rejected: Output `{"artefact_type": "Review", "artefact_payload": "{\"feedback\": \"...\"}", "summary": "Code review feedback"}`

**System prompt (in holt.yml):**
```yaml
prompts:
  execution: |
    You are a lead engineer performing final code review. Your role is to:
    1. Review code for quality, maintainability, and design adherence
    2. Approve with Terminal artefact or reject with Review feedback
    3. Be constructive and specific in feedback

    This is the final gate before implementation is accepted. Be thorough but fair.
```

**Field semantics:**
- **`volumes`**: Optional array of Docker volume mount specifications
- Format: `"<source>:<destination>:<mode>"` (e.g., `"~/.config/gcloud:/root/.config/gcloud:ro"`)
- Tilde (`~`) at the beginning of source paths is expanded to the current user's home directory
- Common modes: `ro` (read-only, recommended for credentials), `rw` (read-write)

## **3. The 'how': implementation & testing plan**

### **3.1. Key design decisions & risks**

**Critical Design Decisions:**

1. **Two Separate Workflows vs. One Continuous Workflow**
   - **Decision:** Two distinct workflows with explicit handoff (Workflow 1 produces GoalDefined for Workflow 2)
   - **Rationale:** Clear separation of concerns (design vs. implementation), enables independent testing
   - **Trade-off:** Requires explicit GoalDefined creation in HumanReviewerAgent, adds one artefact to trace
   - **User benefit:** Can run design-only workflows or implementation-only workflows independently

2. **Human Review via Console vs. Web UI**
   - **Decision:** Console-based HumanReviewerAgent with stdin/stdout
   - **Rationale:** Simplest implementation, no web dependencies, testable via environment variables
   - **Trade-off:** Less user-friendly than web UI, requires terminal access
   - **Future enhancement:** Web UI overlay in Phase 5+
   - **Testing:** `AUTO_APPROVE=true` env var for E2E tests

3. **ChangeSet as JSON Payload vs. Structured Artefact**
   - **Decision:** ChangeSet artefact with JSON string payload
   - **Rationale:** Flexible schema, easy to extend, reviewers can parse JSON
   - **Trade-off:** Payload is opaque to orchestrator (no native JSON field access)
   - **Alternative considered:** New structural_type - rejected as over-engineering

4. **M4.3 Knowledge Artefact Provisioning**
   - **Decision:** Manual provisioning via `holt provision` before running workflow
   - **Rationale:** Explicit control, one-time cost, reusable across runs
   - **Trade-off:** Extra setup step, user must remember to provision
   - **Future enhancement:** Auto-provision on first run

5. **Gemini vs. Claude Role Assignment**
   - **Decision:** Gemini for design/architecture, Claude for detailed specs/code
   - **Rationale:** Demonstrates multi-LLM coordination, plays to each model's strengths
   - **Trade-off:** Requires API keys for both providers
   - **Alternative:** Single LLM for all - less impressive demo

6. **Exclusive vs. Parallel for Q&A Workflow**
   - **Decision:** ArchitectGeminiAgent uses `exclusive` strategy for answering Questions
   - **Rationale:** Questions are sequential, not parallelizable
   - **Trade-off:** Only one question answered at a time
   - **Alternative:** Parallel answering - rejected as complex and unnecessary

7. **Volume Mounts vs. Environment Variables for Credentials**
   - **Decision:** Support both methods - volumes for local license files, env vars for API keys
   - **Rationale:** Different LLM providers use different auth mechanisms (gcloud uses local config, Anthropic uses API keys)
   - **Trade-off:** Volume mounts expose host filesystem to containers, increasing security risk
   - **Security mitigation:** Document as power-user feature, require `:ro` mode, warn about trusted images only
   - **User benefit:** Enables local development workflows (e.g., gcloud auth) without managing API keys
   - **Alternative considered:** Environment variables only - rejected as insufficient for gcloud CLI auth

**Risks:**

1. **LLM API Availability and Cost**
   - **Risk:** Gemini or Claude API outages block workflow
   - **Probability:** Medium (external dependency)
   - **Impact:** High (workflow stalls indefinitely)
   - **Mitigation:**
     - Implement retry logic with exponential backoff in agent scripts
     - Document API key setup clearly
     - Provide mock agents for testing (return hardcoded responses)
   - **Monitoring:** Log all API calls with latency

2. **Human Reviewer Timeout in HumanReviewerAgent**
   - **Risk:** Human doesn't respond within timeout, claim stuck
   - **Probability:** Medium (user may be AFK)
   - **Impact:** Medium (workflow blocks, but resumable)
   - **Mitigation:**
     - Default 5-minute timeout, configurable
     - Clear console prompt with timeout warning
     - Support for extending timeout via stdin command
   - **Future:** M4.2 integration (pause/resume)

3. **Test Failures in TestRunnerAgent Creating Infinite Loop**
   - **Risk:** Tests consistently fail, M3.3 rework loops indefinitely
   - **Probability:** Medium (flaky tests or fundamental code issues)
   - **Impact:** Medium (exhausts max_review_iterations)
   - **Mitigation:**
     - Set reasonable max_review_iterations (default: 3)
     - TestRunnerAgent provides detailed failure logs in Review feedback
     - Human can inspect Failure artefact after max iterations reached
   - **Monitoring:** Alert on workflows exceeding 2 iterations

4. **Git State Corruption from Multiple Agent Checkouts**
   - **Risk:** TestRunnerAgent and SecurityScannerAgent checkout same commit, race condition
   - **Probability:** Low (agents run in separate containers)
   - **Impact:** High (git state corrupted, workflow fails)
   - **Mitigation:**
     - Each agent container has isolated workspace

**API key handling:**
- LLM API keys passed via environment variables
- Never logged or stored in artefacts
- Documented in setup guide

- Agents have isolated workspaces (Docker containers)
- Git commits are local only (no auto-push)
- ChangeSet includes commit hash for traceability

**Code execution:**
- TestRunnerAgent runs in isolated container
- SecurityScannerAgent runs in isolated container
- Workspace mode enforced (ro vs. rw)

**Secrets in design docs:**
- HumanReviewerAgent prompts reviewer to check for secrets
- SecurityScannerAgent can detect hardcoded credentials
- Documentation warns against committing secrets

**Volume mount security (NEW):**

This is the most significant security consideration for M4.5.

**Risk:** Mounting host directories gives containers access to host filesystem
- A malicious or compromised agent image could read, exfiltrate, or modify mounted files
- Credential files (e.g., `~/.config/gcloud`, `~/.aws/credentials`) contain sensitive authentication data
- If an agent image is compromised, attackers gain access to the user's cloud resources

**Mitigations:**

1. **Documentation as security boundary:**
   - Feature documented as "power-user feature for local development"
   - Clear warnings in holt.yml comments and documentation
   - Setup guide includes security checklist

2. **Read-only enforcement:**
   - All examples use `:ro` (read-only) mode
   - Documentation strongly recommends `:ro` for all credential mounts
   - Prevents malicious agents from modifying credential files

3. **Trust-based model:**
   - Users warned to only use this feature with **trusted agent images**
   - Recommend using official images or building from vetted source
   - Document how to inspect agent Dockerfile before use

4. **Minimal exposure principle:**
   - Mount only specific credential directories, not entire home directory
   - Example: `~/.config/gcloud:/root/.config/gcloud:ro` (specific path)
   - Avoid: `~:/root:ro` (entire home directory)

5. **Alternative highlighted:**
   - Environment variable method (API keys) documented as safer alternative
   - Recommend env vars for production/CI environments
   - Volume mounts positioned as "local development convenience"

**User security checklist (in documentation):**

Before using volume mounts, users should:
- [ ] Verify agent image is from trusted source
- [ ] Inspect Dockerfile if building custom images
- [ ] Use `:ro` mode for all credential mounts
- [ ] Mount only specific directories, not entire home
- [ ] Consider using API keys (env vars) instead if possible
- [ ] Use separate test credentials for Holt workflows

**Risk acceptance:**

This feature accepts the risk of host filesystem exposure in exchange for local development convenience. Users who cannot accept this risk should use environment variable authentication exclusively.

## **6. Error scenarios & edge cases**

### **6.1. Failure modes**

**LLM Agent failures:**

1. **API key missing or invalid:**
   - **Behavior:** Agent fails to initialize, exits with error
   - **Detection:** Error logged: "GEMINI_API_KEY not set" or "Authentication failed"
   - **Recovery:** User sets API key, restarts `holt up`
   - **Logging:** Clear error message with setup instructions
   - **Prevention:** Documentation checklist for API key setup

2. **LLM API rate limit exceeded:**
   - **Behavior:** Agent receives 429 error from API
   - **Detection:** HTTP 429 status code
   - **Recovery:** Agent retries with exponential backoff (3 retries, max 60s wait)
   - **Logging:** "Rate limit exceeded, retrying in Xs..."
   - **Prevention:** Document rate limits in README

3. **LLM produces malformed output:**
   - **Behavior:** Agent cannot parse LLM response as valid artefact
   - **Detection:** JSON parsing error or missing required fields
   - **Recovery:** Agent creates Failure artefact with error details
   - **Logging:** "Failed to parse LLM output: <error>"
   - **Prevention:** System prompts include output format examples

**HumanReviewerAgent failures:**

4. **Human review timeout:**
   - **Behavior:** No input received within timeout period (default: 5 minutes)
   - **Detection:** timeout.Context deadline exceeded
   - **Recovery:** Agent creates Failure artefact: "Human review timed out"
   - **Logging:** "No response received within 300s, failing claim"
   - **Prevention:** Clear timeout warning in console prompt

5. **Invalid input format:**
   - **Behavior:** User enters unexpected input (not y/n/comment)
   - **Detection:** Input parsing fails regex match
   - **Recovery:** Re-prompt with error: "Invalid input, please enter y/n/comment: "
   - **Logging:** "Invalid input received: <input>"
   - **Prevention:** Clear prompt with examples

**TestRunnerAgent failures:**

6. **Git checkout fails:**
   - **Behavior:** Commit hash not found in repository
   - **Detection:** `git checkout` returns non-zero exit code
   - **Recovery:** Agent creates Failure artefact: "Commit <hash> not found"
   - **Logging:** "Git checkout failed: <error>"
   - **Prevention:** CoderClaudeAgent validates commit exists before creating ChangeSet

7. **Tests fail to run (not just failing tests):**
   - **Behavior:** `make test-failed` exits with error other than test failures
   - **Detection:** Exit code other than 0 or 1, or no test output
   - **Recovery:** Agent creates Review with feedback: "Tests failed to execute"
   - **Logging:** "Test execution error: <stderr>"
   - **Prevention:** CoderClaudeAgent runs tests locally before creating ChangeSet

**SecurityScannerAgent failures:**

8. **gosec not installed:**
   - **Behavior:** `gosec` command not found
   - **Detection:** Command execution error
   - **Recovery:** Agent creates Failure artefact: "gosec not installed"
   - **Logging:** "gosec: command not found"
   - **Prevention:** Dockerfile ensures gosec is installed

**CoderClaudeAgent failures:**

9. **Git commit fails:**
   - **Behavior:** `git commit` returns non-zero exit code
   - **Detection:** Command execution error
   - **Recovery:** Agent retries commit with different message, or creates Failure
   - **Logging:** "Git commit failed: <error>"
   - **Prevention:** Ensure git config (user.name, user.email) set in Dockerfile

10. **Tests pass locally but fail in TestRunnerAgent:**
    - **Behavior:** CoderClaudeAgent runs tests (pass), but TestRunnerAgent fails
    - **Detection:** M3.3 rework loop triggered
    - **Recovery:** CoderClaudeAgent reviews test logs from Review feedback, fixes issue
    - **Logging:** Test failure logs included in Review feedback
    - **Prevention:** Document environment differences (CoderClaudeAgent vs. TestRunner)

### **6.2. Concurrency considerations**

**Race condition analysis:**

1. **Parallel reviewers (TestRunner and SecurityScanner):**
   - **Possibility:** Both checkout same commit simultaneously
   - **Protection:** Each agent runs in isolated Docker container with isolated workspace
   - **Impact:** No race (workspaces are independent)
   - **Safety:** Docker isolation guarantees no interference

2. **Multiple Q&A interactions:**
   - **Possibility:** CoderClaudeAgent creates multiple Questions simultaneously
   - **Protection:** M4.1 orchestrator handles Questions sequentially (exclusive claims)
   - **Impact:** Questions answered one at a time
   - **Safety:** Orchestrator guarantees exclusive Q&A

3. **Rework loop during review phase:**
   - **Possibility:** CoderClaudeAgent produces ChangeSet v2 while reviews of v1 are ongoing
   - **Protection:** M3.3 terminates original claim before creating feedback claim
   - **Impact:** v1 reviews are orphaned (no claim to complete)
   - **Safety:** Orchestrator phase state cleanup prevents double-transition

### **6.3. Edge case handling**

**Edge case: DesignSpecDraft is empty:**
- **Behavior:** LLM produces empty or very short draft (<100 chars)
- **Detection:** HumanReviewerAgent displays draft, human rejects
- **Recovery:** M3.3 rework loop → DesignerGeminiAgent tries again
- **Logging:** Review feedback: "Draft is too short or empty"

**Edge case: ChangeSet JSON is invalid:**
- **Behavior:** CoderClaudeAgent produces ChangeSet with malformed JSON payload
- **Detection:** TestRunnerAgent fails to parse JSON
- **Recovery:** TestRunnerAgent creates Review with feedback: "Invalid JSON in ChangeSet"
- **Logging:** JSON parsing error logged
- **Prevention:** CoderClaudeAgent validates JSON before creating artefact

**Edge case: All files deleted in ChangeSet:**
- **Behavior:** `files_changed` array is empty or contains only deletions
- **Detection:** SecurityScannerAgent has no files to scan
- **Recovery:** SecurityScannerAgent approves (no code to scan)
- **Logging:** "No files to scan, approving"

**Edge case: Knowledge artefact not provisioned:**
- **Behavior:** ArchitectGeminiAgent receives no Knowledge in context
- **Detection:** `loaded_knowledge` array is empty
- **Recovery:** Agent answers question without project context (less accurate)
- **Logging:** Warning: "No Knowledge artefacts found, answering without project docs"

**Edge case: GoalDefined is ambiguous:**
- **Behavior:** DesignerGeminiAgent cannot create draft without clarification
- **Detection:** Agent logic determines insufficient information
- **Recovery:** Agent creates Question artefact (M4.1)
- **Logging:** "Creating Question for goal clarification"

**Edge case: Human approves incomplete DesignSpec:**
- **Behavior:** ClaudeDesignAgent produces DesignSpec missing sections, human approves

## **8. AI agent implementation guidance**

### **8.1. Development approach**

**Start with the simplest agents:**
1. Begin with HumanReviewerAgent (no LLM dependencies)
2. Then TestRunnerAgent and SecurityScannerAgent (shell scripts)
3. Then mock LLM agents for testing (hardcoded responses)
4. Finally real LLM agents (Gemini, Claude)

**Implement comprehensive error handling from the beginning:**
- All API calls with retry logic and timeouts
- All git operations with error checking
- All JSON parsing with validation
- Clear error messages for users

**Write tests before implementation (TDD approach):**
- Start with unit tests for HumanReviewerAgent input parsing
- Then integration tests for TestRunner/SecurityScanner
- Then E2E tests with mock LLM agents
- Finally E2E tests with real LLM agents (optional, requires API keys)

**Use defensive programming:**
- Always validate artefact schemas before processing
- Never assume context_chain is non-empty
- Handle missing environment variables gracefully
- Log all errors with actionable messages

### **8.2. Common pitfalls to avoid**

**HumanReviewerAgent pitfalls:**

1. **Not handling Ctrl+C (SIGINT):**
   - Always install signal handler to clean up
   - Create Failure artefact on interrupt
   - Don't leave claim hanging

2. **Not sanitizing user input:**
   - Use strings.TrimSpace() on all input
   - Escape special characters in feedback
   - Validate input length (max 1000 chars)

**TestRunnerAgent pitfalls:**

3. **Not isolating test runs:**
   - Use `git worktree` or clean checkout for each run
   - Don't assume clean state
   - Clean up after test execution

4. **Not capturing full test output:**
   - Capture both stdout and stderr
   - Include full test logs in Review feedback (truncate if >10KB)
   - Preserve formatting for readability

**LLM Agent pitfalls:**

5. **Not validating LLM output:**
   - Always parse JSON before creating artefact
   - Check required fields exist
   - Validate artefact_type matches expected value

6. **Not handling API timeouts:**
   - Set reasonable timeout (30s for design, 60s for code)
   - Retry on timeout with backoff
   - Create Failure artefact if all retries exhausted

**CoderClaudeAgent pitfalls:**

7. **Not running tests before creating ChangeSet:**
   - Always run `make test-failed` locally
   - Don't create ChangeSet if tests fail locally
   - Include test results in ChangeSet payload

8. **Not setting git config:**
   - Ensure `user.name` and `user.email` set in Dockerfile
   - Use consistent commit message format
   - Don't push commits (local only)

### **9.2. Rollback and disaster recovery**

**Rollback plan:**

M4.5 is purely additive (new agents only), so rollback is trivial:
1. Stop workflow: `holt down`
2. Remove M4.5 agents from holt.yml
3. Restart: `holt up`

No data migration or cleanup required.

**Disaster recovery scenarios:**

1. **LLM API outage:**
   - **Recovery:** Wait for API to recover, or switch to alternative LLM
   - **Workaround:** Use mock agents for testing
   - **Prevention:** Document backup LLM configuration

2. **Git repository corruption:**
   - **Recovery:** Restore from git remote or backup
   - **Impact:** ChangeSet commit hashes may be invalid
   - **Prevention:** Never run `git gc` or `git prune` during workflow

3. **Human reviewer unavailable (timeout):**
   - **Recovery:** Claim terminates with Failure artefact, user can restart
   - **Workaround:** Set AUTO_APPROVE=true for testing
   - **Prevention:** Document timeout configuration

**Recovery time objectives:**

- Workflow restart after failure: <2 minutes
- LLM API recovery (switch provider): <10 minutes
- Git repository recovery: <30 minutes
