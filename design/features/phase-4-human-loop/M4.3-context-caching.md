# **Feature design: M4.3 - Context Caching**

**Purpose**: To design a mechanism for an agent to perform a one-time, expensive context discovery and cache that context on the Blackboard, keyed by a globally unique name, for efficient reuse across all runs within a work thread.

**Scope**: A new "Declarative Caching" mechanism via the agent tool contract, enhancement of the Agent Pup's context assembly, and a new CLI command for manual provisioning.

Associated phase: Human-in-the-Loop
Status: Approved

---

## **1\. The 'why': goal and success criteria**

### **1.1. Goal statement**

To provide a mechanism for an agent to dynamically select and cache large, reusable blocks of context on the Blackboard, keyed by a globally unique name, for efficient reuse in subsequent runs, even across rework cycles.

### **1.2. User story**

As an agent developer building a complex coding agent, my agent needs to read several large design documents to understand its task. I want the agent to perform the expensive logic to determine which documents to read **only once** per work thread. I then want that context to be automatically and efficiently provided to my agent on all subsequent runs for that task, so that my agent is both efficient and powerful.

### **1.3. Success criteria**

1.  An agent's tool script can return a `checkpoints` array in its final JSON output. The Agent Pup processes this by creating globally unique, versioned `Knowledge` artefacts on the Blackboard and linking them to the current work thread.
2.  On a subsequent run for the same logical thread, the pup automatically finds the linked `Knowledge` artefacts, provides their content in a `knowledge_base` map, and provides their names in a `loaded_knowledge` list within the JSON sent to the tool script.
3.  A human operator can use the `holt provision` command to manually create and version `Knowledge` artefacts from external sources like files and URLs.

### **1.4. Non-goals**

-   The Orchestrator will not be changed. This mechanism is handled entirely by the Agent Pup and CLI.
-   The pup will not implement a special "append" mode for knowledge. Appending is the responsibility of the agent tool via a read-modify-write cycle that creates a new version.

## **2\. The 'what': component impact analysis**

### **2.1. Blackboard changes**

*   **New/modified data structures:**
    *   **New `structural_type`: `Knowledge`**: A new enum value. `Knowledge` artefacts are passive data blobs ignored by the Orchestrator for claim creation.
    *   **`Knowledge` Artefact Schema**: Standard artefact fields plus `context_for_roles`.
        - `type`: string (This is the human-readable, globally unique `<knowledge_name>`)
        - `context_for_roles`: JSON array of strings (e.g., `["coder*", "docs-agent"]`)

*   **New Redis keys:**
    *   **`holt:<instance>:knowledge_index`**: A Redis `HASH` that maps a `<knowledge_name>` to the `logical_id` of its version thread. This enforces global uniqueness.
        -   Example: `HSET holt:my-instance:knowledge_index "go-sdk-docs" "<uuid-of-thread>"`
    *   **`holt:<instance>:thread_context:<logical_id>`**: A Redis `SET` containing the unique `id`s of the `Knowledge` artefacts attached to a specific logical thread of work.

### **2.2. Orchestrator changes**

*   **New/modified logic**: The orchestrator must be updated to **ignore** artefacts with `structural_type: "Knowledge"` when scanning for new claims to process.

### **2.3. Agent pup changes**

*   **New/modified logic:**
    1.  **Context Assembly Enhancement**: Before running a tool script, the pup's context assembly logic is enhanced:
        -   It traverses the artefact history to find all unique `logical_id`s in the current work thread.
        -   For each `logical_id`, it queries the corresponding `thread_context` Redis `SET` to get all attached `Knowledge` artefact IDs.
        -   It loads all unique `Knowledge` artefacts, filters them by matching the pup's own role against the `context_for_roles` glob patterns.
        -   **Merge Strategy**: If multiple `Knowledge` artefacts with the same `type` (name) are found, the pup **must** select the one with the highest `version` number ("latest version wins").
        -   It injects the final, filtered knowledge into the `stdin` JSON.
        -   It sets a new boolean flag, `context_is_declared`, to `true` if any `Knowledge` artefacts were found for the agent's role.

    2.  **Output Processing Enhancement (Declarative Caching)**: After the tool script executes, the pup inspects the output JSON for a new, optional field named `checkpoints`.
        -   For each item in the `checkpoints` array, the pup performs an **atomic** operation (using a Redis transaction or Lua script) to create/version the `Knowledge` artefact:
            a. Check the `knowledge_index` for the `knowledge_name`.
            b. If it exists, use the retrieved `logical_id` to create a new version.
            c. If it does not exist, create a new `Knowledge` artefact (v1) and register its new `logical_id` in the `knowledge_index`.
            d. Add the new `Knowledge` artefact's ID to the `thread_context` set for the current work thread's `logical_id`.

*   **Changes to the tool execution contract (stdin/stdout):**
    *   **Input (stdin)**: Three new optional fields are added:
        ```json
        {
          ...
          "context_is_declared": true, // Is any knowledge already cached for this agent?
          "knowledge_base": { "go-sdk-docs": "..." }, // The full content
          "loaded_knowledge": ["go-sdk-docs"] // The names of loaded knowledge
        }
        ```
    *   **Output (stdout)**: A new optional `checkpoints` array is added:
        ```json
        {
          "artefact_type": "...",
          "checkpoints": [
            {
              "knowledge_name": "go-sdk-docs",
              "knowledge_payload": "...",
              "target_roles": ["coder*"]
            }
          ]
        }
        ```

### **2.4. CLI changes**

*   **New/modified commands:**
    *   **`holt provision --name <name> [source flags]`**: Manually creates/versions a `Knowledge` artefact.
        -   `--from-file <path>`
        -   `--from-literal "<text>"`
        -   `--from-url <url>`
        -   `--from-command "<command>"`
        -   `--target-roles <roles>`: Comma-separated list of role globs (defaults to `*`).

## **3\. The 'how': implementation & testing plan**

### **3.1. Key design decisions & risks**

*   **Key Decision**: The "Declarative Caching" model, where an agent produces its main work and declares context to be cached in a single atomic output. This is efficient and avoids complex multi-turn workflows.
*   **Key Decision**: Tying cached context to the `logical_id` of the work thread. This ensures context persists across rework cycles.
*   **Key Decision**: A global `knowledge_index` hash to enforce that a `knowledge_name` is a globally unique key for a version thread. This prevents naming collisions and ensures predictability.
*   **Risk**: Race conditions during the first creation of a `Knowledge` artefact from two parallel agents. This **must** be mitigated by using a Redis transaction (`MULTI`/`EXEC`) or a Lua script in the pup's checkpointing logic to ensure the check-and-set on the `knowledge_index` is atomic.

### **3.2. Implementation steps**

1.  **[Blackboard]** Add `Knowledge` to `StructuralType` enum. Add `knowledge_index` and `thread_context` key patterns.
2.  **[Orchestrator]** Update claim creation logic to ignore artefacts of type `Knowledge`.
3.  **[Pup]** Implement the enhanced context assembly logic, including the "latest version wins" merge strategy.
4.  **[Pup]** Implement the enhanced output processing logic, including the **atomic** creation/versioning of `Knowledge` artefacts using the `knowledge_index`.
5.  **[CLI]** Implement the new `holt provision` command with all source flags.
6.  **[Tests]** Implement all unit, integration, and E2E tests as specified in the testing strategy.

### **3.4. Testing strategy**

*   **Unit tests**: Pup's context assembly (glob matching, version merging), pup's checkpoint processing (atomic index check-and-set), `holt provision` source handlers.
*   **Integration tests**: Pup's interaction with Redis for creating artefacts and updating both the `knowledge_index` and `thread_context` sets.
*   **E2E tests**:
    1.  **Full Lifecycle Test**: An agent runs on a task. On its first run, its `stdin` has `context_is_declared: false`. It produces a `checkpoint`. On a second run (after a forced rework), verify its `stdin` has `context_is_declared: true` and the correct `knowledge_base` content, and that it does *not* produce a checkpoint again.
    2.  **Race Condition Test**: Two agents running in parallel attempt to create a checkpoint with the *same new `knowledge_name`* at the same time. Verify that only one `logical_id` thread is created and that one of the checkpoints results in `v2` of that thread, not a new, conflicting thread.
