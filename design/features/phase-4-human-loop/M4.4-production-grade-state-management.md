# **Feature design: M4.4: Production-Grade State Management**

Associated phase: Human-in-the-Loop
Status: Draft

**Template purpose:** This document is a blueprint for a single, implementable milestone. Its purpose is to provide an unambiguous specification for a developer (human or AI) to build a feature that is consistent with Holt's architecture and guiding principles.

## **1. The 'why': goal and success criteria**

### **1.1. Goal statement**

Enable Holt to support both externally-managed Redis instances for production deployments and optionally password-protected, Holt-managed Redis for secure development environments.

### **1.2. User story**

As a platform engineer deploying Holt to production, I want to configure Holt to connect to my organization's managed Redis instance (e.g., AWS ElastiCache, Azure Cache for Redis) instead of running a local Redis container, so that I can leverage enterprise-grade security, high availability, backups, and compliance controls. Additionally, as a security-conscious developer, I want to enable password protection on my local Holt-managed Redis instance to prevent unauthorized access during development.

### **1.3. Success criteria**

1. A user can configure `services.redis.uri: "redis://my-redis.example.com:6379"` in `holt.yml`, run `holt up`, and Holt successfully connects to the external Redis without attempting to start a Redis container. The `holt down` command leaves the external Redis running.

2. A user can configure `services.redis.password: "${REDIS_PASSWORD}"` in `holt.yml`, set the environment variable, run `holt up`, and all components (orchestrator, agents) successfully connect to the password-protected, Holt-managed Redis container.

3. A user with a legacy `holt.yml` (containing only `services.redis.image: redis:7-alpine` or no `services.redis` section) can run `holt up` and the system defaults to starting an un-passworded, Holt-managed Redis container with a clear informational log message.

### **1.4. Non-goals**

- Persistent Redis storage with data volumes (deferred to future work)
- `holt destroy` command for permanent data deletion (deferred to future work)
- Advanced TLS configuration (custom CA bundles, certificate verification options)
- Redis Cluster or Sentinel support
- Automatic Redis provisioning or discovery

## **2. The 'what': component impact analysis**

### **2.1. Blackboard changes**

**No changes.**

The Redis data structures (Artefact, Claim, Bids schemas), Pub/Sub channels, and key patterns remain unchanged. This feature affects only how Holt connects to Redis, not the data model.

### **2.2. Orchestrator changes**

**New/modified logic:**
- **Connection initialization**: The orchestrator must read the `REDIS_URL` environment variable (existing behavior) but now this variable may point to an external Redis or include authentication credentials.
- **No orchestration logic changes**: The core claim processing, bidding, and event handling logic is unaffected.

**New/modified configurations (holt.yml):**

The `services.redis` section in `holt.yml` is enhanced with new fields:

```yaml
services:
  redis:
    # Mutually exclusive modes:
    uri: "redis://user:pass@external-host:6379"  # External mode (NEW)
    image: "redis:7-alpine"                      # Managed mode (existing)
    password: "${REDIS_PASSWORD}"                # Optional: password for managed mode (NEW)
    resources: { ... }                           # Existing field (unchanged)
```

**Validation rules:**
1. If both `uri` and `image` are defined, the config loader must return an error.
2. If neither `uri` nor `image` is defined (or `services.redis` section is missing), default to managed mode with `image: redis:7-alpine`.
3. The `password` field is only valid when using managed mode (ignored in external mode with a warning).
4. The `uri` and `password` fields support environment variable expansion using `${VAR_NAME}` syntax.

### **2.3. Agent pup changes**

**No changes.**

Agent pups already receive the Redis connection string via the `REDIS_URL` environment variable. This feature simply changes how that variable is constructed by the CLI during `holt up`. The pup's connection logic is unaffected.

### **2.4. CLI changes**

**New/modified commands:**

**`holt up` changes:**

1. **Config loading enhancement**: Implement environment variable expansion for the entire `holt.yml` file before YAML parsing. This allows `${VAR_NAME}` syntax in any configuration field.

2. **Mode detection**: After loading config, determine Redis mode:
   - If `services.redis.uri` is present → External mode
   - Otherwise → Managed mode (use `services.redis.image` or default to `redis:7-alpine`)

3. **External mode behavior**:
   - Skip Redis container creation entirely
   - Validate that the provided URI is parseable (basic syntax check)
   - Use the URI directly as `REDIS_URL` for orchestrator and agents
   - Log: `INFO: Using external Redis: <sanitized-uri>` (hide credentials)

4. **Managed mode with password**:
   - If `services.redis.password` is set, start Redis container with `--requirepass <password>` flag
   - Construct internal URI: `redis://:<password>@<container-name>:6379`
   - Use this constructed URI as `REDIS_URL` for orchestrator and agents
   - Log: `INFO: Starting Holt-managed Redis with authentication enabled`

5. **Managed mode without password** (existing behavior + new log):
   - Start Redis container without password protection
   - Use simple URI: `redis://<container-name>:6379`
   - Log: `INFO: No external Redis URI found. Starting default Holt-managed Redis container.`

**`holt down` changes:**

1. **Mode detection**: Read the orchestrator container's `REDIS_URL` environment variable (via `docker inspect`) to determine if Redis was externally managed.

2. **External mode behavior**: Skip Redis container cleanup. Log: `INFO: Skipping Redis cleanup (external mode)`.

3. **Managed mode behavior**: Stop and remove Redis container (existing behavior).

**Changes to user output:**
- Clear log messages distinguishing external vs. managed mode during `holt up`
- Sanitized URI display (hide passwords/credentials) in logs
- Error messages for configuration validation failures (e.g., both `uri` and `image` defined)
- Warning if `password` is set in external mode (ignored)

## **3. The 'how': implementation & testing plan**

### **3.1. Key design decisions & risks**

**Decision 1: Environment variable expansion happens before YAML parsing**
- **Rationale**: This is the most secure approach. By expanding variables before unmarshaling, we prevent secrets from appearing in the config file itself. This also allows env vars to be used in any field, not just specific ones.
- **Implementation**: Read file as bytes, perform regex-based `${VAR_NAME}` substitution, then unmarshal YAML.
- **Risk**: Malformed env var syntax (e.g., `${MISSING_VAR}`) could cause confusing errors. Mitigation: Clear error messages with the specific variable name.

**Decision 2: All components use URI-based connection strings**
- **Rationale**: This provides a clean abstraction. Whether Redis is external or managed, orchestrator and agents always receive a URI. This simplifies the code and reduces conditional logic.
- **Implementation**: The CLI constructs the appropriate URI for each mode and passes it via `REDIS_URL` environment variable.
- **Risk**: None significant. This is the standard Redis connection pattern.

**Decision 3: External mode detection in `holt down` via container inspection**
- **Rationale**: We cannot rely on `holt.yml` being present or unchanged during `holt down`. The orchestrator container's environment variables are the source of truth for how the instance was configured.
- **Implementation**: Use `docker inspect` to read `REDIS_URL` from orchestrator container. If it doesn't point to a Holt-managed container name, skip Redis cleanup.
- **Risk**: If the orchestrator container is already removed, we can't detect mode. Mitigation: Try to list Redis containers by instance label and skip cleanup if none found.

**Decision 4: Fail-fast validation for mutually exclusive modes**
- **Rationale**: Ambiguous configuration is dangerous. Requiring explicit choice between external and managed prevents operator error.
- **Implementation**: Config validation returns error if both `uri` and `image` are present.
- **Risk**: None. This is a simple validation check.

### **3.2. Implementation steps**

**Phase 1: Config package enhancements**

1. [Config] Implement `expandEnvVars(yamlBytes []byte) ([]byte, error)` function
   - Use regex to find all `${VAR_NAME}` patterns
   - Replace with `os.Getenv(VAR_NAME)`
   - Return error if variable is not set with message: `"Environment variable 'VAR_NAME' referenced in holt.yml is not set"`

2. [Config] Add new fields to `ServiceOverride` struct:
   ```go
   type ServiceOverride struct {
       Image    string           `yaml:"image,omitempty"`
       URI      string           `yaml:"uri,omitempty"`      // NEW
       Password string           `yaml:"password,omitempty"` // NEW
       Resources *ResourcesConfig `yaml:"resources,omitempty"`
   }
   ```

3. [Config] Enhance `Load()` function to call `expandEnvVars()` before unmarshaling

4. [Config] Add validation in `Validate()` to check mutual exclusivity of `uri` and `image`:
   - Error if both are present: `"Invalid configuration: services.redis.uri and services.redis.image are mutually exclusive"`

5. [Config] Add validation warning if `password` is set when `uri` is also set:
   - Warning: `"services.redis.password is ignored when using external Redis (uri mode)"`

**Phase 2: CLI `holt up` modifications**

1. [CLI/up] After config load, add `determineRedisMode(cfg)` function returning `(mode string, config RedisConfig, error)`
   - Returns `"external"` mode with provided URI, or `"managed"` mode with image/password

2. [CLI/up] In `createInstance()`, add conditional logic based on mode:
   - If external mode: skip Redis container creation, validate URI format, use provided URI
   - If managed mode with password: construct Redis container with `Cmd: []string{"redis-server", "--requirepass", password}`
   - If managed mode without password: existing behavior

3. [CLI/up] Add `sanitizeRedisURI(uri string) string` helper function for logging:
   - Replace password portion with `***` (e.g., `redis://***@host:6379`)

4. [CLI/up] Update orchestrator and agent container creation to use constructed/provided URI for `REDIS_URL` env var

5. [CLI/up] Add appropriate logging for each mode

**Phase 3: CLI `holt down` modifications**

1. [CLI/down] Add `detectRedisMode(ctx, cli, instanceName)` function:
   - Inspect orchestrator container's `REDIS_URL` env var
   - Check if it matches pattern `redis://*holt-<instanceName>-redis*` (managed) or external
   - Return boolean `isManaged`
   - Fallback: If orchestrator missing, check for Redis container by label

2. [CLI/down] In `runDown()`, conditionally skip Redis cleanup if external mode

3. [CLI/down] Add appropriate logging

**Phase 4: Testing**

1. [Unit] Test `expandEnvVars()` with various inputs (valid vars, missing vars, no vars, special characters)
2. [Unit] Test config validation (uri+image error, password warning in external mode, missing env vars)
3. [Unit] Test `sanitizeRedisURI()` with various URI formats
4. [E2E] Test external Redis mode (described in section 3.4)
5. [E2E] Test managed Redis with password (described in section 3.4)
6. [E2E] Test authentication failure scenario
7. [E2E] Verify all existing E2E tests still pass (backward compatibility)

### **3.3. Performance & resource considerations**

**Resource usage:**
- **External mode**: Reduces local resource usage by eliminating Redis container (CPU: ~10-20MB, Memory: ~10-30MB saved)
- **Managed mode with password**: No additional resource usage beyond existing Redis container
- **Config loading**: Environment variable expansion adds negligible overhead (<1ms for typical config files)

**Scalability limits:**
- External Redis can be scaled independently based on cloud provider capabilities
- Managed Redis continues to be single-instance (suitable for development/testing only)

**Performance requirements:**
- Environment variable expansion must complete in <100ms for typical config files (<10KB)
- Redis connection validation during `holt up` must timeout within 5 seconds with clear error message
- No impact on orchestrator or agent runtime performance (connection logic unchanged)

- `config.expandEnvVars()`: Test with set/unset variables, malformed syntax, empty values, special characters

- `sanitizeRedisURI()`: Test with various URI formats (with/without password, username, different schemes)
- URI parsing/construction logic for managed mode with password

**Integration tests:**
- Config loading with actual environment variables → YAML parsing → validation
- `holt up` with each mode → verify correct container creation/skip behavior

## **4. Principle compliance check**

### **4.1. YAGNI (You Ain't Gonna Need It)**

**No new third-party dependencies introduced.**

This feature leverages existing components:
- Go's `os.ExpandEnv()` or regex for environment variable expansion
- Existing `go-redis` client library (already used)
- Docker API (already used for container management)

The environment variable expansion uses Go's standard library. The Redis URI parsing is handled by the existing `go-redis` library.

### **4.2. Auditability**

**No new artefacts created.**

This feature affects infrastructure configuration, not the workflow execution model. The blackboard's artefact trail remains unchanged. However, audit considerations include:

- **Configuration auditability**: The resolved Redis mode (external vs. managed) is logged during `holt up` for operational transparency.
- **Connection string sanitization**: Logs must never display passwords or credentials from URIs. Use sanitized logging (e.g., `redis://***@host:port`).
- **No state mutation**: This feature doesn't add, modify, or delete artefacts on the blackboard. It only changes how components connect to the blackboard.

### **4.3. Small, single-purpose components**

**Confirmed: Changes are localized and maintain separation of concerns.**

- **Config package**: Enhanced with env var expansion and new fields. Responsibility remains configuration loading and validation.
- **CLI (`holt up`)**: Enhanced with mode detection and conditional Redis management. Responsibility remains instance lifecycle management.
- **CLI (`holt down`)**: Enhanced with mode detection for cleanup decisions. Responsibility remains resource teardown.
- **Orchestrator**: No changes. Continues to use `REDIS_URL` from environment.
- **Agent pup**: No changes. Continues to use `REDIS_URL` from environment.

No tight coupling introduced. The mode detection logic is isolated in the CLI and doesn't leak into orchestrator or pup logic.

### **4.4. Security considerations**

**Positive security impact:**

1. **Secrets protection**: Environment variable expansion prevents credentials from being committed to version control in `holt.yml` files. Users can use `password: "${REDIS_PASSWORD}"` instead of hardcoding.

2. **External Redis security**: Enables customers to leverage enterprise-grade security features:
   - TLS encryption (`rediss://` scheme)
   - Customer-managed authentication
   - Network isolation (VPC, private endpoints)
   - Compliance controls (encryption at rest, key rotation)

3. **Managed mode password protection**: Allows developers to enable authentication on local Redis instances, preventing unauthorized access on shared development machines.

**Security risks and mitigations:**

1. **Risk**: Passwords in connection strings appear in environment variables and container inspect output.
   - **Mitigation**: This is standard practice for containerized applications. Users in high-security environments should use external Redis with properly secured connection strings (e.g., fetched from secret managers at runtime).

2. **Risk**: Sanitized logging might fail to hide credentials if URI parsing is incorrect.
   - **Mitigation**: Use battle-tested URI parsing from `go-redis` or Go's `url.Parse()`. Test with various URI formats including edge cases.

3. **Risk**: `holt down` mode detection via container inspection could expose Redis URI to unauthorized users who can run `docker inspect`.
   - **Mitigation**: This is inherent to Docker's design. Users concerned about this should use proper host-level access controls and Docker socket permissions.

4. **Risk**: No validation of TLS certificate validity for `rediss://` connections.
   - **Mitigation**: Document this limitation. Recommend users with strict TLS requirements validate their certificates externally or wait for future enhancements to support custom CA bundles.

**Attack surfaces:**
- No new network listeners or exposed ports
- No new file system access patterns
- Container isolation boundaries unchanged

### **4.5. Backward compatibility**

**Fully backward compatible.**

1. **Existing configs with `image` field**: Continue to work identically. No migration required.

2. **Existing configs with no `services.redis` section**: Default to managed mode with `redis:7-alpine` (existing behavior). New informational log message added but behavior unchanged.

3. **Existing E2E tests**: All tests use default managed mode and should pass without modification.

4. **API/data structure compatibility**: No changes to blackboard schema, Pub/Sub channels, or Redis key patterns. Existing running instances (if persistence is added later) will be compatible.

**Deprecation path:**
No features are deprecated. The `image` field remains fully supported and is the recommended approach for development/testing environments.

**Breaking changes:**
None. The new `uri` field is purely additive. The mutual exclusivity check only rejects invalid new configurations, not existing ones.

### **4.6. Dependency impact**

**No changes to external dependencies.**

- **Redis version requirements**: Unchanged. Continues to support Redis 6.x+ (tested with 7-alpine).
- **Docker requirements**: Unchanged. Continues to require Docker 20.10+.
- **Go version**: Unchanged. Continues to require Go 1.21+.
- **Build dependencies**: No new Go modules required. Uses existing `go-redis` and Docker client libraries.

**Impact on development environment:**
- No new tools or services required for development
- E2E tests will need to spawn external Redis containers but use existing Docker capabilities
- CI/CD pipelines unaffected (backward compatible)

**Redis client compatibility:**
The `go-redis` library already supports:
- Both `redis://` and `rediss://` URI schemes
- Password authentication in URI format
- Username/password authentication
- This feature leverages existing capabilities without library upgrades

## **6. Error scenarios & edge cases**

### **6.1. Failure modes**

**1. External Redis unreachable during `holt up`**
- **Cause**: Network issue, wrong hostname, firewall blocking connection
- **Behavior**: `holt up` must fail immediately with clear error message
- **Error message**: `"Failed to connect to external Redis at redis://host:6379: connection refused. Verify the URI and network connectivity."`
- **Rollback**: Partial resources (network, orchestrator container) must be cleaned up automatically

**2. External Redis authentication failure**
- **Cause**: Wrong password in URI, password changed on Redis server
- **Behavior**: `holt up` must fail during orchestrator startup with authentication error
- **Error message**: `"Redis authentication failed: NOAUTH Authentication required. Verify the password in services.redis.uri"`
- **Rollback**: Full rollback of created resources

**3. Environment variable not set during config loading**
- **Cause**: User specified `password: "${REDIS_PASSWORD}"` but forgot to export the variable
- **Behavior**: Config loader must fail with clear error
- **Error message**: `"Environment variable 'REDIS_PASSWORD' referenced in holt.yml is not set"`
- **Implementation**: Check if expanded value is empty and return error if variable was expected

**4. Both `uri` and `image` defined in config**
- **Cause**: User error or config merge issue
- **Behavior**: Config validation must fail before any resources are created
- **Error message**: `"Invalid configuration: services.redis.uri and services.redis.image are mutually exclusive. Use 'uri' for external Redis or 'image' for managed Redis."`
- **Implementation**: Check in `Validate()` method

**5. Malformed Redis URI**
- **Cause**: Typo in URI, invalid syntax
- **Behavior**: Fail during config validation with parsing error
- **Error message**: `"Invalid Redis URI in services.redis.uri: <parse error details>"`
- **Implementation**: Use Go's `url.Parse()` or `go-redis` URI parsing to validate

**6. TLS required but plaintext connection attempted**
- **Cause**: External Redis requires TLS but user specified `redis://` instead of `rediss://`
- **Behavior**: Connection fails during orchestrator startup
- **Error message**: `"Connection to Redis failed. If the Redis server requires TLS, use 'rediss://' scheme in the URI."`
- **Implementation**: Detect TLS-related errors from go-redis and provide helpful hint

**7. Docker socket unavailable when starting managed Redis**
- **Cause**: Docker daemon not running, permission issue
- **Behavior**: Existing error handling (already implemented)
- **No changes needed**: This is covered by existing Docker client error handling

**8. `holt down` when orchestrator container is missing**
- **Cause**: Container manually deleted, previous cleanup failure
- **Behavior**: Unable to detect Redis mode from orchestrator env vars
- **Fallback**: Attempt to detect Redis container by instance label. If found, remove it (assume managed mode). If not found, log warning and proceed.
- **Error message**: `"Warning: Unable to detect Redis mode (orchestrator container not found). Checking for managed Redis container..."`

### **6.2. Concurrency considerations**

**No new concurrency risks introduced.**

Reasoning:
1. **Config loading**: Happens once during `holt up`, single-threaded, before any concurrent operations
2. **Redis connection**: Each component (orchestrator, agents) connects independently using provided URI. The `go-redis` client handles connection pooling and concurrency internally.
3. **Mode detection in `holt down`**: Happens once during teardown, single-threaded
4. **Environment variable expansion**: Happens once during config load, before any goroutines are spawned

**Existing concurrency patterns preserved:**
- Agent containers launched in parallel (existing M3.1 behavior)
- Orchestrator claim processing (existing concurrent event handling)
- Agent pup concurrent loops (existing claim watcher + work executor)

**No shared state introduced:**
All new logic is stateless function calls or single-threaded CLI operations.

### **6.3. Edge case handling**

**1. Empty password field**
- **Scenario**: `password: ""` or `password: "${UNSET_VAR}"` that expands to empty string
- **Behavior**: Treat as no password (start Redis without `--requirepass`)
- **Log**: `INFO: services.redis.password is empty, starting Redis without authentication`

**2. Password contains special characters**
- **Scenario**: Password includes spaces, quotes, shell metacharacters
- **Behavior**: Must handle correctly when passing to Redis via Docker
- **Implementation**: Use Docker API's string array args, not shell command string, to avoid escaping issues
- **Test**: Include test case with password like `"p@ss w0rd!$#"`

**3. URI with no port specified**
- **Scenario**: `redis://host` or `rediss://host`
- **Behavior**: Redis defaults to port 6379. The `go-redis` client handles this automatically.
- **No special handling needed**

**4. URI with database number**
- **Scenario**: `redis://host:6379/2` (specifying database 2)
- **Behavior**: Should work correctly. The `go-redis` client supports this in URI format.
- **Test**: Include test case verifying different database numbers work

**5. Multiple agents referencing the same Redis**
- **Scenario**: All agents connect to the same Redis (normal behavior)
- **Behavior**: No issue. Redis supports multiple concurrent clients. This is existing behavior.
- **No special handling needed**

**6. Very long passwords or URIs**
- **Scenario**: Password or URI exceeds typical length (e.g., 1KB+)
- **Behavior**: Environment variables support large values. Docker API supports long command args.
- **Limitation**: Document that URIs must be <4KB (practical Docker env var limit)
- **Validation**: Add length check in config validation (e.g., max 4KB for URI)

**7. External Redis becomes unavailable after startup**
- **Scenario**: `holt up` succeeds, then external Redis crashes or network fails
- **Behavior**: Orchestrator and agents will experience Redis connection errors (existing behavior)
- **No changes needed**: This is an operational issue, not a configuration issue. Existing error handling and health checks apply.

**8. User changes holt.yml between `holt up` and `holt down`**
- **Scenario**: User runs `holt up` with managed Redis, then changes config to external Redis, then runs `holt down`
- **Behavior**: `holt down` uses container inspection (source of truth), not current `holt.yml`
- **Result**: Correctly stops managed Redis because that's what was actually running
- **This is correct behavior**: The running instance's configuration is authoritative

**9. Localhost vs 127.0.0.1 in URIs**
- **Scenario**: `redis://localhost:6379` vs `redis://127.0.0.1:6379`
- **Behavior**: Both should work. Docker networking resolves both correctly from inside containers.
- **Note**: For external Redis on host, agents/orchestrator need to use `host.docker.internal` on Docker Desktop (macOS/Windows) or the bridge gateway IP (Linux)
- **Documentation**: Add troubleshooting guidance for localhost URIs

## **8. AI agent implementation guidance**

### **8.1. Development approach**

1. **Start with config package**: Implement environment variable expansion and validation first. This is the foundation for all other changes and can be unit-tested independently.

2. **Test-driven development**: Write unit tests for config changes before implementing CLI logic. This ensures the validation rules are correct before integrating with Docker operations.

3. **Implement `holt up` mode detection**: Add the mode detection function and test it with various config combinations before modifying container creation logic.

4. **Incremental integration**: Add external mode support to `holt up`, verify it works, then add managed mode with password, then update `holt down`.

5. **Error handling first**: For each new code path, implement error handling and validation before the happy path. This prevents partial failures from leaving resources orphaned.

6. **Sanitization is critical**: Any logging or error messages that might display Redis URIs must sanitize credentials. Test this explicitly.

### **8.2. Common pitfalls to avoid**

1. **Forgetting to sanitize Redis URIs in logs**: Never log passwords. Use helper function like `sanitizeRedisURI(uri string) string` that replaces password portion with `***`.

2. **Not handling missing orchestrator container in `holt down`**: When detecting mode via container inspect, the container might be gone. Always have a fallback (check for Redis container by label).

3. **Using shell commands to pass passwords**: Never construct shell command strings with passwords. Use Docker API's structured command arrays to avoid shell escaping issues.

4. **Forgetting environment variable expansion for custom agent env vars**: The expansion must happen globally for the entire config, not just `services.redis` fields.

5. **Breaking existing tests**: When adding new config fields, ensure nil/empty checks don't break existing code paths that expect these fields to be absent.

6. **Incomplete rollback on failure**: If external Redis connection fails during `holt up`, ensure network and any partially-created containers are cleaned up.

7. **Race conditions in E2E tests**: When manually starting external Redis in tests, add a health check loop before running `holt up` to ensure Redis is ready.

8. **Not testing URI edge cases**: Test URIs with database numbers, IPv6 addresses, special characters in passwords, very long URIs.

**New metrics/logs to track:**

1. **Redis mode indicator**: Log during `holt up` which mode is active (external or managed)
   - Log line: `INFO: Using external Redis: redis://***@host:port` OR `INFO: Starting Holt-managed Redis (authentication: enabled/disabled)`
   - Purpose: Operators can confirm configuration is applied as expected

2. **Connection validation**: Log success/failure of initial Redis connection during `holt up`
   - Log line: `INFO: Successfully connected to Redis` OR `ERROR: Failed to connect to Redis: <sanitized error>`
   - Purpose: Immediate feedback if external Redis is unreachable

3. **Mode detection during `holt down`**: Log which mode was detected
   - Log line: `INFO: Detected external Redis mode, skipping cleanup` OR `INFO: Detected managed Redis mode, removing container`
   - Purpose: Transparency in cleanup behavior

**Health check implications:**
- No changes to health check endpoints or behavior
- Orchestrator and agent health checks continue to verify Redis connectivity
- Health check failures now might indicate external Redis issues (operational difference)

**Troubleshooting guidance:**
- If `holt up` fails with "connection refused", check:
  1. Is external Redis running? (`docker ps` or cloud provider console)
  2. Is the URI correct? (hostname, port, scheme)
  3. Can the host reach Redis? (`telnet host port` or `nc -zv host port`)
  4. Is authentication required? (try `redis-cli -u <uri> PING`)

### **9.2. Rollback and disaster recovery**

**Feature disable mechanism:**
- To revert to managed Redis: Remove `services.redis.uri` from `holt.yml`, run `holt down`, then `holt up`
- No complex rollback procedure needed (configuration-driven behavior)

**Rollback scenarios:**

1. **External Redis connection issues after deployment**:
   - **Procedure**: Update `holt.yml` to use managed mode (remove `uri` field), run `holt down --name <instance>`, run `holt up --name <instance>`
   - **Data loss**: Any artefacts/claims on external Redis persist and can be reconnected later
   - **Time to rollback**: ~30 seconds (stop + restart)

2. **Password authentication issues with managed Redis**:
   - **Procedure**: Remove `password` field from `holt.yml`, run `holt down`, run `holt up`
   - **Data loss**: Managed Redis is ephemeral (current design), so all data lost on restart
   - **Time to rollback**: ~30 seconds

3. **Config expansion errors (missing env vars)**:
   - **Procedure**: Set required environment variables or remove expansion syntax from config
   - **Impact**: `holt up` fails before creating any resources (safe failure)
   - **Time to fix**: Immediate (no cleanup needed)

**Disaster recovery:**
- **External Redis failure**: Restore from cloud provider backups, update URI in `holt.yml` if endpoint changed
- **Managed Redis failure**: Container is ephemeral (no persistent data in current design), restart with `holt down` + `holt up`

**Data migration requirements:**
- No data migration needed for rollback (configuration change only)
- If switching from managed to external: manual Redis data export/import required (use `redis-cli --rdb` or `BGSAVE`)
