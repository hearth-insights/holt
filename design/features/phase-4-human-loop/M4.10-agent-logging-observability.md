***Template purpose:*** *This document specifies the "FD 3 Return" architecture where agents can be noisy (stdout/stderr for logs) and return clean JSON via FD 3, implements the missing `holt logs` CLI command, and adds worker container retention for debugging.*

## **1. The 'why': goal and success criteria**

### **1.1. Goal statement**

Implement the "FD 3 Return" architecture where agents write their final JSON result to FD 3, allowing stdout/stderr to be used freely for logging and tool output, and provide seamless debugging via `holt logs` and optional worker container retention.

### **1.2. User story**

**As an agent developer**, I want to run noisy tools (like `npm install`, `git fetch`, `echo "debug"`) and see their output in logs, WITHOUT worrying about corrupting my JSON response. I just want to write my result to `>&3` and be done.

**As a Holt operator**, I want to run `holt logs <agent-role>` to instantly see ALL agent output (tool progress, debug prints, errors) without manually finding Docker container IDs.

**As a debugging developer**, I want to keep worker containers around after they exit so I can inspect their state, logs, and filesystem.

### **1.3. Success criteria**

1. **FD 3 Return Architecture**: Agent scripts are simple - stdout/stderr for logs (anything!), FD 3 for the final JSON result.
   ```bash
   #!/bin/sh
   # Read input
   input=$(cat)

   # Be noisy! All this goes to docker logs
   echo "Starting work..."
   npm install
   git fetch --all
   echo "Processing complete"

   # Return result via FD 3
   cat <<EOF >&3
   {
     "artefact_type": "CodeCommit",
     "artefact_payload": "abc123",
     "summary": "Work complete"
   }
   EOF
   ```

2. **FD 3 Validation**: Pup detects if agent exits without writing valid JSON to FD 3, creates Failure artefact with helpful error message.

3. **`holt logs` Command Works**:
   ```bash
   holt logs coder-agent          # Streams container logs via Docker SDK
   holt logs orchestrator         # Streams orchestrator logs
   holt logs -f coder-agent       # Follows logs (tail -f style)
   holt logs --since=1h coder     # Historical logs from last hour
   ```

4. **Worker Container Retention**: Developers can configure `keep_containers: true` in holt.yml to retain worker containers for post-mortem debugging.

5. **Helper Libraries**: Shell helper `holt_return` function writes JSON to FD 3.

6. **All Agents Migrated**: Every agent in `agents/` directory uses FD 3 Return model.

### **1.4. Non-goals**

- **NOT storing logs in Redis** - logs stay in Docker, period
- **NOT implementing structured event logging to Redis** - no Redis Pub/Sub for logs
- **NOT implementing log aggregation** - Docker handles this natively
- **NOT creating a custom logging pipeline** - YAGNI, Docker is sufficient

### **2.3. Agent pup changes**

#### **Critical architectural change: FD 3 Return Model**

**Channel redefinition:**

| FD | Channel | Purpose | Pup Handling |
|----|---------|---------|--------------|
| 0 | stdin | JSON input from pup → agent | Write once, close immediately |
| 1 | stdout | **Logging, tool output, debug prints** | **Stream in real-time to container stdout** |
| 2 | stderr | **Logging, errors, stack traces** | **Stream in real-time to container stderr** |
| 3 | **NEW** | **Final JSON result (ONLY)** | Read completely, parse, validate |

**Why this is brilliant:**

1. **Agents can be noisy**: stdout/stderr are for humans, not parsers
2. **No stdout pollution**: Final JSON goes to FD 3, completely separate
3. **Simpler agent code**: No need to silence tools or redirect output
4. **Better debugging**: All output visible in `docker logs` automatically

**Example agent script (NEW):**
```bash
#!/bin/sh
# Agent using FD 3 Return model

# Read input
input=$(cat)

# BE NOISY! This is encouraged now.
echo "Starting Maven build..."
mvn clean install  # All output goes to docker logs
echo "Build complete!"

# Return result via FD 3
cat <<EOF >&3
{
  "artefact_type": "BuildArtifact",
  "artefact_payload": "v1.0.0",
  "summary": "Maven build succeeded"
}
EOF
```

**Example agent script (OLD - for comparison):**
```bash
#!/bin/sh
# OLD way - had to silence everything

input=$(cat)

# Had to silence to avoid stdout pollution
mvn clean install > /dev/null 2>&1  # ❌ Can't see what happened!

# JSON to stdout
echo '{"artefact_type":"BuildArtifact","artefact_payload":"v1.0.0","summary":"Done"}'
```

#### **Implementation changes:**

**New:** Create FD 3 pipe, stream stdout/stderr through

```go
// executeToolSubprocess modifications
func (e *Engine) executeToolSubprocess(ctx context.Context, inputJSON string) (int, string, string, error) {
    // Validate /workspace directory exists
    if _, err := os.Stat("/workspace"); os.IsNotExist(err) {
        return -1, "", "", fmt.Errorf("/workspace directory does not exist")
    }

    // Create context with timeout
    execCtx, cancel := context.WithTimeout(ctx, toolExecutionTimeout)
    defer cancel()

    // M4.10: Create pipe for FD 3 (result JSON)
    fd3Reader, fd3Writer, err := os.Pipe()
    if err != nil {
        return -1, "", "", fmt.Errorf("failed to create FD 3 pipe: %w", err)
    }
    defer fd3Reader.Close()

    // Create command
    cmd := exec.CommandContext(execCtx, e.config.Command[0], e.config.Command[1:]...)
    cmd.Dir = "/workspace"

    // Create stdin pipe
    stdinPipe, err := cmd.StdinPipe()
    if err != nil {
        fd3Writer.Close()
        return -1, "", "", fmt.Errorf("failed to create stdin pipe: %w", err)
    }

    // M4.10: STREAM stdout/stderr to container (for docker logs)
    // This is a major change - we no longer buffer these!
    cmd.Stdout = os.Stdout  // Pass through to container stdout
    cmd.Stderr = os.Stderr  // Pass through to container stderr

    // M4.10: Attach FD 3 for result JSON
    cmd.ExtraFiles = []*os.File{fd3Writer}  // FD 3
    log.Printf("[DEBUG] M4.10: Attached FD 3 for result JSON")

    // Start process
    if err := cmd.Start(); err != nil {
        fd3Writer.Close()
        return -1, "", "", fmt.Errorf("failed to start process: %w", err)
    }

    // Write input JSON to stdin and close
    go func() {
        defer stdinPipe.Close()
        if _, err := io.WriteString(stdinPipe, inputJSON); err != nil {
            log.Printf("[WARN] Failed to write to stdin: %v", err)
        }
    }()

    // M4.10: Read result JSON from FD 3
    // Close write end so we get EOF when agent closes it
    fd3Writer.Close()

    resultBuf := &bytes.Buffer{}
    resultReader := &limitedReader{r: fd3Reader, limit: maxOutputSize}

    // Read FD 3 in background (agent may write incrementally)
    resultDone := make(chan error, 1)
    go func() {
        _, err := io.Copy(resultBuf, resultReader)
        resultDone <- err
    }()

    // Wait for process to complete
    processErr := cmd.Wait()

    // Wait for FD 3 read to complete
    if readErr := <-resultDone; readErr != nil {
        log.Printf("[WARN] Error reading FD 3: %v", readErr)
    }

    resultJSON := resultBuf.String()

    // Get exit code
    exitCode := 0
    if processErr != nil {
        if exitErr, ok := processErr.(*exec.ExitError); ok {
            exitCode = exitErr.ExitCode()
        } else {
            // Process couldn't be started or context timeout
            if execCtx.Err() == context.DeadlineExceeded {
                return -1, "", "", fmt.Errorf("tool execution timeout (5 minutes)")
            }
            return -1, "", "", processErr
        }
    }

    // M4.10: Validate FD 3 output
    if len(resultJSON) == 0 {
        return exitCode, "", "", fmt.Errorf("agent did not write result to FD 3. "+
            "HINT: Use 'cat <<EOF >&3' to return JSON. See docs/AGENT_LOGGING_GUIDE.md")
    }

    // Check for output size limit
    if resultBuf.Len() >= maxOutputSize {
        return exitCode, "", "", fmt.Errorf("result JSON exceeded 10MB limit")
    }

    // Non-zero exit code is an error
    if exitCode != 0 {
        return exitCode, resultJSON, "", fmt.Errorf("process exited with code %d", exitCode)
    }

    // Return: exitCode, FD3 result, empty stderr (now unused), error
    // Note: We no longer return stderr separately since it's streamed
    return exitCode, resultJSON, "", nil
}
```

**Note:** The function signature for `executeToolSubprocess` changes:
- **Before:** Returns `(exitCode, stdout, stderr, error)`
- **After:** Returns `(exitCode, fd3Result, "", error)` - stderr string is always empty (streamed live)

**Enhanced:** FD 3 result validation

```go
func (e *Engine) parseToolOutput(fd3Result string) (*ToolOutput, error) {
    if len(fd3Result) == 0 {
        return nil, fmt.Errorf("agent produced no output on FD 3. "+
            "HINT: Write JSON result using 'cat <<EOF >&3'. See docs/AGENT_LOGGING_GUIDE.md")
    }

    // M4.10: Validate it's JSON (could be multiline, that's OK now)
    trimmed := strings.TrimSpace(fd3Result)
    if !strings.HasPrefix(trimmed, "{") && !strings.HasPrefix(trimmed, "[") {
        return nil, fmt.Errorf("FD 3 output does not start with JSON. "+
            "Found: %q. HINT: Use 'cat <<EOF >&3' for JSON. See docs/AGENT_LOGGING_GUIDE.md",
            truncate(trimmed, 50))
    }

    var output ToolOutput
    if err := json.Unmarshal([]byte(trimmed), &output); err != nil {
        return nil, fmt.Errorf("invalid JSON on FD 3: %w. "+
            "HINT: Verify JSON syntax. See docs/AGENT_LOGGING_GUIDE.md", err)
    }

    if err := output.Validate(); err != nil {
        return nil, fmt.Errorf("validation failed: %w", err)
    }

    return &output, nil
}
```

**New helper type:**
```go
// limitedReader limits bytes read from a reader
type limitedReader struct {
    r     io.Reader
    limit int
    read  int
}

func (lr *limitedReader) Read(p []byte) (n int, err error) {
    if lr.read >= lr.limit {
        return 0, fmt.Errorf("read limit exceeded (%d bytes)", lr.limit)
    }

    maxRead := lr.limit - lr.read
    if len(p) > maxRead {
        p = p[:maxRead]
    }

    n, err = lr.r.Read(p)
    lr.read += n
    return n, err
}
```

### **2.4. CLI changes**

#### **New command: `holt logs`**

**New file:** `cmd/holt/commands/logs.go`

**Command specification:**

```bash
holt logs <role-or-orchestrator> [flags]

Flags:
  --name, -n <instance>   # Target instance (auto-inferred if omitted)
  --follow, -f            # Follow logs (like tail -f)
  --since <duration>      # Show logs since time (e.g., 1h, 30m)
  --tail <n>              # Show last N lines (default: all)
  --timestamps            # Include timestamps in output
```

**Examples:**
```bash

# Follow logs in real-time
holt logs -f coder-agent

# Show last 100 lines
holt logs --tail=100 coder-agent

# Show logs from last hour
holt logs --since=1h orchestrator

# Combine flags
holt logs -f --since=30m --timestamps coder-agent
```

**Implementation approach:**
1. Resolve instance name (auto-infer or use `--name` flag)
2. Translate role to container name:
   - Agent role → `holt-{instance}-{role}` (controller container)
   - "orchestrator" → `holt-{instance}-orchestrator`
3. Use Docker SDK to stream logs
4. Handle missing containers gracefully with helpful error

**Dependencies:**
- Uses existing `internal/docker` package for Docker SDK operations
- Uses existing `internal/instance` package for instance resolution
- Uses existing `internal/printer` package for formatted output

#### **Updated command: `holt up`**

**Warning for keep_containers:**

If any agent has `keep_containers: true`, print warning:

```
⚠️  WARNING: Worker container retention enabled for: coder-agent
   Stopped containers will NOT be automatically removed.
   Use 'holt down' to clean up all containers.

   This is intended for debugging only. Disable in production.
```

#### **Updated command: `holt down`**

**Aggressive container cleanup:**

Must find and remove ALL containers for the instance, including:
- Running containers
- Stopped containers (from `keep_containers: true`)
- Orphaned worker containers

**Implementation:**
```bash
# List all containers with instance label or name pattern
docker ps -a --filter "label=holt.instance={instance}" --format "{{.ID}}"
docker ps -a --filter "name=holt-{instance}-" --format "{{.ID}}"

# Remove all (stopped + running)
docker rm -f <container-ids>
```

## **3. The 'how': implementation & testing plan**

### **3.1. Key design decisions & risks**

**Decision 1: FD 3 for result (not logs)**

**Rationale:**
- Solves the "stdout pollution" problem permanently
- Agent scripts can be noisy without worrying about corruption
- Simpler mental model: FD 1/2 = logs (human), FD 3 = result (machine)
- Pup streams stdout/stderr → appears in `docker logs` automatically

**Risk:** Breaking change for existing agents. **Mitigation:** Migration script updates all agents automatically.

**Decision 2: Logs stay in Docker (no Redis)**

**Rationale:**
- YAGNI principle - Docker logs are battle-tested
- No custom log aggregation pipeline needed
- Consistent with Holt's philosophy: "wrap existing tools, don't reimplement"
- `holt logs` is a thin wrapper around Docker SDK

**Risk:** Can't query logs without Docker access. **Mitigation:** This is acceptable - operators have Docker access.

**Decision 3: Worker container retention is opt-in**

**Rationale:**
- Debugging feature, not for production
- Default `false` prevents container accumulation
- Explicit warning when enabled
- `holt down` cleanup is mandatory

**Risk:** Users forget to clean up. **Mitigation:** Warning on `holt up`, aggressive cleanup in `holt down`.

### **3.3. Performance & resource considerations**

#### **Resource usage:**

- **FD 3 pipe:** Minimal overhead - single pipe for result JSON
- **Stdout/stderr streaming:** Zero buffering overhead (direct pass-through)
- **`holt logs` streaming:** Minimal - Docker SDK streams efficiently
- **Retained containers:** Disk space proportional to container count

#### **Scalability limits:**

- **Docker log size:** Default 10MB per container, rotated automatically
- **FD 3 result size:** 10MB hard limit (enforced by limitedReader)
- **Concurrent `holt logs`:** No limit - each is independent Docker SDK client
- **Retained containers:** Limited by disk space (user responsibility)

#### **Performance requirements:**

- **Stdout/stderr streaming:** <1ms latency (direct pass-through)
- **FD 3 read:** Completes when agent closes FD 3 (on exit)
- **`holt logs` startup:** <100ms to first log line

## **4. Principle compliance check**

### **4.1. YAGNI (You Ain't Gonna Need It)**

**No new dependencies:**
- ✅ Uses existing Docker SDK
- ✅ No Redis logging infrastructure
- ✅ No custom log aggregation

**No custom logging pipeline:**
- ✅ Wraps Docker logs (battle-tested)
- ✅ FD 3 is standard Unix IPC
- ✅ No log storage/rotation (Docker handles it)

### **4.2. Auditability**

**All logs captured:**
- ✅ Docker logs retain full stdout/stderr for all containers
- ✅ `holt logs` provides access to historical logs via `--since`
- ✅ Logs persist after container stops (Docker retention policy)

**Result JSON separate:**
- ✅ FD 3 result is captured separately from logs
- ✅ Pup validates FD 3 result before creating artefact
- ✅ Immutable artefacts on blackboard maintain audit trail

### **4.3. Small, single-purpose components**

**Clear separation:**
- ✅ Pup: Validates FD 3 result, streams stdout/stderr, doesn't store logs
- ✅ CLI: Wraps Docker logs, doesn't implement custom logging
- ✅ Docker: Handles log storage, rotation, streaming

**No tight coupling:**
- ✅ `holt logs` is independent from pup, orchestrator, blackboard
- ✅ Logging is declarative (FD 3 Return model), not imperative

### **4.4. Security considerations**

**No new attack surfaces:**
- ✅ Logs don't contain secrets (agent responsibility)
- ✅ No network communication (local Docker socket only)
- ✅ No new authentication/authorization

**Sensitive data handling:**
- ⚠️ **Risk:** Agents might log API keys, tokens
- ✅ **Mitigation:** Document best practices - don't log secrets
- ✅ **Docker:** Log rotation limits exposure window

**Container retention risk:**
- ⚠️ **Risk:** Retained containers could contain sensitive data
- ✅ **Mitigation:** Opt-in feature with warning, disabled by default
- ✅ **Cleanup:** `holt down` removes all containers

### **4.5. Backward compatibility**

**BREAKING CHANGE:**
- ⚠️ **Breaking:** All agents must be updated to write result to FD 3
- ✅ **Mitigation:** Migration script updates all agents in one pass
- ✅ **Benefit:** Simpler, more robust architecture going forward

**Graceful error handling:**
- ✅ If agent doesn't write to FD 3, creates Failure artefact with helpful error
- ✅ Error message includes migration guide link

### **4.6. Dependency impact**

**No new dependencies:**
- ✅ Docker SDK already in use
- ✅ No new Go packages required
- ✅ No new infrastructure

**Build/runtime requirements:**
- ✅ No changes to Go version or Docker requirements

## **6. Error scenarios & edge cases**

### **6.1. Failure modes**

| Failure Mode | Behavior | Recovery |
|--------------|----------|----------|
| **Agent doesn't write to FD 3** | Pup detects, creates Failure artefact | Developer adds `>&3` to result |
| **Agent writes invalid JSON to FD 3** | Pup detects, creates Failure artefact | Developer fixes JSON syntax |
| **FD 3 pipe creation fails** | Pup returns error, no claim processing | Investigate system limits |
| **Container not found** | `holt logs` prints error with suggestion | User verifies instance/agent name |
| **Docker daemon down** | `holt logs` fails with connection error | User starts Docker |
| **Retained containers accumulate** | Disk space warning | User runs `holt down` to cleanup |

### **6.2. Concurrency considerations**

**Multiple `holt logs` invocations:**
- ✅ Each creates independent Docker SDK client
- ✅ No shared state, no race conditions
- ✅ Docker handles concurrent log access

**FD 3 read goroutine:**
- ✅ Reads from pipe until agent closes FD 3 (on exit)
- ✅ Uses limited reader to prevent unbounded memory growth
- ✅ Terminates when pipe closed

**Stdout/stderr streaming:**
- ✅ Direct pass-through to container stdout/stderr
- ✅ No buffering, no goroutines needed
- ✅ No risk of goroutine leak

**Worker container retention:**
- ⚠️ **Race:** `holt down` during worker execution
- ✅ **Mitigation:** `docker rm -f` handles running containers

### **6.3. Edge case handling**

| Edge Case | Handling |
|-----------|----------|
| **Very large FD 3 result** | limitedReader enforces 10MB limit, returns error |
| **Agent writes to FD 3 incrementally** | Pup reads until agent closes FD 3 |
| **Agent never closes FD 3** | Pup reads until agent exits (timeout), then checks result |
| **Binary output on FD 3** | JSON parse fails, Failure artefact created |
| **Unicode in logs** | Supported - Docker preserves encoding |
| **Logs from stopped container** | Supported - Docker retains until container removed |
| **Worker container name collision** | UUID in container name prevents collisions |

## **8. Migration plan**

### **8.1. Migration prompt for coding agent**

> **Migration Task: Switch to FD 3 Return Protocol**
>
> We are switching from the stdout-based result model to the FD 3 Return protocol. This is a breaking change that must be applied atomically across the entire codebase.
>
> **Perform the following refactoring:**
>
> **1. Update Pup (`internal/pup/executor.go`):**
>
> - Modify `executeToolSubprocess` to create an `os.Pipe()`.
> - Pass the write end as `cmd.ExtraFiles = []*os.File{writePipe}` (FD 3).
> - Change `cmd.Stdout` and `cmd.Stderr` to stream directly to `os.Stdout` and `os.Stderr` (pass-through).
> - Read the result JSON from the read end of the FD 3 pipe.
> - Update function signature: return `(exitCode, fd3Result, "", error)` where the third string is always empty.
> - Add `limitedReader` helper type to enforce 10MB limit on FD 3.
> - Update `parseToolOutput` to `parseFD3Output` and adjust validation logic.
> - Update `executeWork` to pass `fd3Result` instead of `stdout`.
>
> **2. Update All Agents (`agents/*/run.sh`, `agents/*/*.sh`):**
>
> - Scan all shell script files in the `agents/` directory recursively.
> - Identify where the final JSON is output (usually `cat <<EOF`, `echo '{'`, or similar).
> - Redirect this output to FD 3 by adding `>&3` (e.g., `cat <<EOF >&3`).
> - Remove any "silencing" logic that was previously added to prevent stdout pollution:
>   - Remove `> /dev/null 2>&1`
>   - Remove `2>&1 | grep -v "verbose output"`
>   - Remove any other output suppression
> - Verbose logs are now **encouraged**! Agents can be noisy.
>
> **3. Update Helper Scripts:**
>
> - Create `agents/lib/log.sh` if it doesn't exist.
> - Add a helper function `holt_return()` that accepts JSON and writes it to FD 3:
>   ```bash
>   holt_return() {
>       cat <<EOF >&3
>   $1
>   EOF
>   }
>   ```
> - Create `agents/lib/log.py` with Python helper:
>   ```python
>   import os
>   import json
>
>   def holt_return(data):
>       """Write result JSON to FD 3"""
>       try:
>           fd3 = os.fdopen(3, 'w')
>           json.dump(data, fd3)
>           fd3.write('\n')
>           fd3.close()
>       except Exception as e:
>           # FD 3 not available (running outside pup)
>           print(json.dumps(data))  # Fallback to stdout
>   ```
>
> **4. Update Tests:**
>
> - Update `internal/pup/executor_test.go` to verify FD 3 reading.
> - Add test: Agent writes to FD 3 → pup reads result.
> - Add test: Agent doesn't write to FD 3 → error with helpful message.
> - Add test: Agent writes to stdout/stderr → not parsed, appears in logs.
>
