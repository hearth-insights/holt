# **Feature design: M5.1 - Declarative Fan-In Synchronization**

**Purpose**: Enable agents to declaratively wait for multiple different prerequisite artefacts from parallel workflow branches before executing, using atomic indexing and crash-resistant deduplication.

**Scope**: Phase 5 Complex Coordination - Milestone 1
**Estimated tokens**: ~12,000 tokens
**Read when**: Implementing fan-in synchronization, DAG orchestration, scatter-gather patterns

Associated phase: **Complex Coordination (Phase 5)**
Status: **Draft**

***Template purpose:*** This document is a blueprint for M5.1, an implementable milestone that enables declarative fan-in synchronization for DAG-based workflows. It provides an unambiguous specification for implementing the "synchronizer" pattern with atomic indexing, producer-declared counts, and crash-resistant deduplication.

---

## **1. The 'why': goal and success criteria**

### **1.1. Goal statement**

Enable declarative configuration of "Synchronizer Agents" that wait for multiple distinct prerequisite artefacts (from different workflow branches) to complete before executing, using atomic indexing for efficient graph traversal and lock-based deduplication to prevent race conditions.

### **1.2. User story**

**As an agent developer**, when I need to build a workflow that merges multiple parallel branches (e.g., a deployer that waits for tests, linting, and security scans to complete), I want to declaratively specify the synchronization requirements in `holt.yml` without writing complex, race-condition-prone bidding logic.

**As a system architect**, I want the fan-in pattern to be:
- **Declarative**: Specified in configuration, not agent code
- **Atomic**: Artefact creation and reverse index updates happen together
- **Safe**: Deduplication prevents double-execution when multiple prerequisites complete simultaneously
- **Efficient**: Reverse indexes enable fast descendant traversal without full graph scans
- **Observable**: Synchronization events are logged and auditable

**Core Insight**: Fan-in is a fundamental DAG coordination primitive. It must be a first-class feature of Holt, implemented with distributed systems safety guarantees (atomicity, idempotency, deduplication), not left to agent developers to implement incorrectly.

### **1.3. Success criteria**

1. **Named Pattern Works**: A Synchronizer agent configured with:
   ```yaml
   synchronize:
     ancestor_type: "CodeCommit"
     wait_for:
       - type: "TestResult"
       - type: "LintResult"
       - type: "SecurityScan"
   ```
   Only bids when all three distinct artefact types exist as descendants of a single CodeCommit ancestor. When granted, receives ancestor + descendants in context.

2. **Producer-Declared Pattern Works**: A Synchronizer configured with:
   ```yaml
   synchronize:
     ancestor_type: "DataBatch"
     wait_for:
       - type: "ProcessedRecord"
         count_from_metadata: "batch_size"
   ```
   Waits until N ProcessedRecord artefacts exist, where N is read from the DataBatch's metadata field. Works correctly when N is unknown at configuration time.

3. **Atomic Indexing**: All artefact creation (Pup, CLI, Orchestrator) uses a Redis Lua script that atomically:
   - Creates the artefact Hash
   - Updates the reverse index Set (`holt:{inst}:index:children:{parent_id}`)
   - Publishes the artefact event

   The reverse index is always consistent with artefact state.

4. **Deduplication Prevents Double-Execution**: When the final two prerequisites complete simultaneously on different worker containers, only ONE Synchronizer bid is submitted. The second acquires a Redis lock (`SET NX`) and finds it already taken, so it skips bidding.

5. **Multi-Artefact Output**: An agent can produce multiple artefacts in one execution:
   ```json
   {"artefact_type": "TestResult", "artefact_payload": "test1.json"}
   {"artefact_type": "TestResult", "artefact_payload": "test2.json"}
   ```
   The Pup buffers all outputs, counts them, injects `{"batch_size": "2"}` metadata into each, and creates all artefacts atomically via Lua.

6. **Recursive Descendant Traversal**: Synchronizers find descendants at any depth below the ancestor (children, grandchildren, etc.), with optional `max_depth` limit to prevent runaway recursion.

7. **Full Context Assembly**: Synchronizer agents receive the full historical context chain, plus the ancestor artefact and all matched descendants, enabling complex aggregation logic.

8. **Backward Compatibility**: Existing agents without `synchronize` blocks continue working unchanged. Non-Synchronizer workflows are unaffected by the new Lua script and reverse index.

### **1.4. Non-goals**

- **Timeout-based synchronization** (Future): Synchronizers wait indefinitely. If a prerequisite fails, the workflow hangs. No timeout or failure-state detection in V1.
- **Sharded pattern with fixed count** (Removed): Originally proposed `count: 5` pattern removed due to ambiguity (what if 6 arrive?). Only Named and Producer-Declared patterns supported.
- **Cross-ancestor synchronization** (Future): Synchronizers wait for descendants of a single ancestor, not multiple ancestors.
- **Dynamic wait_for modification** (Future): Synchronization conditions are static (from holt.yml), not runtime-configurable.
- **Orphaned lock cleanup** (Future): If a Pup crashes after acquiring the deduplication lock but before bidding, the workflow deadlocks until TTL expiry (10 minutes). Manual intervention required.
- **Partial failure handling** (Future): If 4 of 5 shards complete and 1 fails, the Synchronizer hangs. No automatic failure propagation in V1.

#### **2.1.2. New Redis key patterns**

**Reverse index (Set)**:
```
holt:{inst}:index:children:{parent_artefact_id}
```
- **Type**: Redis Set
- **Members**: Child artefact IDs (UUIDs)
- **Purpose**: Fast "downward" traversal from ancestor to descendants
- **Lifecycle**: Created atomically with artefact via Lua script
- **Example**:
  ```
  SMEMBERS holt:default-1:index:children:commit-uuid-123
  # Returns: ["test-uuid-1", "lint-uuid-2", "scan-uuid-3"]
  ```

**Deduplication lock (String with TTL)**:
```
holt:{inst}:sync_dedup:{ancestor_id}:{agent_role_hash}
```
- **Type**: Redis String
- **Value**: "1" (presence indicates lock is held)
- **TTL**: 10 minutes (600 seconds)
- **Purpose**: Prevent multiple simultaneous bids from same agent on same ancestor
- **Lifecycle**: Set via `SET NX EX 600`, never explicitly released
- **Example**:
  ```
  SET holt:default-1:sync_dedup:commit-abc:deployer-hash 1 NX EX 600
  # Returns: OK (lock acquired) or nil (lock already held)
  ```

#### **2.1.3. Lua script for atomic artefact creation**

**File**: `pkg/blackboard/lua/create_artefact.lua` (new file)

```lua
-- M5.1: Atomic artefact creation with reverse index update
-- KEYS[1]: artefact hash key (holt:{inst}:artefact:{uuid})
-- KEYS[2]: thread sorted set key (holt:{inst}:thread:{logical_id})
-- KEYS[3]: artefact events channel (holt:{inst}:artefact_events)
-- ARGV[1]: artefact ID (UUID)
-- ARGV[2]: logical ID (UUID)
-- ARGV[3]: version (integer)
-- ARGV[4]: structural_type (string)
-- ARGV[5]: type (string)
-- ARGV[6]: payload (string)
-- ARGV[7]: source_artefacts (JSON array string)
-- ARGV[8]: produced_by_role (string)
-- ARGV[9]: metadata (JSON object string)

local artefact_key = KEYS[1]
local thread_key = KEYS[2]
local events_channel = KEYS[3]

local artefact_id = ARGV[1]
local logical_id = ARGV[2]
local version = tonumber(ARGV[3])
local structural_type = ARGV[4]
local type = ARGV[5]
local payload = ARGV[6]
local source_artefacts_json = ARGV[7]
local produced_by_role = ARGV[8]
local metadata = ARGV[9]

-- Step 1: Create artefact hash
redis.call('HSET', artefact_key,
    'id', artefact_id,
    'logical_id', logical_id,
    'version', version,
    'structural_type', structural_type,
    'type', type,
    'payload', payload,
    'source_artefacts', source_artefacts_json,
    'produced_by_role', produced_by_role,
    'metadata', metadata
)

-- Step 2: Update thread tracking
redis.call('ZADD', thread_key, version, artefact_id)

-- Step 3: Update reverse index for each parent
-- Parse source_artefacts JSON array
local source_artefacts = cjson.decode(source_artefacts_json)
for _, parent_id in ipairs(source_artefacts) do
    local children_key = 'holt:' .. string.match(artefact_key, 'holt:([^:]+):') .. ':index:children:' .. parent_id
    redis.call('SADD', children_key, artefact_id)
end

-- Step 4: Publish artefact event
redis.call('PUBLISH', events_channel, artefact_id)

return artefact_id
```

**Error handling**: If the Lua script fails (Redis error, OOM, etc.), the calling component (Pup, CLI, Orchestrator) MUST panic/exit immediately. Partial artefact creation is unacceptable.

func (c *Client) CreateArtefact(ctx context.Context, artefact *Artefact) error {
    artefactKey := fmt.Sprintf("holt:%s:artefact:%s", c.instanceName, artefact.ID)
    threadKey := fmt.Sprintf("holt:%s:thread:%s", c.instanceName, artefact.LogicalID)
    eventsChannel := fmt.Sprintf("holt:%s:artefact_events", c.instanceName)

    // Encode source_artefacts as JSON
    sourceArtefactsJSON, err := json.Marshal(artefact.SourceArtefacts)
    if err != nil {
        return fmt.Errorf("failed to marshal source_artefacts: %w", err)
    }

    // Ensure metadata is valid JSON (default to empty object)
    metadata := artefact.Metadata
    if metadata == "" {
        metadata = "{}"
    }

    // Load and execute Lua script
    script := redis.NewScript(createArtefactLuaScript)
    _, err = script.Run(ctx, c.client,
        []string{artefactKey, threadKey, eventsChannel},
        artefact.ID,

        artefact.StructuralType,
        artefact.Type,
        artefact.Payload,
        string(sourceArtefactsJSON),
        artefact.ProducedByRole,
        metadata,
    ).Result()

    if err != nil {
        // M5.1: Lua failure is fatal
        log.Fatalf("[Blackboard] FATAL: Lua script create_artefact failed: %v", err)
    }

    return nil
}

// M5.1: GetDescendants retrieves all descendants of an artefact (recursive)
func (c *Client) GetDescendants(ctx context.Context, ancestorID string, maxDepth int) ([]*Artefact, error) {
    visited := make(map[string]bool)
    var descendants []*Artefact

    var traverse func(string, int) error
    traverse = func(nodeID string, depth int) error {
        if visited[nodeID] {
            return nil // Already visited, skip
        }
        visited[nodeID] = true

        if maxDepth > 0 && depth > maxDepth {
            return nil // Max depth reached
        }

        // Get children from reverse index
        childrenKey := fmt.Sprintf("holt:%s:index:children:%s", c.instanceName, nodeID)
        childIDs, err := c.client.SMembers(ctx, childrenKey).Result()
        if err != nil {
            return fmt.Errorf("failed to get children of %s: %w", nodeID, err)
        }

        for _, childID := range childIDs {
            // Load child artefact
            child, err := c.GetArtefact(ctx, childID)
            if err != nil {
                return fmt.Errorf("failed to load child artefact %s: %w", childID, err)
            }

            descendants = append(descendants, child)

            // Recurse into grandchildren
            if err := traverse(childID, depth+1); err != nil {
                return err
            }
        }

        return nil
    }

    if err := traverse(ancestorID, 0); err != nil {
        return nil, err
    }

    return descendants, nil
}

// M5.1: AcquireSyncLock attempts to acquire deduplication lock
func (c *Client) AcquireSyncLock(ctx context.Context, ancestorID, agentRole string) (bool, error) {
    // Hash agent role for lock key
    roleHash := fmt.Sprintf("%x", sha256.Sum256([]byte(agentRole)))[:8]
    lockKey := fmt.Sprintf("holt:%s:sync_dedup:%s:%s", c.instanceName, ancestorID, roleHash)

    // SET NX EX 600 (10 minutes)
    result, err := c.client.SetNX(ctx, lockKey, "1", 10*time.Minute).Result()
    if err != nil {
        return false, fmt.Errorf("failed to acquire sync lock: %w", err)
    }

    return result, nil
}

// M5.1: parseToolOutputs parses multiple JSON objects from FD3
func (p *Pup) parseToolOutputs(data []byte) ([]ToolOutput, error) {
    var outputs []ToolOutput
    decoder := json.NewDecoder(bytes.NewReader(data))

    for {
        var output ToolOutput
        if err := decoder.Decode(&output); err == io.EOF {
            break
        } else if err != nil {
            return nil, fmt.Errorf("invalid JSON output: %w", err)
        }

        // Default structural_type to "Standard"
        if output.StructuralType == "" {
            output.StructuralType = blackboard.StructuralTypeStandard
        }

        outputs = append(outputs, output)
    }

    if len(outputs) == 0 {
        return nil, fmt.Errorf("no outputs produced by tool")
    }

    return outputs, nil
}

// M5.1: Synchronizer implements declarative fan-in coordination
type Synchronizer struct {
    config    *config.SynchronizeConfig
    bbClient  *blackboard.Client
    agentRole string
}

// shouldBidOnClaim checks if synchronization conditions are met for a claim
func (s *Synchronizer) shouldBidOnClaim(ctx context.Context, claim *blackboard.Claim) (bool, error) {
    log.Printf("[Synchronizer] Evaluating claim %s for synchronization", claim.ID)

    // Load target artefact
    targetArtefact, err := s.bbClient.GetArtefact(ctx, claim.ArtefactID)
    if err != nil {
        return false, fmt.Errorf("failed to load target artefact: %w", err)
    }

    // Step 1: Check if target artefact is a potential trigger
    if !s.isPotentialTrigger(targetArtefact) {
        log.Printf("[Synchronizer] Artefact %s (type=%s) is not a potential trigger, ignoring", targetArtefact.ID, targetArtefact.Type)
        return false, nil
    }

    // Step 2: Find common ancestor
    ancestor, err := s.findCommonAncestor(ctx, targetArtefact)
    if err != nil {
        log.Printf("[Synchronizer] Failed to find ancestor: %v", err)
        return false, nil // Not an error, just not ready
    }
    if ancestor == nil {
        log.Printf("[Synchronizer] No ancestor of type '%s' found for artefact %s", s.config.AncestorType, targetArtefact.ID)
        return false, nil
    }

    log.Printf("[Synchronizer] Found ancestor %s (type=%s)", ancestor.ID, ancestor.Type)

    // Step 3: Verify all dependencies (fan-in check)
    allReady, err := s.checkAllDependenciesMet(ctx, ancestor)
    if err != nil {
        return false, fmt.Errorf("failed to check dependencies: %w", err)
    }
    if !allReady {
        log.Printf("[Synchronizer] Not all dependencies met for ancestor %s", ancestor.ID)
        return false, nil
    }

    log.Printf("[Synchronizer] All dependencies met for ancestor %s", ancestor.ID)

    // Step 4: Acquire deduplication lock
    lockAcquired, err := s.bbClient.AcquireSyncLock(ctx, ancestor.ID, s.agentRole)
    if err != nil {
        return false, fmt.Errorf("failed to acquire sync lock: %w", err)
    }
    if !lockAcquired {
        log.Printf("[Synchronizer] Lock already held for ancestor %s, skipping bid (deduplication)", ancestor.ID)
        return false, nil
    }

    log.Printf("[Synchronizer] Lock acquired for ancestor %s, ready to bid", ancestor.ID)
    return true, nil
}

// isPotentialTrigger checks if target artefact type is in wait_for list
func (s *Synchronizer) isPotentialTrigger(artefact *blackboard.Artefact) bool {
    for _, condition := range s.config.WaitFor {
        if artefact.Type == condition.Type {
            return true
        }
    }
    return false
}

// findCommonAncestor traverses upward to find first ancestor matching ancestor_type
func (s *Synchronizer) findCommonAncestor(ctx context.Context, artefact *blackboard.Artefact) (*blackboard.Artefact, error) {
    visited := make(map[string]bool)
    queue := []string{artefact.ID}

    for len(queue) > 0 {
        currentID := queue[0]
        queue = queue[1:]

        if visited[currentID] {
            continue
        }
        visited[currentID] = true

        current, err := s.bbClient.GetArtefact(ctx, currentID)
        if err != nil {
            return nil, err
        }

        // Check if this is the ancestor we're looking for
        if current.Type == s.config.AncestorType {
            return current, nil
        }

        // Add parents to queue (traverse upward)
        queue = append(queue, current.SourceArtefacts...)
    }

    return nil, nil // Not found
}

// checkAllDependenciesMet verifies all wait_for conditions are satisfied
func (s *Synchronizer) checkAllDependenciesMet(ctx context.Context, ancestor *blackboard.Artefact) (bool, error) {
    // Get all descendants of ancestor
    descendants, err := s.bbClient.GetDescendants(ctx, ancestor.ID, s.config.MaxDepth)
    if err != nil {
        return false, fmt.Errorf("failed to get descendants: %w", err)
    }

    log.Printf("[Synchronizer] Found %d descendants of ancestor %s", len(descendants), ancestor.ID)

    // Group descendants by type
    descendantsByType := make(map[string][]*blackboard.Artefact)
    for _, desc := range descendants {
        descendantsByType[desc.Type] = append(descendantsByType[desc.Type], desc)
    }

    // Check each wait condition
    for _, condition := range s.config.WaitFor {
        artefactsOfType := descendantsByType[condition.Type]

        if condition.CountFromMetadata != "" {
            // Producer-Declared pattern: Read expected count from metadata
            if len(artefactsOfType) == 0 {
                log.Printf("[Synchronizer] No artefacts of type '%s' found yet", condition.Type)
                return false, nil
            }

            // Read batch_size from any artefact's metadata (all siblings have same value)
            expectedCount, err := s.getExpectedCountFromMetadata(artefactsOfType[0], condition.CountFromMetadata)
            if err != nil {
                log.Printf("[Synchronizer] Failed to read metadata '%s': %v", condition.CountFromMetadata, err)
                return false, nil // Not ready (metadata missing or invalid)
            }

            if len(artefactsOfType) < expectedCount {
                log.Printf("[Synchronizer] Type '%s': found %d of %d expected", condition.Type, len(artefactsOfType), expectedCount)
                return false, nil
            }

            log.Printf("[Synchronizer] Type '%s': all %d artefacts present", condition.Type, expectedCount)
        } else {
            // Named pattern: Exactly 1 required
            if len(artefactsOfType) == 0 {
                log.Printf("[Synchronizer] Type '%s' not found", condition.Type)
                return false, nil
            }

            log.Printf("[Synchronizer] Type '%s': present", condition.Type)
        }
    }

    return true, nil
}

// getExpectedCountFromMetadata parses metadata to extract expected count
func (s *Synchronizer) getExpectedCountFromMetadata(artefact *blackboard.Artefact, metadataKey string) (int, error) {
    var metadata map[string]string
    if err := json.Unmarshal([]byte(artefact.Metadata), &metadata); err != nil {
        return 0, fmt.Errorf("invalid metadata JSON: %w", err)
    }

    countStr, exists := metadata[metadataKey]
    if !exists {
        return 0, fmt.Errorf("metadata key '%s' not found", metadataKey)
    }

    count, err := strconv.Atoi(countStr)
    if err != nil {
        return 0, fmt.Errorf("metadata value '%s' is not a valid integer: %w", countStr, err)
    }

    if count <= 0 {
        return 0, fmt.Errorf("metadata value '%s' must be positive", countStr)
    }

    return count, nil
}

## **3. The 'how': implementation & testing plan**

### **3.1. Key design decisions & risks**

**Critical Design Decisions:**

1. **Universal Lua Script Adoption**:
   - **Decision**: ALL artefact creation (Pup, CLI, Orchestrator) must use the Lua script
   - **Rationale**: Ensures reverse index is always consistent with artefact state
   - **Trade-off**: Adds complexity to all components vs. optional adoption
   - **Safety**: Lua script failures are FATAL (panic/exit immediately)

2. **Buffer-and-Flush for Multi-Artefact Outputs**:
   - **Decision**: Pup reads all outputs from FD3 until process exits, then processes as batch
   - **Rationale**: Enables accurate batch_size counting and metadata injection
   - **Trade-off**: Delays artefact creation until process completes vs. streaming
   - **Memory impact**: Buffering large outputs could consume memory, but acceptable for V1

3. **Producer-Side Metadata Injection**:
   - **Decision**: Pup (not agent script) calculates and injects metadata
   - **Rationale**: Enforces correctness, prevents agent bugs from corrupting counts
   - **Trade-off**: Pup complexity increases vs. agent flexibility
   - **Safety**: Downstream Synchronizers can trust metadata values

4. **Lock-Based Deduplication with TTL**:
   - **Decision**: `SET NX EX 600` (10 minutes), never release lock
   - **Rationale**: Presence of key means "already bid," prevents race conditions
   - **Trade-off**: Deadlock risk if Pup crashes vs. complex lock management
   - **Documented Limitation**: Crash after lock acquisition = deadlock until TTL expiry

5. **Recursive Descendant Traversal**:
   - **Decision**: Find descendants at any depth, with optional max_depth limit
   - **Rationale**: Supports complex DAGs, prevents runaway recursion
   - **Trade-off**: Expensive for deep trees vs. shallow traversal
   - **Optimization**: Use reverse index (O(n) traversal, not O(n²) graph scan)

6. **Synchronizer Exclusivity**:
   - **Decision**: Agents with `synchronize` block cannot have `bidding_strategy` or `bid_script`
   - **Rationale**: Clear separation of concerns, simpler implementation
   - **Trade-off**: Less flexibility vs. implementation simplicity

7. **Removal of Sharded Pattern**:
   - **Decision**: Only Named and Producer-Declared patterns supported
   - **Rationale**: Fixed count ("wait for 5") is ambiguous (what if 6 arrive?)
   - **Impact**: Users must use Producer-Declared pattern for dynamic counts

**Risks:**

1. **Lua Script Failure (CRITICAL)**:
   - **Risk**: Redis OOM, syntax error, network partition during script execution
   - **Probability**: Low (Lua is stable, Redis is reliable)
   - **Impact**: CATASTROPHIC (artefact creation fails, workflow halts)
   - **Mitigation**: Panic/exit immediately, require manual intervention, log loudly

2. **Deadlock from Lock Orphaning**:
   - **Risk**: Pup crashes after acquiring lock but before bidding
   - **Probability**: Low-Medium (container crashes, OOM kills)
   - **Impact**: High (workflow stalls until 10-minute TTL expiry)
   - **Mitigation**: Document as known limitation, recommend monitoring, manual lock deletion

3. **Partial Fan-In Hang**:
   - **Risk**: 4 of 5 shards complete, 1 crashes, Synchronizer waits forever
   - **Probability**: Medium (worker failures are common)
   - **Impact**: High (workflow never completes)
   - **Mitigation**: Document as known limitation, recommend monitoring, manual intervention

4. **Reverse Index Bloat**:
   - **Risk**: Long-running instances accumulate large reverse index sets
   - **Probability**: High (no cleanup in V1)
   - **Impact**: Low (Redis memory usage increases, but sets are efficient)
   - **Mitigation**: Document as known limitation, recommend periodic instance recycling

5. **Deep Recursion in Descendant Traversal**:
   - **Risk**: Artefact graph has depth > 100, traversal times out or OOMs
   - **Probability**: Low (most workflows are shallow)
   - **Impact**: Medium (Synchronizer fails to bid)
   - **Mitigation**: Provide `max_depth` configuration to limit recursion

6. **Metadata Missing or Invalid**:
   - **Risk**: Producer Pup crashes before injecting metadata, or metadata is corrupted
   - **Probability**: Low (Pup injects metadata atomically via Lua)
   - **Impact**: High (Synchronizer cannot determine expected count, hangs)
   - **Mitigation**: Document as known limitation, Pup logs metadata injection

### **3.3. Performance & resource considerations**

**Resource usage:**

- **Metadata field**: ~100 bytes per artefact (JSON-encoded map)
- **Reverse index**: ~50 bytes per parent-child relationship (Set member)
- **Deduplication lock**: ~100 bytes per lock (key + value + TTL metadata)
- **Lua script overhead**: ~1ms per artefact creation (Redis blocks during execution)
- **Descendant traversal**: O(n) where n = total descendants (recursive BFS)

**Memory impact:**
- **Buffer-and-flush**: Holds all tool outputs in memory until process exits
  - Example: 100 artefacts × 1KB each = 100KB buffered
  - Risk: Large batches (1000+ artefacts) could consume significant memory
  - Mitigation: Document recommended batch sizes (< 1000 artefacts)

**Scalability limits:**

- **Reverse index size**: Unbounded growth (no cleanup in V1)
  - Example: 1M artefacts × 2 parents average = 2M set members
  - Redis memory: ~100MB for 2M set members (acceptable)
  - Recommendation: Periodic instance recycling for long-running workflows

- **Descendant traversal depth**: Limited by `max_depth` configuration
  - Default: Unlimited (0 = infinite recursion)
  - Recommended: `max_depth: 10` for most workflows
  - Risk: Depth > 100 could timeout or OOM

- **Concurrent Lua execution**: Redis processes Lua serially (single-threaded)
  - Impact: High artefact creation rate (> 1000/sec) could bottleneck
  - Mitigation: Redis Lua is extremely fast (~10μs per script), unlikely to bottleneck in V1

**Performance requirements:**

- **Artefact creation via Lua**: < 5ms (including network round-trip)
- **Descendant traversal**: < 100ms for typical workflows (< 100 descendants)
- **Lock acquisition**: < 10ms (single SET NX command)
- **Synchronizer bid evaluation**: < 200ms (includes traversal + dependency check)
- **No performance regression**: Non-synchronizer workflows maintain M4.x performance

## **4. Principle compliance check**

### **4.1. YAGNI (You Ain't Gonna Need It)**

**No new third-party dependencies** beyond existing Redis and Docker clients.

**New complexity justified:**
- **Lua script**: Mandatory for atomic indexing, prevents race conditions
- **Buffer-and-flush**: Required for accurate batch_size counting
- **Synchronization logic**: Core feature of Phase 5, not premature

**Deferred features (YAGNI for V1):**
- Timeout-based synchronization (wait indefinitely in V1)
- Orphaned lock cleanup (manual intervention in V1)
- Partial failure handling (hang indefinitely in V1)
- Dynamic wait_for modification (static config in V1)
- Reverse index garbage collection (periodic recycling in V1)

### **4.2. Auditability**

**New artefacts created**: None (synchronization is coordination-only, Synchronizer agents create artefacts via standard flow).

**Immutable audit trail enhanced:**
- **Reverse index**: Permanent record of parent-child relationships (never deleted)
- **Metadata field**: Batch size and other provenance data immutable after creation
- **Deduplication locks**: Logged when acquired (evidence of concurrent execution prevention)
- **Synchronizer decisions**: Logged at each evaluation step (ancestor found, dependencies met, lock acquired)

**State changes captured:**
- Artefact creation: Logged with metadata and reverse index update
- Lock acquisition: Logged with ancestor ID and agent role
- Descendant traversal: Logged with count and types found
- Synchronizer bid: Logged with decision reasoning

**Compliance value:**
- Full DAG provenance tracking (reverse index enables ancestor → descendant audits)
- Scatter-gather coordination visible (metadata shows batch sizes, dependencies)
- Race condition prevention auditable (lock acquisition events prove deduplication)

### **4.3. Small, single-purpose components**

**Component responsibilities remain clear:**
- **Blackboard**: Adds Lua script execution and reverse index storage (data layer responsibilities)
- **Pup**: Adds producer (buffer-and-flush) and consumer (synchronization) logic (agent coordination responsibilities)
- **CLI/Orchestrator**: Minimal changes (just Lua adoption, no new logic)

**No tight coupling introduced:**
- Lua script is encapsulated in blackboard client (callers don't know it's Lua)

### **4.4. Security considerations**

**No new attack surfaces:**
- Lua script runs server-side in Redis (no user input)
- Reverse index uses standard Redis Sets (no new data exposure)
- Deduplication lock uses SHA-256 hash (prevents key collision attacks)

**Data exposure:**
- Metadata field contains batch size (not sensitive)
- Reverse index reveals parent-child relationships (already visible in `source_artefacts`)
- Locks contain agent role hash (not reversible to original role name)

**Input validation:**
- Lua script parameters: All strings/integers from trusted sources (Pup, CLI, Orchestrator)
- Metadata JSON: Validated during parsing (invalid JSON → error, not execution)
- Lock key construction: Agent role is hashed (prevents injection attacks)

**Container isolation:**
- No changes to agent container privileges
- Synchronizers communicate via Redis (same as standard agents)
- No new network paths or ports

**Privilege boundaries:**
- Synchronizers cannot impersonate other agents (lock uses agent's own role)
- Metadata injection by Pup (agents cannot forge batch sizes)
- Lua script atomic (no partial updates that could be exploited)

## **6. Error scenarios & edge cases**

### **6.1. Failure modes**

**Lua script failures:**

1. **Redis unavailable during script execution**:
   - **Behavior**: Lua script execution fails, `CreateArtefact()` returns error
   - **Detection**: Redis client returns connection error
   - **Recovery**: Caller MUST panic/exit immediately (documented requirement)
   - **Logging**: "FATAL: Lua script create_artefact failed: connection refused"
   - **Impact**: Artefact NOT created, workflow halts
   - **Rationale**: Partial artefact creation is unacceptable (no reverse index update)

2. **Redis out of memory during script**:
   - **Behavior**: Lua script execution fails with OOM error
   - **Detection**: Redis returns OOM error
   - **Recovery**: Panic/exit immediately
   - **Logging**: "FATAL: Lua script create_artefact failed: OOM"
   - **Impact**: Workflow halts, requires Redis memory increase or instance cleanup

3. **Lua script syntax error**:
   - **Behavior**: Script fails to load (developer error)
   - **Detection**: Redis returns syntax error on first execution
   - **Recovery**: None (code bug, requires fix and redeploy)
   - **Testing**: Unit tests MUST catch syntax errors before deployment

**Synchronizer failures:**

4. **Deduplication lock acquisition failure (already held)**:
   - **Behavior**: `AcquireSyncLock()` returns false (lock exists)
   - **Detection**: SET NX returns 0
   - **Recovery**: Skip bid (log "deduplication")
   - **Logging**: "Lock already held for ancestor abc-123, skipping bid"
   - **Impact**: Prevents duplicate bid (CORRECT behavior)

5. **Pup crashes after acquiring lock but before bidding**:
   - **Behavior**: Lock remains in Redis, prevents other Pups from bidding
   - **Detection**: TTL expiry after 10 minutes
   - **Recovery**: Automatic (TTL expires), or manual (delete lock key)
   - **Logging**: No automatic detection (monitoring required)
   - **Impact**: Workflow deadlocks for 10 minutes
   - **Documented Limitation**: Known risk, acceptable for V1

6. **Partial fan-in (4 of 5 shards complete, 1 fails)**:
   - **Behavior**: Synchronizer waits indefinitely for missing shard
   - **Detection**: No automatic detection (no timeouts in V1)
   - **Recovery**: Manual intervention (create Terminal artefact or delete instance)
   - **Logging**: Synchronizer logs "Type 'X': found 4 of 5 expected" repeatedly
   - **Impact**: Workflow hangs forever
   - **Documented Limitation**: Known risk, acceptable for V1

7. **Metadata missing from producer artefact**:
   - **Behavior**: `getExpectedCountFromMetadata()` returns error
   - **Detection**: JSON parsing error or missing key
   - **Recovery**: Synchronizer skips bid (logs "metadata missing")
   - **Logging**: "Failed to read metadata 'batch_size': key not found"
   - **Impact**: Workflow hangs (Synchronizer never bids)
   - **Mitigation**: Pup MUST inject metadata (tested in unit tests)

8. **Metadata value invalid (not integer)**:
   - **Behavior**: `strconv.Atoi()` returns error
   - **Detection**: Conversion error
   - **Recovery**: Synchronizer skips bid (logs "invalid metadata")
   - **Logging**: "Metadata value 'abc' is not a valid integer"
   - **Impact**: Workflow hangs
   - **Mitigation**: Pup validates metadata format before injection

**Descendant traversal failures:**

9. **Cyclic reference in artefact graph**:
   - **Behavior**: `GetDescendants()` detects cycle via `visited` map
   - **Detection**: Artefact ID already in visited set
   - **Recovery**: Skip revisiting (prevents infinite loop)
   - **Logging**: No log (normal branch handling)
   - **Impact**: Correct traversal (cycle ignored)

10. **Deep recursion exceeds max_depth**:
    - **Behavior**: Traversal stops at configured depth
    - **Detection**: `depth > maxDepth` check
    - **Recovery**: Return collected descendants (partial tree)
    - **Logging**: No log (normal limiting behavior)
    - **Impact**: Synchronizer may miss deep descendants (expected)

11. **Redis unavailable during descendant traversal**:
    - **Behavior**: `SMEMBERS` or `GetArtefact()` fails
    - **Detection**: Redis connection error
    - **Recovery**: Return error, Synchronizer skips bid
    - **Logging**: "Failed to get descendants: connection refused"
    - **Impact**: Synchronizer doesn't bid this cycle (retried on next claim event)

**Buffer-and-flush failures:**

12. **Tool process exits before writing to FD3**:
    - **Behavior**: `outputBuffer` is empty
    - **Detection**: `parseToolOutputs()` returns zero outputs
    - **Recovery**: Return error "no outputs produced by tool"
    - **Logging**: "Tool produced no outputs"
    - **Impact**: Pup fails, orchestrator creates Failure artefact

13. **Tool writes malformed JSON to FD3**:
    - **Behavior**: `json.Decode()` returns error
    - **Detection**: JSON parsing error
    - **Recovery**: Return error "invalid JSON output"
    - **Logging**: "Failed to parse tool outputs: invalid character 'x'"
    - **Impact**: Pup fails, orchestrator creates Failure artefact

14. **Tool writes extremely large output (> 10MB)**:
    - **Behavior**: `outputBuffer` consumes large memory
    - **Detection**: No automatic detection
    - **Recovery**: None (buffer grows to fit)
    - **Logging**: No log
    - **Impact**: Pup memory usage increases, possible OOM
    - **Mitigation**: Document recommended batch sizes (< 1000 artefacts)

### **6.2. Concurrency considerations**

**Race condition: Multiple Synchronizers bid on same ancestor**:
- **Scenario**: Two worker containers evaluate claim for final prerequisite simultaneously
- **Protection**: Deduplication lock (SET NX) guarantees only one acquires lock
- **Behavior**:
  1. Worker A and Worker B both evaluate claim (target completes prerequisites)
  2. Both find ancestor, both check dependencies (all met)
  3. Worker A calls `AcquireSyncLock()` → SET NX succeeds
  4. Worker B calls `AcquireSyncLock()` → SET NX fails (key exists)
  5. Worker A submits bid
  6. Worker B skips bid (logs "deduplication")
- **Safety**: Only 1 bid submitted (CORRECT)
- **Test**: E2E test with 2 workers completing simultaneously

**Race condition: Pup crash between lock acquisition and bid submission**:
- **Scenario**: Pup acquires lock, container crashes (OOM, SIGKILL)
- **Protection**: TTL on lock (10 minutes)
- **Behavior**:
  1. Pup acquires lock
  2. Pup crashes before submitting bid
  3. Lock remains in Redis (orphaned)
  4. Other Pups cannot acquire lock for same ancestor
  5. After 10 minutes, TTL expires, lock deleted
  6. Next Pup can acquire lock and bid
- **Impact**: 10-minute deadlock
- **Safety**: Eventually recovers (TTL), but workflow delayed
- **Mitigation**: Monitor for orphaned locks, manual deletion if needed
- **Documented Limitation**: Acceptable for V1

**Race condition: Concurrent artefact creation with same parent**:
- **Scenario**: Two agents create artefacts with same `source_artefacts` simultaneously
- **Protection**: Lua script atomicity (Redis single-threaded execution)
- **Behavior**:
  1. Agent A creates artefact-1 (parent: X)
  2. Agent B creates artefact-2 (parent: X)
  3. Lua script 1 executes: SADD holt:inst:index:children:X artefact-1
  4. Lua script 2 executes: SADD holt:inst:index:children:X artefact-2
  5. Final Set: {artefact-1, artefact-2}
- **Safety**: Both children added to reverse index (CORRECT)
- **Test**: Integration test with concurrent artefact creation

**Race condition: Orchestrator restart during synchronization**:
- **Scenario**: Orchestrator crashes while Synchronizer claim is pending_consensus
- **Protection**: M3.5 recovery logic (scans pending claims on restart)
- **Behavior**:
  1. Synchronizer evaluates claim, acquires lock, submits bid
  2. Orchestrator crashes before processing bid
  3. Orchestrator restarts, scans claims in pending_consensus
  4. Bid still exists in Redis (persistent)
  5. Orchestrator processes bid normally
  6. Synchronizer claim granted
- **Safety**: No data loss, workflow continues
- **Test**: Integration test (restart orchestrator during consensus)

**Race condition: Worker exits while Synchronizer is traversing descendants**:
- **Scenario**: Worker completes, creates artefact, exits; Synchronizer traverses simultaneously
- **Protection**: Artefact published to Redis before worker exits
- **Behavior**:
  1. Worker creates final prerequisite artefact via Lua
  2. Lua publishes artefact event
  3. Worker exits
  4. Synchronizer receives artefact event
  5. Synchronizer traverses descendants (artefact exists in Redis)
  6. Synchronizer bids
- **Safety**: Artefact visible to Synchronizer (CORRECT)
- **No race**: Lua publishes before worker exits

### **6.3. Edge case handling**

**Edge case: Empty source_artefacts**:
- **Scenario**: Artefact has `source_artefacts: []` (e.g., GoalDefined)
- **Behavior**: Lua script skips reverse index update (no parents)
- **Impact**: No reverse index entry (CORRECT, no parent)
- **Test**: Unit test for GoalDefined artefact creation

**Edge case: Single artefact output (batch_size=1)**:
- **Scenario**: Agent writes 1 JSON object to FD3
- **Behavior**: Pup injects `{"batch_size": "1"}` metadata
- **Impact**: Metadata exists even for single outputs (consistent behavior)
- **Test**: Unit test for single output

**Edge case: Agent produces zero outputs**:
- **Scenario**: Agent exits cleanly but writes nothing to FD3
- **Behavior**: `parseToolOutputs()` returns error "no outputs produced"
- **Recovery**: Pup fails, orchestrator creates Failure artefact
- **Test**: Unit test for zero outputs

**Edge case: Metadata key collision**:
- **Scenario**: Agent output includes `batch_size` in payload, Pup injects `batch_size` in metadata
- **Behavior**: No collision (payload and metadata are separate fields)
- **Impact**: Both values coexist (CORRECT)

**Edge case: Synchronizer with max_depth=0 (unlimited)**:
- **Scenario**: Config omits `max_depth` (defaults to 0)
- **Behavior**: Traversal continues until all descendants found (no limit)
- **Impact**: Could be expensive for deep trees
- **Mitigation**: Document recommended max_depth values

**Edge case: Synchronizer evaluates claim for non-trigger artefact**:
- **Scenario**: Claim for artefact type not in `wait_for` list
- **Behavior**: `isPotentialTrigger()` returns false, bid skipped immediately
- **Impact**: Efficient (no traversal performed)
- **Test**: Unit test for non-trigger claim

**Edge case: Multiple wait_for conditions with same type**:
- **Scenario**: Config has `wait_for: [{type: "TestResult"}, {type: "TestResult"}]`
- **Behavior**: Config validation allows (no duplicate check in V1)
- **Impact**: Both conditions check same descendants (redundant but harmless)
- **Future**: Add duplicate detection in validation

**Edge case: Producer-Declared with Named pattern mixed**:
- **Scenario**:
  ```yaml
  wait_for:
    - type: "TestResult"
      count_from_metadata: "test_count"
    - type: "LintResult"  # Named
  ```
- **Behavior**: Synchronizer checks both conditions (dynamic count + presence)
- **Impact**: Mixed patterns supported (CORRECT)
- **Test**: E2E test with mixed patterns

**Edge case: Lock TTL expires mid-bid**:
- **Scenario**: Pup acquires lock, takes 11 minutes to submit bid
- **Behavior**: Lock expires after 10 minutes, another Pup could acquire lock
- **Impact**: Possible duplicate bid (RARE, Pup bidding takes < 200ms typically)
- **Mitigation**: TTL is generous (10 minutes), extremely unlikely
- **Accepted Risk**: Not mitigated in V1

**Edge case: Reverse index bloat over time**:
- **Scenario**: Long-running instance (weeks) creates millions of artefacts
- **Behavior**: Reverse index Sets grow indefinitely (no cleanup)
- **Impact**: Redis memory usage increases
- **Mitigation**: Periodic instance recycling (documented recommendation)
- **Future**: Implement reverse index garbage collection

**Edge case: Ancestor artefact deleted from Redis**:
- **Scenario**: Manual deletion or Redis eviction (unlikely)
- **Behavior**: `findCommonAncestor()` fails to load artefact
- **Recovery**: Returns error, Synchronizer skips bid
- **Impact**: Workflow hangs (ancestor not found)
- **Mitigation**: Redis persistence enabled, no eviction policy

### **8.2. Common pitfalls to avoid**

**Lua script pitfalls:**

1. **Forgetting to parse source_artefacts JSON**:
   - Lua receives JSON array as string: `"[\"uuid1\",\"uuid2\"]"`
   - MUST decode with `cjson.decode()` before iterating
   - Test with empty array, single parent, multiple parents

2. **Extracting instance name from key incorrectly**:
   - Script needs instance name to build reverse index key
   - Pattern: `holt:({instance_name}):artefact:{uuid}`
   - Use `string.match()` to extract
   - Test with different instance names

3. **Not handling empty source_artefacts**:
   - Empty array (`[]`) should skip reverse index update
   - Don't create empty Sets in Redis
   - Test with GoalDefined artefact

**Buffer-and-flush pitfalls:**

4. **Reading FD3 before process exits**:
   - MUST wait for `cmd.Wait()` to complete
   - Otherwise buffer may be incomplete
   - Use goroutine to copy FD3 → buffer
   - Test with slow-exiting processes

5. **Not validating batch_size > 0**:
   - `len(outputs)` is always ≥ 1 (zero outputs = error)
   - But good to validate explicitly
   - Test with single output (batch_size=1)

6. **Parsing multiple JSON objects incorrectly**:
   - Use `json.Decoder` with loop, NOT `json.Unmarshal` once
   - Decoder advances stream after each object
   - Test with 1, 5, 100 outputs

**Synchronizer pitfalls:**

7. **Not checking lock acquisition result**:
   - `AcquireSyncLock()` returns `bool` (not error)
   - If false, MUST skip bid (deduplication)
   - Test with simultaneous completions

8. **Forgetting to hash agent role for lock key**:
   - Lock key uses SHA-256 hash of role (not raw role)
   - Prevents key collision attacks
   - Use first 8 chars of hex digest

9. **Not handling metadata missing gracefully**:
   - `getExpectedCountFromMetadata()` can fail
   - Don't panic, just skip bid (workflow hangs, but safe)
   - Log clearly for debugging

10. **Traversing upward instead of downward**:
    - `findCommonAncestor()` traverses UP (source_artefacts)
    - `GetDescendants()` traverses DOWN (reverse index)
    - Don't confuse the two directions

11. **Not preventing cycles in traversal**:
    - Artefact graph SHOULD be acyclic, but validate
    - Use `visited` map in both directions
    - Test with manually created cycle

**Configuration pitfalls:**

12. **Not validating synchronize exclusivity**:
    - `synchronize` + `bidding_strategy` = ERROR
    - `synchronize` + `bid_script` = ERROR
    - Enforce in validation, not runtime

13. **Not defaulting max_depth to 0 (unlimited)**:
    - Omitted `max_depth` should mean unlimited
    - Don't accidentally set to 1 (would break deep trees)

**Lua adoption pitfalls:**

14. **Forgetting to update ALL artefact creation paths**:
    - Grep for direct `HSET holt:*:artefact:*` calls
    - MUST replace all with Lua script
    - CLI, Orchestrator, Pup all use Lua
    - Test that no direct writes remain

15. **Not handling Lua failure as FATAL**:
    - If Lua fails, PANIC/EXIT immediately
    - Don't return error and continue
    - Document that Lua failure is unrecoverable

### **9.2. Rollback and disaster recovery**

**Rollback to M4.x:**

This is **difficult** because Lua script adoption is universal. Rollback requires:

1. **Code rollback**: Deploy M4.x orchestrator + CLI + pup binaries
2. **Artefact compatibility**: Artefacts with `Metadata` field are backward compatible (field optional)
3. **Reverse index**: Will NOT be maintained for new artefacts (index stops growing)
4. **Synchronizers**: Stop working (no reverse index updates, can't find descendants)
5. **Impact**: Synchronizer workflows break, standard workflows continue

**Recommendation**: Do NOT roll back unless critical bug. Instead, fix forward.

**Disaster recovery scenarios:**

1. **Lua script bug causes all artefact creation to fail**:
   - **Symptom**: All workflows halt (no artefacts created)
   - **Detection**: Logs show "FATAL: Lua script create_artefact failed" repeatedly
   - **Recovery**:
     1. Identify bug in Lua script (syntax error, logic bug)
     2. Fix Lua script in `pkg/blackboard/lua/create_artefact.lua`
     3. Deploy hotfix to all components
     4. Restart all components (`holt down && holt up`)
   - **Prevention**: Comprehensive Lua unit tests before deployment

2. **Redis out of memory (reverse index bloat)**:
   - **Symptom**: Lua script fails with OOM error
   - **Detection**: Redis logs show memory eviction warnings
   - **Recovery**:
     1. Increase Redis memory limit (config change)
     2. OR delete old reverse index keys (manual cleanup):
        ```bash
        redis-cli --scan --pattern "holt:*:index:children:*" | xargs redis-cli DEL
        ```
     3. Restart workflows after cleanup
   - **Prevention**: Monitor Redis memory usage, periodic instance recycling

3. **Lock orphaning deadlock (Pup crash after lock acquisition)**:
   - **Symptom**: Workflow stalls, synchronizer logs show "lock already held" repeatedly
   - **Detection**: Lock exists in Redis, no active Pup holding it
   - **Recovery**:
     1. Identify orphaned lock key:
        ```bash
        redis-cli KEYS "holt:*:sync_dedup:*"
        ```
     2. Verify lock is orphaned (check Pup container still exists):
        ```bash
        docker ps | grep pup
        ```
     3. Delete orphaned lock:
        ```bash
        redis-cli DEL holt:inst:sync_dedup:commit-abc:deployer-hash
        ```
     4. Workflow resumes (next claim event triggers re-evaluation)
   - **Prevention**: Monitor for locks older than 10 minutes (alert on TTL near expiry)

4. **Partial fan-in hang (missing prerequisite)**:
   - **Symptom**: Synchronizer evaluates claim repeatedly but never bids
   - **Detection**: Logs show "found X of Y expected" repeatedly
   - **Recovery**:
     1. Identify missing prerequisite type
     2. Check if prerequisite failed (Failure artefact exists)
     3. Manual intervention:
        - Option A: Create Terminal artefact to end workflow
        - Option B: Manually create missing artefact (if safe)
        - Option C: Delete instance and restart
   - **Prevention**: Monitor for synchronizers waiting > 1 hour

**Recovery time objectives:**

- Lua script bug fix: <30 minutes (hotfix + deploy + restart)
- Redis OOM recovery: <10 minutes (increase memory or cleanup)
- Lock orphaning: <5 minutes (manual lock deletion)
- Partial fan-in hang: <10 minutes (manual intervention)

   # Fan-In Synchronization Guide

   ## Overview
   Fan-in synchronization allows agents to wait for multiple parallel prerequisites before executing.

   ## Configuration

   ### Named Pattern
   Wait for specific distinct types:
   ```yaml
   agents:
     deployer:
       role: "Deployer"
       image: "deployer:latest"
       command: ["/app/deploy.sh"]
       synchronize:
         ancestor_type: "CodeCommit"
         wait_for:
           - type: "TestResult"
           - type: "LintResult"
           - type: "SecurityScan"
   ```

   ### Producer-Declared Pattern
   Wait for dynamic count from metadata:
   ```yaml
   agents:
     aggregator:
       role: "Aggregator"
       image: "aggregator:latest"
       command: ["/app/aggregate.sh"]
       synchronize:
         ancestor_type: "DataBatch"
         wait_for:
           - type: "ProcessedRecord"
             count_from_metadata: "batch_size"
   ```

   ## Multi-Artefact Output

   Agents can produce multiple artefacts in one execution:

   ```bash
   #!/bin/bash
   # run.sh

   # Output artefact 1 to FD3
   echo '{"artefact_type":"TestResult","artefact_payload":"test1.json"}' >&3

   # Output artefact 2 to FD3
   echo '{"artefact_type":"TestResult","artefact_payload":"test2.json"}' >&3

   # Pup will inject {"batch_size":"2"} metadata automatically
   ```

   ## Known Limitations

   ### Deadlock on Pup Crash
   If a Pup crashes after acquiring the synchronization lock but before submitting a bid, the workflow will deadlock for 10 minutes (TTL expiry). Manual lock deletion required.

   ### Partial Fan-In Hang
   If one prerequisite fails (e.g., 4 of 5 shards complete, 1 crashes), the Synchronizer will wait indefinitely. No automatic failure detection or timeout in V1.

   ### Reverse Index Bloat
   Reverse index Sets grow indefinitely (no cleanup). For long-running instances, periodic recycling recommended.
   ```
