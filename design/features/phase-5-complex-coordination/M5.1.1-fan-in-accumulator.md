# **Feature design: M5.1.1 - Fan-In Accumulator (Merge Phase)**

**Purpose**: Move fan-in synchronization from Pup-based locking to Orchestrator-driven accumulation using a new Merge phase.

**Scope**: Orchestrator, Pup, Blackboard, and Configuration
**Estimated tokens**: ~8,000 tokens
**Read when**: Implementing the Fan-In Accumulator pattern for batch synchronization workflows.

Associated phase: **Complex Coordination (Phase 5)**
Status: **Draft**

***Template purpose:*** This document specifies the architecture for Fan-In Accumulator using the **Merge Phase** pattern to eliminate race conditions and provide proper audit trails for batch synchronization.

---

## **1. The 'why': goal and success criteria**

### **1.1. Goal statement**

Robustly handle the synchronization of N parallel workflow branches into a single merge operation by introducing a `merge` phase as the 4th phase in Holt's claim lifecycle. This moves accumulation logic from Pup (graph-scanning with locks) to Orchestrator (stateful accumulation), eliminating race conditions and providing complete audit trails.

### **1.2. User story**

**As an agent developer**, I want to declare that my agent merges N parallel results without writing complex graph-traversal code or managing distributed locks. I configure a `synchronize` block with batch expectations, and Holt automatically triggers my agent when all N items complete their review/parallel/exclusive phases.

**As a system operator**, I want to audit exactly which claims were accumulated into each merge operation, and see the complete lifecycle of fan-in synchronization in the blackboard without relying on ephemeral Redis locks.

### **1.3. Success criteria**

1. **Declarative Configuration**: Agents declare merge requirements in `holt.yml` using existing `synchronize` block with `count_from_metadata` field.

2. **Automatic Triggering**: When N claims reach merge phase and batch_size is met, Orchestrator automatically grants a Fan-In Claim to the merge agent without agent polling.

3. **Race-Free Accumulation**: Multiple concurrent merge workflows (different ancestors) operate independently. Within a batch, exactly-once guarantee prevents duplicate Fan-In Claims even under concurrent bid processing.

4. **Complete Audit Trail**: Every Fan-In Claim records all accumulated claim IDs, merge ancestor, batch size, and lifecycle transitions (accumulating → granted → complete) on the blackboard.

5. **Review Integration**: Merge phase occurs AFTER all other phases, so accumulated claims have already passed reviews. Rejected artefacts create new claims for revised versions.

6. **E2E Test Validation**: Existing M5.1 and M5.2 tests pass with merge phase implementation, demonstrating backward compatibility with Named Pattern and Producer-Declared Pattern.

### **1.4. Non-goals**

- **Cross-Ancestor Merging**: This milestone does not support merging items from multiple different ancestors into a single operation.
- **Partial Batch Handling**: No timeout-based "partial batch" triggering. Agent only executes when batch_size is exactly met.
- **Dynamic Batch Resizing**: Batch size is read once from ancestor metadata and remains fixed for that batch.
- **Merge Phase Ordering**: This milestone does not support ordering/sequencing of items within a batch.

## **2. The 'what': component impact analysis**

### **2.1. Blackboard changes**

#### **New Data Structure: Claim-Accumulator (Redis Hash)**

**Purpose**: The Claim-Accumulator IS the Fan-In Claim. It tracks accumulated claims and lifecycle state.

**Redis Key Pattern**:
```
holt:{inst}:claim-accumulator:{ancestor_id}:{role}
```

**Redis Hash Fields**:
```go
{
  "id":                 "uuid",                    // Fan-In Claim ID
  "status":             "accumulating|granted|complete",
  "claimer":            "recomposer",              // Agent role
  "merge_ancestor":     "ancestor-artefact-uuid",  // The common ancestor
  "batch_size":         "15",                      // Expected count
  "target_type":        "HPOMappingResult",        // Type being accumulated
  "granted_at_ms":      "timestamp",               // When granted
  "completed_at_ms":    "timestamp",               // When agent finished
  "created_at_ms":      "timestamp"                // When first claim added
}
```

**Redis Set Field** (within same Hash):
```
accumulated_claims: Set of claim IDs
```

**Key Characteristics**:
- One accumulator per (ancestor, role) tuple
- Supports multiple concurrent batches for different ancestors
- Persisted for audit trail (never auto-deleted)
- Unique identifier for batch merge operation

#### **Updated Bid Schema**

**Add Metadata Field** (already exists from M5.1 draft):
```go

    TimestampMs int64             `json:"timestamp_ms"`

}
```

**Merge Bid Metadata** (submitted by Pup):
```json
{
  "ancestor_id": "uuid-of-goal",
  "batch_size": "15",
  "target_type": "HPOMappingResult"
}
```

#### **New BidType Enum Value**


#### **No New Pub/Sub Channels**

Existing `grant_notifications:{instance}:{role}` channel is reused. Grant message includes Fan-In Claim ID and type indicator.

### **2.2. Orchestrator changes**

#### **New Phase: Merge Phase (4th Phase)**

**Phase Order** (updated from 3 to 4 phases):
```
1. Review Phase   (optional, multi-bidder)
2. Claim Phase    (parallel, multi-bidder)
3. Exclusive Phase (single bidder)
4. Merge Phase    (NEW - accumulation-based) ← Happens AFTER all others
```

**Key Insight**: By placing Merge AFTER Exclusive, we guarantee that all accumulated claims have:
- Passed reviews (or were terminated and re-created for v2)
- Completed exclusive work (if any exclusive bidders existed)
- Reached a stable, final state for their individual artefact

#### **New File: `internal/orchestrator/merge_phase.go`**

**Function: `GrantMergePhase(ctx, claim, mergeBids)`**

**Pseudocode**:
```go
func (e *Engine) GrantMergePhase(ctx context.Context, claim *blackboard.Claim, mergeBids []*blackboard.Bid) error {
    // For each merge bid (usually only one agent bids merge for a given claim)
    for _, bid := range mergeBids {
        ancestorID := bid.Metadata["ancestor_id"]
        batchSize := bid.Metadata["batch_size"]
        targetType := bid.Metadata["target_type"]
        role := bid.AgentName

        // Atomic operation via Lua script:
        // 1. Add claim ID to accumulator
        // 2. Check if batch complete
        // 3. Return status
        result := e.executeAccumulatorLuaScript(ctx, ancestorID, role, claim.ID, batchSize, targetType)

        if result == BATCH_COMPLETE {
            // Mark all accumulated claims as complete
            accumulatedClaimIDs := e.getAccumulatedClaims(ctx, ancestorID, role)
            for _, claimID := range accumulatedClaimIDs {
                e.markClaimComplete(ctx, claimID)
            }

            // Update accumulator status to "granted"
            e.updateAccumulatorStatus(ctx, ancestorID, role, "granted")

            // Publish grant notification (agent receives Fan-In Claim)
            e.publishGrantNotification(ctx, role, ancestorID, "merge")

            log.Printf("[Orchestrator] ✓ Batch complete for %s: Granted Fan-In Claim to %s", ancestorID, role)
        } else {
            // Still accumulating, no action needed
            log.Printf("[Orchestrator] Accumulating %s for %s: %d/%s claims", targetType, role, result.CurrentCount, batchSize)
        }
    }

    // Mark the current claim as complete (it's been added to accumulator)
    claim.Status = ClaimStatusComplete
    e.client.UpdateClaim(ctx, claim)

    return nil
}
```

#### **Lua Script: `accumulator_add_and_check.lua`**

**Purpose**: Atomically add claim to accumulator, check batch completion, create/update accumulator Hash.

**Script**:
```lua
-- KEYS[1] = claim-accumulator:{ancestor}:{role}
-- ARGV[1] = claim_id
-- ARGV[2] = batch_size (string)
-- ARGV[3] = merge_ancestor (ancestor artefact ID)
-- ARGV[4] = target_type
-- ARGV[5] = claimer (role name)

local key = KEYS[1]
local claim_id = ARGV[1]
local batch_size = tonumber(ARGV[2])
local merge_ancestor = ARGV[3]
local target_type = ARGV[4]
local claimer = ARGV[5]

-- Check if accumulator exists
local exists = redis.call('EXISTS', key)

if exists == 0 then
    -- First claim: Initialize accumulator Hash
    local accumulator_id = redis.call('INCR', 'holt:claim-accumulator:id-counter')
    local now_ms = redis.call('TIME')[1] .. redis.call('TIME')[2]:sub(1,3)

    redis.call('HSET', key, 'id', accumulator_id)
    redis.call('HSET', key, 'status', 'accumulating')
    redis.call('HSET', key, 'claimer', claimer)
    redis.call('HSET', key, 'merge_ancestor', merge_ancestor)
    redis.call('HSET', key, 'batch_size', batch_size)
    redis.call('HSET', key, 'target_type', target_type)
    redis.call('HSET', key, 'created_at_ms', now_ms)
end

-- Add claim to accumulated set (idempotent)
redis.call('SADD', key .. ':claims', claim_id)

-- Get current count
local count = redis.call('SCARD', key .. ':claims')

-- Check if batch complete
if count == batch_size then
    return 1  -- BATCH_COMPLETE
else
    return 0  -- ACCUMULATING (return current count for logging)
end
```

**Return Values**:
- `1`: Batch complete, trigger grant
- `0`: Still accumulating

#### **Updated Main Claim Processing Loop**

**File**: `internal/orchestrator/engine.go`

**Modification**: After Exclusive phase, add Merge phase:

```go
func (e *Engine) processClaim(ctx context.Context, claim *blackboard.Claim) error {
    // ... existing review/claim/exclusive phase logic ...

    // NEW: Merge Phase (4th phase)
    mergeBids := e.collectBidsOfType(ctx, claim.ID, blackboard.BidTypeMerge)
    if len(mergeBids) > 0 {
        if err := e.GrantMergePhase(ctx, claim, mergeBids); err != nil {
            log.Printf("[Orchestrator] Failed to grant merge phase: %v", err)
        }
        return nil // Merge phase handled
    }

    // If no merge bids, claim completes normally
    claim.Status = ClaimStatusComplete
    e.client.UpdateClaim(ctx, claim)

    return nil
}
```

### **2.3. Agent pup changes**

#### **New Logic: Merge Bid Evaluation**

**File**: `internal/pup/controller.go`

**Function**: `shouldBidMerge(ctx, claim, config)`

**Pseudocode**:
```go
func (c *Controller) shouldBidMerge(ctx context.Context, claim *blackboard.Claim, syncConfig *config.SynchronizeConfig) (*blackboard.Bid, error) {
    // Only bid merge if agent has synchronize config with count_from_metadata
    if syncConfig == nil {
        return nil, nil
    }

    // Check if this is batch pattern (count_from_metadata present)
    var countFromMetadata string
    for _, condition := range syncConfig.WaitFor {
        if condition.CountFromMetadata != "" {
            countFromMetadata = condition.CountFromMetadata
            break
        }
    }

    if countFromMetadata == "" {
        // Named pattern, not merge pattern
        return nil, nil
    }

    // Get claim's artefact
    artefact, err := c.bbClient.GetArtefact(ctx, claim.ArtefactID)
    if err != nil {
        return nil, err
    }

    // Traverse graph to find ancestor
    ancestor, err := c.findAncestor(ctx, artefact, syncConfig.AncestorType, syncConfig.MaxDepth)
    if err != nil {
        log.Printf("[Pup] Cannot find ancestor type %s: %v", syncConfig.AncestorType, err)
        return nil, nil // Cannot bid
    }

    // Get direct children of ancestor
    children, err := c.getDirectChildren(ctx, ancestor)
    if err != nil || len(children) == 0 {
        return nil, fmt.Errorf("ancestor has no children")
    }

    // Read batch_size from first child's metadata
    firstChild := children[0]
    var metadata map[string]interface{}
    if err := json.Unmarshal([]byte(firstChild.Header.Metadata), &metadata); err != nil {
        return nil, fmt.Errorf("failed to parse child metadata: %w", err)
    }

    batchSizeRaw, ok := metadata[countFromMetadata]
    if !ok {
        return nil, fmt.Errorf("metadata field '%s' not found in child artefact", countFromMetadata)
    }

    batchSize := fmt.Sprintf("%v", batchSizeRaw) // Convert to string

    // Construct merge bid
    bid := &blackboard.Bid{
        AgentName:   c.config.Role,
        BidType:     blackboard.BidTypeMerge,
        TimestampMs: time.Now().UnixMilli(),
        Metadata: map[string]string{
            "ancestor_id": ancestor.ID,
            "batch_size":  batchSize,
            "target_type": artefact.Header.Type,
        },
    }

    log.Printf("[Pup] Bidding MERGE for %s (ancestor: %s, batch: %s)", artefact.Header.Type, ancestor.ID[:16], batchSize)

    return bid, nil
}
```

**Integration Point**: Add to existing bid evaluation loop:

```go
func (c *Controller) evaluateClaims(ctx context.Context) {
    for _, claim := range pendingClaims {
        // ... existing review/claim/exclusive bid logic ...

        // NEW: Evaluate merge bid
        if c.config.Synchronize != nil {
            mergeBid, err := c.shouldBidMerge(ctx, claim, c.config.Synchronize)
            if err != nil {
                log.Printf("[Pup] Merge bid evaluation failed: %v", err)
            } else if mergeBid != nil {
                c.bbClient.SubmitBid(ctx, claim.ID, mergeBid)
            }
        }
    }
}
```

```json
  },
  "descendant_artefacts": [
    {
      "id": "review-result-1",

    },
    {
      "id": "review-result-2",

```

**Key Points**:
- `ancestor_artefact`: The merge ancestor (e.g., GoalDefined)
- `descendant_artefacts`: Filtered to wait_for types only (reuse M5.2 filtering logic)
- Deduplicated by LogicalThreadID (keep latest version)
- `accumulated_claims`: List of claim IDs that were merged (for audit)

**Reuse Existing Logic**: M5.2's `prepareToolInput` filtering applies:
```go
// Filter descendants to only include wait_for types
waitForTypes := make(map[string]bool)
for _, condition := range e.config.SynchronizeConfig.WaitFor {
    waitForTypes[condition.Type] = true
}

// Deduplicate by LogicalThreadID (keep latest version)
latestByThread := make(map[string]*blackboard.Artefact)
for _, desc := range descendants {
    if !waitForTypes[desc.Header.Type] {
        continue // Not a wait_for type, skip
    }

    threadID := desc.Header.LogicalThreadID
    existing, exists := latestByThread[threadID]
    if !exists || desc.Header.Version > existing.Header.Version {
        latestByThread[threadID] = desc
    }
}
```

## **3. The 'how': implementation & testing plan**

### **3.1. Key design decisions & risks**

#### **Decision 1: Claim-Accumulator IS the Claim**

**Why**: Eliminates duplication between tracking structure and claim entity. Provides single source of truth for batch state.

**Alternative Considered**: Separate tracking Set + Create Claim when complete
**Rejected Because**: Requires two Redis structures and complex lifecycle synchronization.

**Risk**: Redis Hash might grow large for massive batches.
**Mitigation**: Batch sizes expected to be < 1000 items. Hash overhead is minimal.

#### **Decision 2: Merge Phase After Exclusive Phase**

**Why**: Guarantees all accumulated claims have completed reviews and exclusive work. Sidesteps the M5.2 event-driven re-evaluation problem (UpdateClaim doesn't publish events).

**Alternative Considered**: Merge bids evaluated in parallel with exclusive
**Rejected Because**: Would require event-driven re-evaluation when claim status changes.

**Risk**: None. This is strictly better than alternatives.

#### **Decision 3: Batch Size from First Direct Child**

**Why**: Simplifies configuration. Producer declares count once in ancestor's children metadata, avoiding N redundant metadata entries in leaf artefacts.

**Alternative Considered**: Read from all children and validate consistency
**Rejected Because**: Unnecessary complexity. Misconfiguration will fail fast (wrong count).

**Risk**: Silent failure if metadata key is misspelled.
**Mitigation**: Validation error logged by Pup if key not found.

#### **Decision 4: Lua Script for Atomicity**

**Why**: Prevents race conditions where two bids both see count==batch_size and create duplicate Fan-In Claims.

**Alternative Considered**: Go-side SADD + SCARD (non-atomic)
**Rejected Because**: Race condition under concurrent bid processing.

**Risk**: Lua script bugs harder to debug than Go code.
**Mitigation**: Comprehensive unit tests for Lua script using Redis testcontainer.

### **3.3. Performance & resource considerations**

#### **Resource Usage**

**Redis Memory**:
- Per accumulator: ~500 bytes (Hash overhead + metadata)
- Per claim in accumulator: ~40 bytes (Set member = UUID string)
- Example: Batch of 100 items = 500 + (100 * 40) = 4.5 KB
- Expected: 1000 concurrent batches = 4.5 MB (negligible)

**CPU**:
- Lua script execution: ~1ms per merge bid evaluation
- Graph traversal (Pup): ~10ms per ancestor lookup
- Expected: Merge bids processed sequentially, no CPU bottleneck

**Network**:
- Additional bid submission per claim: ~200 bytes
- Fan-In grant message: ~1 KB (includes accumulated claim IDs)
- Expected: Negligible increase vs existing bid traffic

#### **Scalability Limits**

**Batch Size**: Tested up to 1000 items. Theoretical limit ~10,000 (Redis Set member limit).

**Concurrent Batches**: No limit. Each (ancestor, role) tuple is independent.

**Accumulator Persistence**: Accumulators never auto-deleted. Long-running instances may accumulate millions.
**Mitigation**: Future cleanup job to archive completed accumulators > 30 days old.

#### **Performance Requirements**

**Latency**: Merge phase adds < 5ms to claim processing (Lua script + status updates).

**Throughput**: Orchestrator can process 100 merge bids/second (sequential Lua execution).

**Grant Delay**: Agent receives Fan-In grant within 1 second of batch completion.

## **4. Principle compliance check**

### **4.1. YAGNI (You Ain't Gonna Need It)**

**No new dependencies introduced**. Implementation uses:
- Existing Redis (Lua script support)
- Existing Go standard library
- Existing Holt blackboard client

**Justification**: Lua scripts are standard Redis feature, no third-party dependency.

### **4.2. Auditability**

**New Artefacts Created**: None. Fan-In Claims are tracked as Claim-Accumulator Hashes in Redis.

**Audit Trail**:
- Every accumulator records: ID, ancestor, batch_size, target_type, claimer, timestamps
- Accumulated claim IDs stored in Set (immutable once granted)
- Status transitions logged: accumulating → granted → complete
- All state persisted in Redis (never auto-deleted)

**Query Example**: Find all Fan-In Claims for ancestor:
```bash
redis-cli KEYS "holt:*:claim-accumulator:{ancestor-id}:*"
```

### **4.3. Small, single-purpose components**

**Component Responsibilities**:
- **Pup**: Graph traversal, bid submission (no locking, no state)
- **Orchestrator**: Accumulator management, Fan-In Claim lifecycle (centralized state)
- **Blackboard**: Atomic operations via Lua (data layer only)

**No Tight Coupling**: Pup doesn't know about accumulator internals. Orchestrator doesn't know about graph traversal.

### **4.4. Security considerations**

**Attack Surfaces**:
- **Lua Script Injection**: Not possible. Script is embedded at build time, no user input.
- **Accumulator Poisoning**: Malicious agent cannot add arbitrary claims (claims validated by Orchestrator).
- **Batch Size Tampering**: Agent reads metadata from blackboard (immutable artefacts).

**Data Exposure**:
- Claim IDs in accumulator are UUIDs (no sensitive data).
- Ancestor ID is artefact hash (public within instance).

**No New Risks Introduced**.

### **4.5. Backward compatibility**

**Breaking Changes**: None.

**Additive Changes**:
- New BidType enum value (`merge`)
- New Redis key pattern (claim-accumulator)
- New orchestrator phase (4th phase)

**Migration Path**: Existing synchronizer-based agents can migrate to merge phase by:
1. Keeping `synchronize` config as-is
2. Removing manual locking code (if any)
3. Testing with new merge phase

## **6. Error scenarios & edge cases**

### **6.1. Failure modes**

#### **Failure 1: Redis Unavailable During Merge Bid**

**Scenario**: Orchestrator receives merge bid, but Redis is down when executing Lua script.

**Behavior**:
- Lua script execution fails with connection error
- Orchestrator logs error and skips merge grant
- Claim status remains pending (not marked complete)
- Next evaluation cycle retries merge bid evaluation

**Recovery**: When Redis recovers, merge bid re-evaluated, accumulator updated correctly.

**Test**: Simulate Redis failure during Lua script execution using testcontainer pause.

#### **Failure 2: Metadata Key Not Found**

**Scenario**: Pup tries to read `count_from_metadata` from ancestor's child, but metadata field missing.

**Behavior**:
- Pup logs error: "Metadata field 'batch_size' not found in child artefact"
- Pup does NOT submit merge bid
- Claim cannot progress (blocked waiting for merge bid that never comes)

**Detection**: Agent container logs show repeated errors.

**Mitigation**: Configuration validation at `holt up` could warn if ancestor children lack expected metadata (future enhancement).

**Test**: E2E test with misconfigured metadata, verify error logged and no hang.

#### **Failure 3: Batch Size Mismatch**

**Scenario**: Different claims provide different batch_size values in merge bids (configuration drift).

**Behavior**:
- Lua script uses batch_size from first merge bid (created accumulator)
- Subsequent bids with different batch_size log warning
- Accumulator continues with original batch_size

**Detection**: Orchestrator logs: "WARNING: Batch size mismatch for {ancestor}: expected {X}, got {Y}"

**Mitigation**: Validation could ensure all children have consistent metadata (future).

**Test**: E2E test with inconsistent metadata, verify warning logged and first value used.

#### **Failure 4: Agent Fails After Receiving Fan-In Grant**

**Scenario**: Agent command crashes or produces Failure artefact after Fan-In Claim granted.

**Behavior**:
- Pup marks claim-accumulator status as `complete` (same as any failed claim)
- Accumulated claims remain marked `complete` (not reverted)
- No automatic retry (same as exclusive claim failure)

**Recovery**: Manual intervention or retry logic (outside scope of this milestone).

**Test**: E2E test where agent returns non-zero exit code, verify accumulator marked complete.

### **6.2. Concurrency considerations**

#### **Concurrent Merge Bids for Same Batch**

**Scenario**: Two claims for same batch reach merge phase simultaneously, orchestrator processes bids concurrently.

**Protection**: Lua script provides atomic SADD + SCARD. First bid to reach batch_size triggers grant, second bid sees count > batch_size and returns "already granted" signal.

**Test**: Unit test with concurrent Lua script executions (Redis is single-threaded, but test Go-side concurrency).

#### **Concurrent Batches for Different Ancestors**

**Scenario**: Claims from Patient A batch and Patient B batch processed concurrently.

**Protection**: Separate accumulator keys per ancestor. No shared state.

**Test**: E2E test creating 10 concurrent batches, verify all complete correctly.

#### **Claim Completion Race**

**Scenario**: Orchestrator marks accumulated claims as complete, but concurrent process tries to read claim status.

**Protection**: Redis operations are atomic. Status reads always return consistent state.

**Test**: Integration test with concurrent claim status reads during marking.

### **6.3. Edge case handling**

#### **Edge Case 1: Empty Batch (batch_size=0)**

**Behavior**: Validation error at config load time. `batch_size` must be > 0.

**Test**: Config validation test rejects `{"batch_size": "0"}`.

#### **Edge Case 2: Batch Size Exceeds Claims**

**Scenario**: Metadata says `batch_size=10`, but only 5 claims ever created.

**Behavior**: Accumulator remains in `accumulating` state forever. No timeout.

**Detection**: Operator sees accumulator stuck in `accumulating` state.

**Mitigation**: Future enhancement: timeout-based partial batch triggering (out of scope).

**Test**: E2E test verifies accumulator waits indefinitely (no false triggers).

#### **Edge Case 3: Ancestor Has No Children**

**Scenario**: Ancestor artefact exists, but has no direct children to read batch_size from.

**Behavior**: Pup logs error: "Ancestor has no children". No merge bid submitted.

**Test**: Unit test with childless ancestor, verify error logged.

#### **Edge Case 4: Multiple Agents Bid Merge on Same Claim**

**Scenario**: Two different agents both have `synchronize` config targeting same ancestor.

**Behavior**: Two separate accumulators created (different role in key). Each agent gets independent Fan-In Claim.

**Example**:
- `claim-accumulator:{ancestor}:aggregator` (batch_size=5)
- `claim-accumulator:{ancestor}:reporter` (batch_size=5)

**Test**: E2E test with two agents synchronizing on same ancestor, verify independent grants.

## **7. Open questions & decisions**

### **Q1: Should accumulators be auto-deleted after completion?**

**Decision**: No. Accumulators persist for audit trail.

**Future Enhancement**: Cleanup job to archive accumulators > 30 days old to separate storage.

### **Q2: Should we validate batch_size consistency across all children?**

**Current**: Use first child only.

**Alternative**: Read all children, verify all agree on batch_size.

**Decision**: Current approach (first child only) for simplicity. Future enhancement if configuration drift becomes a real issue.

### **Q3: Should merge phase support partial batches (timeout-based)?**

**Current**: No timeout. Batch must reach exact batch_size.

**Alternative**: After N minutes, trigger Fan-In Claim with partial batch.

**Decision**: Out of scope for M5.1.1. Future M5.3 feature.

### **Q4: How to handle accumulator key collisions?**

**Scenario**: Same ancestor processed twice (e.g., retry workflow).

**Current**: Key `claim-accumulator:{ancestor}:{role}` would reuse same accumulator.

**Problem**: Second batch's claims added to first batch's accumulator.

**Solution**: This is actually correct behavior if the ancestor is truly the same artefact. If it's a different workflow, the ancestor ID should be different (different GoalDefined instance).

**Decision**: Current key structure is correct. No collision risk for distinct workflows.

## **8. AI agent implementation guidance**

### **8.1. Development approach**

1. **Start with Lua script**: Implement and test accumulator logic in isolation using Redis testcontainer. Verify atomicity and correctness before integrating with Go.

2. **Implement Pup merge bidding**: Graph traversal and bid submission can be tested independently of orchestrator. Use mock blackboard.

3. **Implement Orchestrator merge phase**: Test with mock blackboard and pre-populated accumulators. Focus on state transitions.

4. **Integration testing**: Only after all components work individually. Use docker-compose test environment.

5. **E2E testing last**: Verify complete workflow end-to-end with real agents.

### **8.2. Common pitfalls to avoid**

- **Non-atomic accumulator operations**: MUST use Lua script. Go-side SADD + SCARD has race condition.

- **Forgetting to mark accumulated claims complete**: Critical for audit trail. Implement before granting Fan-In Claim.

- **Not handling metadata parsing errors**: Agent metadata may be malformed JSON. Gracefully handle parse errors.

- **Assuming single wait_for type**: Code must support both Named Pattern (multiple types) and Batch Pattern (single type with count_from_metadata). Validate mutually exclusive.

- **Creating duplicate Fan-In Claims**: Lua script return value MUST be checked. Only create claim if script returns 1.

### **9.2. Rollback and disaster recovery**

**Feature Disablement**: Cannot disable merge phase selectively. To rollback:
1. Redeploy previous Holt version without merge phase
2. Remove `count_from_metadata` from agent configs (fall back to Named Pattern)

**Data Migration**: None required. Accumulators are forward-compatible (new keys, no schema changes to existing data).

**Cleanup Requirements**: Orphaned accumulators (from rollback) remain in Redis but have no effect. Optional cleanup:
```bash
redis-cli --scan --pattern "holt:*:claim-accumulator:*" | xargs redis-cli DEL
```

**Rollback Speed**: < 5 minutes (docker-compose restart with previous image).
