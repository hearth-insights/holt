## **1. The 'why': goal and success criteria**

### **1.1. Goal statement**
Enable the Orchestrator to automatically detect when a workflow has reached a "dead end" (quiescence) without a Terminal artefact and generate one to formally close the workflow.

### **1.2. User story**
As a user running a complex multi-agent workflow, I want the system to automatically recognize when all work is finished—even if the final agent didn't explicitly create a "Terminal" artefact—so that the `holt watch` command exits and the workflow is marked as complete.

### **1.3. Success criteria**
*   The Orchestrator automatically creates a `Terminal` artefact when:
    *   All existing claims are in a final state (`complete` or `terminated`).
    *   All `Standard` and `Answer` artefacts have been claimed (no pending work).
    *   No `Terminal` artefact currently exists.
*   `holt watch` successfully exits with a "Workflow completed (System Detected)" message in this scenario.

### **1.4. Non-goals**
*   This feature will not attempt to predict "future" work if an agent is currently running (active grant). It only acts on total system quiescence.
*   It does not replace the best practice of agents explicitly creating `Terminal` artefacts when they know they are done.

## **2. The 'what': component impact analysis**

### **2.1. Blackboard changes**
*   **No changes** to schema.
*   **New Artefact**: The Orchestrator will produce a `SystemManifest` or `Terminal` artefact with `produced_by_role="orchestrator"`.

### **2.2. Orchestrator changes**
*   **New Logic**: Implement a "Quiescence Check" loop (or event-driven check).
    *   Triggered after any Claim transitions to `complete` or `terminated`.
    *   Check 1: Are there any Claims in `pending_*` state?
    *   Check 2: Are there any `Standard`/`Answer` artefacts that do not have a corresponding Claim?
    *   If both are False, and no `Terminal` artefact exists, generate a `Terminal` artefact.
*   **Configuration**: Add `orchestrator.auto_terminate: boolean` (default `true`) to `holt.yml` to allow disabling this behavior.

### **2.3. Agent pup changes**
*   **No changes**.

### **2.4. CLI changes**
*   **No changes**. `holt watch` already respects `Terminal` artefacts.

## **3. The 'how': implementation & testing plan**

### **3.1. Key design decisions & risks**
*   **Decision**: Use "Quiescence" (No active claims + No unclaimed artefacts) as the proxy for completion.
    *   *Justification*: In Holt's model, work is only generated by Artefacts. If all Artefacts are claimed and processed, and no new ones are generated, the system is dead.
*   **Risk**: **The "Thinking" Agent Problem**. An agent might complete a claim (update status to `complete`) but still be doing internal processing before posting a *new* artefact.
    *   *Mitigation*: The "All agents must bid" rule helps, but doesn't solve the "post-grant" latency.
    *   *Refinement*: The Orchestrator can't know if an agent *intends* to post a new artefact after finishing a claim.
    *   *Counter-argument*: Agents should post their output artefact *before* or *as part of* completing the claim. In Holt, the claim is usually completed *by* the arrival of the output artefact.
    *   *Conclusion*: Since the Orchestrator marks a claim as `complete` ONLY when the output artefact arrives (for Exclusive/Parallel), there is no gap. The only gap is if an agent produces *multiple* artefacts sequentially. This is rare/discouraged (1 claim = 1 output).

### **3.2. Implementation steps**
1.  **[Orchestrator]** Add `auto_terminate` field to `Config` struct and `holt.yml`.
2.  **[Orchestrator]** Implement `CheckQuiescence(ctx)` function in `Engine`.
    *   Query Redis for all active claims.
    *   Query Redis for all unclaimed artefacts.
3.  **[Orchestrator]** Hook `CheckQuiescence` into the main event loop, triggering it whenever a claim enters a final state.
4.  **[Orchestrator]** Implement `GenerateSystemTerminal()` to create the artefact.

### **3.3. Performance & resource considerations**
*   **Redis Scan**: Scanning for "unclaimed artefacts" could be expensive if there are thousands of historical artefacts.
    *   *Optimization*: Maintain a Redis Set of `unclaimed_artefacts`. Add on creation, remove on claim creation. This makes the check O(1).

### **3.4. Testing strategy**
*   **Unit Tests**: Test `CheckQuiescence` with various mock states (active claims, pending artefacts, empty).
*   **E2E Test**:
    *   Run a simple workflow (e.g., "Hello World").
    *   Ensure the agent does NOT produce a Terminal artefact.
    *   Verify that the Orchestrator produces one automatically after the agent finishes.

## **4. Principle compliance check**

### **4.1. YAGNI**
*   Necessary to prevent "hanging" workflows in complex DAGs where the final node isn't explicitly known or is dynamic.

### **4.2. Auditability**
*   The system-generated Terminal artefact provides a clear audit trail of *why* the workflow ended (auto-termination).

### **4.3. Small, single-purpose components**
*   Logic resides entirely within the Orchestrator's state management.

### **4.4. Security considerations**
*   None.

### **4.5. Backward compatibility**
*   Fully backward compatible. Existing workflows that produce Terminal artefacts will just exit slightly earlier (or the check will see the existing Terminal and do nothing).

### **4.6. Dependency impact**
*   None.

## **5. Definition of done**
*   [ ] `holt.yml` supports `auto_terminate`.
*   [ ] Orchestrator correctly identifies quiescent state.
*   [ ] System generates Terminal artefact when quiescent.
*   [ ] `holt watch` exits automatically for "implicit" workflows.

## **6. Error scenarios & edge cases**

### **6.1. Failure modes**
*   **Redis Failure**: If Redis fails during the check, log error and retry later. Do not terminate.

### **6.2. Concurrency considerations**
*   **Race Condition**: A new artefact arrives *exactly* as the Orchestrator decides to terminate.
    *   *Mitigation*: Use a Redis transaction (Lua script) or optimistic locking to verify state hasn't changed before writing the Terminal artefact.

### **6.3. Edge case handling**
*   **Manual Intervention**: If a human is slow to answer a question (StructuralTypeQuestion), the system is "quiescent" in terms of *agent* activity, but not *workflow* activity.
    *   *Fix*: `StructuralTypeQuestion` artefacts that are unclaimed (waiting for answer) count as "unclaimed artefacts", so the system will NOT auto-terminate. Correct.

## **7. Open questions & decisions**
*   **Q**: Should we wait a "grace period" (e.g., 5 seconds) after quiescence before terminating?
    *   *Decision*: Yes, a small configurable delay (default 2s) prevents jitter in highly asynchronous systems.

## **8. AI agent implementation guidance**
*   Use the existing `blackboard.Client` methods to query state.
*   Implement the "Unclaimed Artefacts" set optimization if performance testing suggests it's needed (likely not for V1).

## **9. Operational readiness**
*   Log "Auto-terminating workflow due to quiescence" at INFO level.
