# **Feature design: Enhanced Review Artefacts with Approval Comments**

**Purpose**: Enable Review artefacts to contain comments, warnings, and metadata while still counting as APPROVAL
**Scope**: Enhancement - M5.1.1
**Estimated tokens**: ~5,000 tokens
**Read when**: Implementing enhanced review artefact validation and approval logic

Associated phase: Complex Coordination
Status: Approved
***Template purpose:*** *This document provides an unambiguous specification for implementing enhanced Review artefacts that support explicit status fields, enabling reviewers to provide positive feedback and metadata without triggering the automated feedback loop.*

## **1. The 'why': goal and success criteria**

### **1.1. Goal statement**

Enable Review artefacts to explicitly signal approval while containing non-empty payloads (comments, warnings, metadata), preventing the current limitation where any content triggers rejection and the M3.3 feedback loop.

### **1.2. User story**

As a review agent developer, I want to output Review artefacts that signal approval while including positive feedback (e.g., "Approved - great error handling approach") or warnings (e.g., "Approved with minor nit: consider adding a TODO for future refactoring"). Currently, any non-empty Review payload triggers the feedback loop, forcing me to choose between providing helpful context and approving the work. With this enhancement, I can include rich metadata in approvals, improving audit trail quality and enabling better observability in `holt watch` output.

### **1.3. Success criteria**

1. **Explicit Approval with Content**: A review agent outputs `{"status": "approved", "comments": "Excellent error handling"}` and the orchestrator treats this as APPROVAL, transitioning the claim to the next phase instead of triggering the M3.3 feedback loop.

2. **Event Enrichment Visibility**: When running `holt watch`, the `review_approved` event displays the reviewer's comments (e.g., "Approved: Excellent error handling") instead of just "Approved", providing richer audit trail context.

3. **Backward Compatibility Preservation**: Existing review agents that output `{}` for approval and `{"issue": "needs tests"}` for rejection continue to work exactly as they do today, with zero behavior changes.

4. **Validation at Circuit Breaker**: The agent pup rejects invalid Review artefacts (e.g., `{"status": "bored"}`) by creating a `ToolExecutionFailure` artefact instead of allowing ambiguous data to reach the Blackboard, maintaining data integrity.

### **1.4. Non-goals**

- **Conditional approval workflows**: This milestone does NOT support "approved with conditions" that require follow-up verification. Status is binary: approved or rejected.
- **Structured comment schemas**: This milestone does NOT enforce a specific JSON schema for comments/warnings fields. Any valid JSON is acceptable alongside the `status` field.
- **Multi-level approval**: This milestone does NOT support partial approvals or approval hierarchies (e.g., "approved by junior reviewer, pending senior review").
- **Review artefact templates**: This milestone does NOT provide templates or helpers for constructing Review payloads. Agents construct JSON directly.

## **2. The 'what': component impact analysis**

### **2.1. Blackboard changes**

**No changes required.**

- The `Artefact` schema already supports arbitrary JSON payloads as `string` type.
- No new Redis keys, channels, or data structures are needed.
- The `StructuralType` enum already includes `Review` (no additions required).

### **2.2. Orchestrator changes**

**Modified logic:**

1. **`internal/orchestrator/review_phase.go::isApproval()`** - Update approval detection logic:
   - Current: Returns `true` only for empty object `{}` or empty array `[]`
   - New: Check for explicit `status` field first (precedence), then fall back to legacy logic
   - Validation: Orchestrator MUST handle edge cases defensively (e.g., pup validation bypass scenarios)

2. **`internal/orchestrator/review_phase.go::CheckReviewPhaseCompletion()`** - Update event publishing:
   - Extract review summary/comments from approved Review artefacts
   - Pass summary to `publishReviewApprovedEvent()` as new parameter

3. **`internal/orchestrator/review_phase.go::publishReviewApprovedEvent()`** - Add review summary field:
   - Current signature: `(ctx, originalArtefactID, reviewArtefactID, reviewerRole string)`
   - New signature: `(ctx, originalArtefactID, reviewArtefactID, reviewerRole, reviewSummary string)`
   - Add `review_summary` field to workflow event payload

**No configuration changes (holt.yml).**

### **2.3. Agent pup changes**

**New validation logic in `internal/pup/execution.go`:**

1. **Circuit Breaker Validation** - Before creating Review artefacts on Blackboard:
   - IF `structural_type == "Review"` AND payload contains `"status"` key:
     - Parse payload as JSON
     - Extract `status` field value
     - Validate value is string type (reject null, boolean, number, object, array)
     - Normalize value: trim whitespace, convert to lowercase
     - Validate value is exactly `"approved"` OR `"rejected"`
     - IF validation fails: Create `ToolExecutionFailure` artefact instead of Review

2. **Error Reporting** - On validation failure:
   - Log detailed error message to agent stderr (visible in `holt logs`)
   - Create Failure artefact with structured error payload
   - Include original tool output in Failure payload for debugging

**Changes to tool execution contract:**

**No changes to stdin JSON structure.**

**Clarification of stdout JSON structure** (documentation only):
- Agents MAY now include `status` field in `artefact_payload` for Review artefacts
- Valid values: `"approved"` or `"rejected"` (case-insensitive)
- Pup will validate and reject invalid values before Blackboard submission

### **2.4. CLI changes**

**No changes to commands.**

**Changes to user output:**

- **`holt watch` enhancement**: When displaying `review_approved` events, include the `review_summary` field if present:
  - Current: `[12:34:56] REVIEW_APPROVED: CodeCommit abc-123 approved by SecurityScanner`
  - New: `[12:34:56] REVIEW_APPROVED: CodeCommit abc-123 approved by SecurityScanner - "Excellent error handling"`

**Implementation note**: The CLI's `cmd/holt/commands/watch.go` already renders workflow event data dynamically. No structural changes needed, only formatting enhancement for `review_approved` event type.

## **3. The 'how': implementation & testing plan**

### **3.1. Key design decisions & risks**

**Decision 1: Two-layer validation (Pup + Orchestrator)**
- **Rationale**: Defense-in-depth. Pup acts as circuit breaker preventing invalid data from reaching Blackboard. Orchestrator still validates defensively in case of pup bugs or direct Blackboard manipulation.
- **Risk**: Validation logic duplication could lead to inconsistency. **Mitigation**: Comprehensive unit tests for both layers, shared test cases.

**Decision 2: Status field takes absolute precedence over content**
- **Rationale**: Explicit is better than implicit. If agent declares `status`, honor it regardless of other fields (e.g., `{"status": "approved", "issue": "..."}` is APPROVAL).
- **Risk**: Confusing for developers who expect `issue` field to imply rejection. **Mitigation**: Clear documentation, validation warnings in pup logs, example agents demonstrating best practices.

**Decision 3: Strict validation of status values (only "approved"/"rejected")**
- **Rationale**: Prevent ambiguity. Rejecting creative values like "conditionally_approved" or "needs_minor_fixes" forces clarity.
- **Risk**: Frustration for developers wanting richer status taxonomy. **Mitigation**: Document that additional metadata belongs in separate fields (e.g., `{"status": "approved", "severity": "minor_concerns", "comments": "..."}`).

**Decision 4: Array payloads remain legacy-only (empty array = approval)**
- **Rationale**: Arrays are semantically confusing for status (which element represents the decision?). Objects provide clear key-value semantics.
- **Risk**: Breaking change if agents use populated arrays. **Mitigation**: Codebase analysis shows no agents use populated arrays; only empty `[]` used in tests as legacy approval signal.

### **3.3. Performance & resource considerations**

**Resource usage:**
- **CPU**: Negligible. Additional JSON parsing in pup (1 parse per Review artefact) and orchestrator (1 parse per review check). Parsing 10KB JSON payload: ~0.1ms overhead.
- **Memory**: Minimal. Review summaries capped at 500 chars in events. Full payloads already stored in Redis.
- **Storage**: No change. Artefact payloads already stored as strings in Redis.
- **Network**: Negligible. Workflow events increase by ~500 bytes per approval (review_summary field).

**Scalability limits:**
- **Payload size**: Existing artefact size limits apply (recommended max 100KB per artefact). Review comments typically 1-10KB.
- **Concurrent reviews**: No impact. Validation is stateless per artefact.
- **Agent count**: No impact. Review phase already handles N parallel reviewers.

**Performance requirements:**
- **Validation latency**: `validateReviewArtefact()` MUST complete in <10ms for 99th percentile (typical JSON payload <10KB)
- **Event publishing latency**: No change from current `publishReviewApprovedEvent()` performance (<5ms)
- **Watch display latency**: Summary extraction and rendering MUST add <1ms to event display

## **6. Error scenarios & edge cases**

### **6.1. Failure modes**

**Failure Mode 1: Redis unavailable during validation**
- **Scenario**: Pup validates Review artefact successfully, but Redis is down when attempting Blackboard submission.
- **Behavior**: Pup's existing error handling applies. Agent retries or creates Failure artefact per existing logic. No special handling needed.
- **Test**: Covered by existing pup integration tests (Redis failure scenarios).

**Failure Mode 2: Malformed JSON in Review payload**
- **Scenario**: Agent outputs `{"status": "approved"` (missing closing brace).
- **Pup behavior**: JSON parsing fails in `validateReviewArtefact()`. Creates `ToolExecutionFailure` with "Invalid JSON in Review payload" error.
- **Orchestrator behavior**: Never sees malformed Review (pup blocks it). If somehow bypassed, `isApproval()` JSON parsing fails → defaults to `false` (rejection).
- **Test**: Unit test `TestValidateReviewArtefact_MalformedJSON`.

**Failure Mode 3: Pup validation bypassed (direct Blackboard manipulation)**
- **Scenario**: Developer manually creates Review artefact on Blackboard with `{"status": "invalid"}`.
- **Orchestrator behavior**: `isApproval()` parses JSON, detects `status` field with invalid value, returns `false` (REJECTION). Claim proceeds to feedback loop.
- **Logging**: Orchestrator logs warning "Review artefact [ID] has invalid status value 'invalid', treating as rejection".
- **Test**: Unit test `TestIsApproval_DefensiveInvalidStatus`.

**Failure Mode 4: Summary extraction from extremely large payload**
- **Scenario**: Review payload contains 1MB `comments` field.
- **Behavior**: `extractReviewSummary()` truncates to 500 chars. Full payload remains in artefact (immutable). Event publishing succeeds.
- **Performance**: Minimal impact (truncation is O(1) string slice operation).
- **Test**: Unit test `TestExtractReviewSummary_LargePayload`.

**Failure Mode 5: Status field value has unexpected whitespace**
- **Scenario**: Agent outputs `{"status": "  APPROVED  \n"}` (spaces, newlines).
- **Pup behavior**: `validateReviewArtefact()` trims whitespace, normalizes to lowercase → `"approved"` → valid.
- **Orchestrator behavior**: Same normalization logic → APPROVAL.
- **Test**: Unit test `TestValidateReviewArtefact_WhitespaceTrimming`.

**Failure Mode 6: Agent outputs both old and new format**
- **Scenario**: Agent outputs `{"status": "approved", "artefact_payload": "{}"}` (confusion between pup fields and payload content).
- **Clarification**: This is not possible. The `status` field goes INSIDE `artefact_payload` (which is a JSON string). The validation checks the parsed content of `artefact_payload`, not top-level fields.
- **Correct structure**:
  ```json
  {
    "artefact_type": "Review",
    "artefact_payload": "{\"status\": \"approved\", \"summary\": \"...\"}",
    "structural_type": "Review"
  }
  ```

### **6.2. Concurrency considerations**

**Scenario 1: Multiple review agents approve simultaneously**
- **Current behavior**: Orchestrator collects all reviews in `CheckReviewPhaseCompletion()`, iterates through received artefacts sequentially.
- **New behavior**: Each approved artefact has summary extracted and published independently. No shared state modifications.
- **Race conditions**: None. Summary extraction is read-only operation on immutable artefacts.
- **Test**: Existing E2E tests with multiple reviewers cover this (no new concurrency risks).

**Scenario 2: Pup validates Review while orchestrator reads same artefact**
- **Flow**:
  1. Pup validates → creates Review artefact on Blackboard
  2. Orchestrator receives artefact creation event → reads artefact
- **Timing**: Sequential by design. Artefact must exist on Blackboard before orchestrator receives event.
- **Race conditions**: None. Redis guarantees atomicity of SET operation (artefact creation).

**Scenario 3: Controller-worker pattern with Review agents**
- **Current behavior**: Review phase grants to all review bidders (no workers launched).
- **New behavior**: No change. Review artefacts created by granted agents (controllers), not workers.
- **Impact**: None. Feature does not interact with controller-worker scaling pattern.

### **6.3. Edge case handling**

**Edge Case 1: Empty string status value**
- **Input**: `{"status": ""}`
- **Pup**: Detects empty string after trim → validation fails → `ToolExecutionFailure`
- **Error message**: "Review artefact validation failed: status field cannot be empty"

**Edge Case 2: Status field is boolean true/false**
- **Input**: `{"status": true}` or `{"status": false}`
- **Pup**: Type check detects non-string → validation fails → `ToolExecutionFailure`
- **Error message**: "Review artefact validation failed: status field must be string, got bool"

**Edge Case 3: Status field is JSON object**
- **Input**: `{"status": {"decision": "approved"}}`
- **Pup**: Type check detects object → validation fails → `ToolExecutionFailure`
- **Error message**: "Review artefact validation failed: status field must be string, got object"

**Edge Case 4: Status field is JSON array**
- **Input**: `{"status": ["approved"]}`
- **Pup**: Type check detects array → validation fails → `ToolExecutionFailure`
- **Error message**: "Review artefact validation failed: status field must be string, got array"

**Edge Case 5: Multiple status fields (impossible in JSON)**
- **Input**: Not possible in valid JSON (duplicate keys). Parser uses last value.
- **Behavior**: Standard JSON parsing behavior applies. No special handling needed.

**Edge Case 6: Status field with numeric string**
- **Input**: `{"status": "1"}` or `{"status": "200"}`
- **Pup**: String value validation detects not "approved" or "rejected" → validation fails
- **Error message**: "Review artefact validation failed: status must be 'approved' or 'rejected', got '1'"

**Edge Case 7: Unicode/non-ASCII in status value**
- **Input**: `{"status": "approved✓"}` or `{"status": "批准"}`
- **Pup**: Exact string match fails → validation fails
- **Error message**: "Review artefact validation failed: status must be 'approved' or 'rejected', got 'approved✓'"

**Edge Case 8: Null payload (not JSON null, but empty string)**
- **Input**: `artefact_payload: ""`
- **Pup**: JSON parsing fails → validation fails (but this is caught before Review-specific validation)
- **Existing behavior**: Already handled by pup's artefact creation logic

**Edge Case 9: Review with status but no summary fields**
- **Input**: `{"status": "approved"}`
- **Orchestrator**: `extractReviewSummary()` returns empty string (no text fields found)
- **Event**: `review_approved` event published without `review_summary` field (omitted if empty)
- **Watch output**: Displays "Approved" (no summary suffix)

**Edge Case 10: Capital 'S' in Status field name**
- **Input**: `{"Status": "approved"}`
- **Behavior**: Field name is case-sensitive in JSON. Pup/orchestrator look for lowercase `"status"`.
- **Result**: No `status` field found → falls back to legacy logic → non-empty object → REJECTION
- **Rationale**: Enforces standard JSON convention (lowercase keys). Not a bug, working as designed.

**Edge Case 11: Legacy array approval with new status inside**
- **Input**: `[{"status": "approved"}]` (populated array)
- **Pup**: Validation detects array type (not object) → validation fails → `ToolExecutionFailure`
- **Error message**: "Review artefact validation failed: populated arrays not supported (use object with status field)"

### **8.2. Common pitfalls to avoid**

**Pitfall 1: Forgetting to normalize status value**
- **Issue**: Treating `"APPROVED"` differently from `"approved"`.
- **Solution**: Always `strings.ToLower(strings.TrimSpace(statusStr))` before comparison.

**Pitfall 2: Not handling JSON parsing errors gracefully**
- **Issue**: Panicking on malformed JSON instead of creating Failure artefact.
- **Solution**: Wrap JSON parsing in error check:
  ```go
  var payload map[string]interface{}
  if err := json.Unmarshal([]byte(payloadStr), &payload); err != nil {
      return fmt.Errorf("invalid JSON in Review payload: %w", err)
  }
  ```

**Pitfall 3: Including status field in summary extraction**
- **Issue**: `extractReviewSummary()` returning "approved" when no comment fields present.
- **Solution**: Check specific fields (`summary`, `message`, `comment`) instead of arbitrary keys.

**Pitfall 4: Breaking existing empty payload approvals**
- **Issue**: Accidentally changing `isApproval()` logic to require `status` field.
- **Solution**: Always check for `status` field existence FIRST, then fall back to legacy logic:
  ```go
  if statusValue, exists := payload["status"]; exists {
      // New logic
  } else {
      // Legacy logic (check empty object/array)
  }
  ```

**Pitfall 5: Not testing case sensitivity of field names**
- **Issue**: Assuming `{"Status": "approved"}` works (it shouldn't per spec).
- **Solution**: Add explicit unit test case for capital-S Status (should be treated as legacy rejection).

**Pitfall 6: Forgetting to update event schema in tests**
- **Issue**: E2E tests fail because they expect old event structure without `review_summary`.
- **Solution**: Update test assertions to check for optional `review_summary` field.

**Pitfall 7: Not truncating summary in events**
- **Issue**: Publishing 10KB summary to Redis workflow events, causing performance issues.
- **Solution**: Always truncate in `extractReviewSummary()`:
  ```go
  if len(summary) > 500 {
      summary = summary[:500] + "..."
  }
  ```

### **9.2. Rollback and disaster recovery**

**Rollback procedure:**

1. **Identify failure**: Monitor validation failure rate spike or orchestrator claim processing errors.

2. **Disable feature (if needed)**:
   - **Option A**: Redeploy previous orchestrator/pup binaries (no configuration change needed).
   - **Option B**: If validation logic is causing issues, agents can revert to legacy `{}` format immediately (no pup/orchestrator changes needed).

3. **Data cleanup**:
   - **Artefacts**: No cleanup needed. Invalid Review artefacts are blocked by pup (never created). Failure artefacts provide audit trail.
   - **Claims**: Standard orchestrator failure handling applies (claims terminate on Failure artefacts).

**Rollback time estimate**: <10 minutes (redeploy orchestrator and pup containers).

**Data migration requirements**:
- **None**. Feature is additive. Rolling back does not require data migration.
- Artefacts created with new format remain valid (orchestrator interprets them correctly even in old code via legacy fallback).

**Disaster scenario: All reviews fail validation**
- **Cause**: Bug in pup validation logic (e.g., always rejects valid `status` field).
- **Detection**: `pup.review_validation_failures_total` spikes to 100%.
- **Mitigation**:
  1. Redeploy previous pup version to all agents (10 min)
  2. Agents fall back to legacy `{}` format (no code changes, just payload format)
  3. Investigate bug in staging environment before re-deploying fix
